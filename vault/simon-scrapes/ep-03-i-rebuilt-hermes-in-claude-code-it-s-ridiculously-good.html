Layer 2 — rebuilding the memory system

So, let's go on now to talk about memory, which is probably the most important feature after this shared brand context for getting better results. And I've got to give it to Hermes — they've actually really thought through the way you store, inject, and recall information at various points in the life cycle.

When you consider memory, you've basically got three levels. You've got storage of context: how does information get saved, and at what point? You've got injection: how does that context actually get pushed into every conversation? And then more long-term, how do you recall memories that aren't recent but are still important — the ones you have to go back and search for?

Simply put, Hermes autosaves and summarises conversations every single conversation turn. It then injects important memories back into every conversation through the memory.md, the user.md, and sibling MD files — capped at around 1,300 tokens, meaning you're only loading in a limited snapshot of recent important information for every session. But its biggest limitation is when you go back to recall information that hasn't been injected into that recent memory. That's because it's searching by keyword and not meaning. So you might be able to recall exact long-term memories if you remember the exact words you used when you were talking to Claude. But it's much harder if you can't remember those exact words — which is pretty likely. And it kind of renders long-term recall in this case a bit useless. Who remembers the exact words they used with a client six months ago in a conversation they were having with Claude?

Claude Code offers two distinct persistence mechanisms. CLAUDE.md is a file the human curates — always injected into context, zero search needed, stable. The Memory tool is a newer capability the model itself writes to across sessions, creating structured entries that can be retrieved later. The self-learning loop — the part Simon skipped

Now, here's the bit where Hermes gets controversial: the self-learning loop we talked about earlier. One of Hermes's biggest selling points is that an agent finishes a task, it writes itself effectively a new skill every time, and uses it the next time. Which sounds brilliant in practice — and the first time it happens, it's probably pretty special.

But what happens by the 10th skill, or the 20th skill, when you've made tiny iterations on effectively the same process? Effectively what happens is: you start on day one, you tell it to do a specific task, and then a couple of weeks later when you come back to do a similar task, it's going to create two skills that are fairly similar, have a similar description, but are kept as separate skills and maintained as separate skills — because it's not going to capture the nuance in your process, and you also have poor visibility of all the skills that already exist. So it's just going to continue to create more skills, and each one is going to capture that approach at the moment in time, with that context, for that specific situation.

Over time you risk ending up with 15 skills that all do roughly the same thing — like LinkedIn post V1, V2; LinkedIn post for this client, that client; post writer one and two — all with slightly different contexts and slightly different bits of logic baked in. They've all got similar descriptions, so it doesn't know which one to use any given time. Then when your brand voice shifts, or when a client's positioning changes, you've got 15 places to go and update and maintain. So yes, it's absolutely faster to build this way initially. But it's a serious commitment to actually maintain properly, and basically impossible to scale across multiple clients without the whole thing turning into a mess.

Layer 3 — building a proper skill-systems architecture

Now, in-house we've created our own logic around how to tackle this, and we call this skill systems. A skill shouldn't be just a one-off task. A skill is a modular component that feeds into a skill system. Each one does one job. It lives in one place. It has a consistent naming format and gets updated in one place — and all the updates propagate to the rest of the system.

So when you want to do something complex like write a LinkedIn post in your brand voice, for a specific audience, in a specific format, you don't create a "write a LinkedIn post" skill that bakes in all of those things. You actually have the voice, the ICP, the formatting already maintained as separate skills — and then the LinkedIn post system just grabs the correct, up-to-date context from one single file for the voice, for the ICP, and for the formatting. The skill system prompt is effectively chaining those together in the right order.

When your brand voice shifts, you just update one file, and then every skill system that uses it will pull from that single file. It's infinitely maintainable and scalable. Hermes is faster to build the first skill, but building your own approach is going to be faster to build the 10th — and the 100th skill system that depends on the actual skill — and infinitely easier to maintain.

Concept · Skill chaining

Skill chaining is a usage pattern — not a named Claude Code feature — where one skill's prompt references or invokes other skills in sequence. Because skills use progressive disclosure (only name + description in context until invoked), chaining doesn't blow up your context window; each component loads only when needed. Simon's "skill systems" architecture formalises this: shared atomic skills (voice, ICP, formatting) are imported by higher-level task skills rather than duplicating their logic. This is analogous to function composition in code — small, single-responsibility units assembled into larger workflows.

Glossary — terms from this episode

Hermes
A community-created agentic AI framework for Claude Code, notable for reaching 40,000 GitHub stars in 46 days. Provides identity layers, memory injection, and a self-learning loop. Not an official Anthropic product.
OpenClaw
An earlier community agentic framework (predecessor category to Hermes), known for rapid growth but later found to have over 200 filed vulnerabilities including malicious packages in its skills marketplace. Not an Anthropic product.
Agentic OS / Agentic operating system
Creator branding for an opinionated assembly of CLAUDE.md files, skills, memory patterns, and slash commands into a structured personal AI system. Simon's term for what he built. Not an official Anthropic feature.
Identity layer
The files (user.md, memory.md, brand-context files) that tell the agent who you are, what your brand voice is, and who your audience is — so outputs sound like you rather than a generic AI. Corresponds to CLAUDE.md and companion files in Claude Code's native model.
Skills (Claude Code)
SKILL.md files with YAML frontmatter (name, description) that package reusable workflows. Progressive disclosure means only name + description are in context until invoked; the body loads on demand. The primary scaling mechanism for a large personal workflow library.
Skill systems
Simon's pattern of composing multiple single-responsibility skills into a higher-level workflow skill, rather than building monolithic skills that bake in all context. Analogous to function composition — each atomic skill is updated once and all callers pick up the change.
Self-learning loop
A Hermes feature where the agent auto-generates and self-validates a new skill after each completed task. Simon's critique: the same model that writes the skill grades its own correctness, producing skill proliferation and no audit trail.
Memory injection
The practice of loading a capped summary of recent/important memories (e.g. memory.md) into the system prompt at the start of each session, improving short-term recall without burning the full context window.
Mem search
A community memory system that retrieves memories by semantic meaning rather than exact keyword match, solving the long-term recall problem that keyword-only systems like Hermes's native recall have.
Progressive disclosure (skills)
The property that only a skill's name and one-line description sit in context at all times; the full SKILL.md body loads only when that skill is actually invoked. Keeps context lean across large skill libraries.
CLAUDE.md
Claude Code's native memory/instructions file, auto-loaded into context. Hierarchical: a user-global file at ~/.claude/CLAUDE.md and per-project files at each repo root. The primary way to give Claude persistent project context and house rules without burning tokens on skills you're not using.
