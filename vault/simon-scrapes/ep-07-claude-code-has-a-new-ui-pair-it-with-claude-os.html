Concept · Agent view (multi-agent dashboard)

The agent view is a built-in terminal UI inside Claude Code that aggregates all your backgrounded Claude Code sessions into one pane. You can see session names, their current status (running, waiting for input, idle), the last thing the model said, and how long it's been since you interacted — without opening each terminal separately. Getting in — version check and the first command

The first thing you need to do — because it's in research preview — is make sure you're on the latest version of Claude Code. Run claude --version in your terminal; it will tell you exactly which version you're on, and you need to be on 2.1.139 or higher. Migrating existing sessions into the dashboard

The first time you start the agent view, if you've already got sessions running in separate terminals it will appear blank — those sessions aren't pulled in automatically. Here's how to migrate them. Go to an existing terminal session and hit /bg (background this session and free the terminal). That task gets backgrounded, the terminal closes itself, and when you go back to the agent view another session appears in the dashboard. Do the same for each running terminal, and now you can actually close those terminal windows down. They've been backgrounded and put into the agent view.

Concept · Background sessions (/bg)

The /bg slash command tells Claude Code to detach the current session from the foreground terminal and continue running it as a background process. The session remains alive, retains its full context and tool permissions, and becomes visible in the agent view. It's the mechanism that makes the multi-agent dashboard useful: you convert terminal-bound sessions into managed background tasks without interrupting their work.

Applied economist lens

For a Companies House extraction run — where you might have kicked off five separate sessions each scraping a cohort of 20 firms — /bg on each is the clean way to hand them all to the agent view without losing state. You no longer need to keep a dedicated terminal window open per session; everything consolidates into one manageable pane. This is especially useful when sessions run overnight: start them, background them, close your laptop, and the dashboard is waiting for you in the morning with status and last-output for each.

Concept · Session grouping and status view

The agent view organises sessions into two main viewing modes: status view (sorted by whether a session is running, waiting for input, or idle) and a grouped view where you assign sessions to named groups (e.g. by client, by project, by phase). The status view is the default because it surfaces the sessions that actually need human attention. Grouping is a manual labelling step — Claude Code doesn't auto-assign sessions to groups — but once set up it gives you a persistent project-level map of all active work.

What's more useful still is that you can talk to sessions directly from the dashboard without diving into their full view. Hit spacebar to expand any session and go up and down to see the latest status for all conversations, then reply directly in-line. No context is lost from the session — you're just interacting through a thin overlay rather than the full terminal view.

The approval flow — when human input is required

One important caveat Simon highlights: if a session requires you to actually approve a tool call — running a command, writing a file, calling an API — you can't just reply in the summary view. You need to jump into the full conversation view for that session and give the approval there. The dashboard shows you that input is needed, but the approval itself is session-level, not dashboard-level.

Worth flagging

This is a meaningful operational constraint if you're running agents with broad tool permissions. You may set a session to high-trust in its own terminal (e.g. allow all file writes) but still be prompted for certain actions. The dashboard shows you the session is waiting — but you have to enter it to actually approve. Build that approval step into your workflow rather than assuming everything runs unattended.

Concept · "Claude OS" / "Agentic OS" (community framing)

"Claude OS" or "Agentic OS" is not an official Anthropic product — it is Simon's and the community's term for an opinionated folder structure that systematically injects context into Claude Code sessions. In practice it means a hierarchy of CLAUDE.md files, skills, slash commands, and memory files organised so that each session automatically loads the right project rules, client brief, and workflow definitions without manual copy-pasting. The agent view's job is to manage the sessions; the agentic OS's job is to manage the context inside each session.

Applied economist lens

This is the architecture insight that makes multi-agent work scalable for a research practice. Your consultancy-research folder already has a CLAUDE.md with the 96-firm schema, AQuA rules, and Companies House extraction instructions. When you spin up a new session from the agent view — scoped to that folder — it inherits all that context automatically. You don't need to re-paste the schema into each agent; the folder does the work. The agent view then gives you the oversight layer on top: you can see five extraction sessions running in parallel, all correctly contextualised, and intervene only when one needs approval or has stalled.

In practice

Structure: your project folder holds CLAUDE.md (schema + rules) → start each session from that folder → background with /bg → manage all from claude agents. Context is loaded per session via folder; the dashboard handles oversight.

Glossary — terms from this episode

Agent view
A built-in terminal UI in Claude Code (accessed via claude agents) that aggregates all backgrounded Claude Code sessions into one pane, showing status, last output, and idle time.
/bg (background command)
A slash command that detaches the current Claude Code session from its terminal and moves it into the agent view as a background process. The session stays alive and retains its full context.
Research preview
Anthropic's label for features that are functional but not yet in stable release — they may change, and require being on a sufficiently recent version of Claude Code.
Status view
The default sort order in the agent view: sessions waiting for input appear at the top, followed by running, then idle. Makes it easy to triage which agent needs human attention first.
Session grouping
A manual labelling mechanism in the agent view that lets you organise sessions by client, project, or phase, so you can see all related work aggregated rather than in a flat list.
Approval gate
A moment when Claude Code pauses and waits for explicit human approval before executing a tool call (file write, shell command, API call). The agent view shows that a session is blocked, but the approval itself must be given inside the full session view.
Claude OS / Agentic OS (community term)
Not an official Anthropic product. A community framing for an opinionated folder structure — CLAUDE.md files, skills, commands, memory — that automatically injects the right context into each Claude Code session. The agent view manages sessions; the agentic OS manages what's inside them.
Tmux
A terminal multiplexer that allows multiple terminal sessions to run side by side in one window. Was used as an unofficial multi-agent dashboard before the native agent view shipped.
CLAUDE.md
A file that Claude Code auto-loads into context at session start. Hierarchical: a global one at ~/.claude/CLAUDE.md and per-project ones at ./CLAUDE.md. The mechanism by which a folder-scoped session automatically inherits the right schema, rules, and instructions.
Skills
Reusable workflow packages: a SKILL.md file with YAML frontmatter naming the skill plus the full instructions. Only the name and description sit in context until the skill is invoked — progressive disclosure that keeps multi-session context lean.
