Concept · Skills

A Claude Code skill is a SKILL.md file with YAML frontmatter (name, description) plus optional bundled scripts or reference files. It packages a reusable workflow Claude invokes by name. The critical design property is progressive disclosure: only the name and description sit in context at all times; the full body loads only when the skill is invoked. This keeps your always-on context lean and fast, and is precisely what breaks when you build mega-skills.

Concept · Progressive disclosure

Progressive disclosure is the key scaling property of the skills design: Claude loads only a skill's name and description into the always-on context, not its full body. The full instructions only load when the skill is actually invoked. This keeps the base context lean for sessions with many skills defined, and it's the property a mega-skill destroys by forcing Claude to hold a thousand lines of instructions from the moment it fires.

The right answer — the skill system

With skills in isolation, you're still the connector between them. But on the other side, mega skills lose all the benefits of the original skill design. So both aren't quite right, and the right answer is actually somewhere in the middle. You build small, focused skills, and then you wire them together into something bigger using one orchestrator skill. So skills are effectively treated then as components of the skill system.

And really, what we're talking about here is not novel at all. It's an automation made up of multiple skills being chained together in a sequence, with the intelligence of an LLM to route and format inputs and outputs, as well as check in with the human when required. It's exactly the same way that OpenClaw and Hermes are able to execute tasks on your behalf. So at its crux, a skill system is a prompt and an instruction set wired around multiple skills. The prompt kicks off, and the instruction set is the brain that runs the chain or the sequence of multiple skills.

That instruction set has to understand five things fundamentally. One: the skill architecture — which skills are involved and in what order. Two: the inputs needed for each skill — what does each step need to do its job properly. Three: how outputs hand off between the skills — the output of skill one has to become the clean input for skill two. Four: the human-in-the-loop checkpoints — where do you step in to approve or adjust before the system carries on, and this varies from user to user. And five: how visual results are actually going to get displayed back to the user — markdown files with a link, an HTML dashboard, PNGs — it totally depends on what the skill is creating. And the wiring between them — and the ability to work until completion with this loop — is what makes it actually work end to end without you sitting there copying outputs and pasting them into the next prompt.

Concept · Orchestrator skill

An orchestrator skill is a SKILL.md that doesn't do the work itself — it manages the sequence. Concept · Context management in skill chains

Simon makes a point that often gets missed: when chaining skills, each skill in the chain gets exactly what it needs to do its job — nothing more and nothing less. This is achieved by spinning off subagents at relevant parts of the chain to keep each context window narrow and the quality of each output high before it's passed on. Glossary — terms from this episode

Skill
A SKILL.md file with YAML frontmatter (name, description) plus optional bundled scripts — packages a reusable, single-job workflow Claude invokes by name.
Progressive disclosure
The design property of skills where only the name and description are loaded into context at all times; the full body loads only when the skill is invoked — keeps base context lean.
Skill system
A community pattern (not an official Anthropic term) for an orchestrator skill that wires several component skills in sequence, managing inputs, outputs, handoffs, and human checkpoints end to end. Equivalent to what Anthropic calls sequential workflow orchestration.
Orchestrator skill
The coordinating SKILL.md that specifies skill architecture, routes inputs/outputs between component skills, and controls checkpoints — it directs without doing the work itself.
Mega-skill
An anti-pattern: a single massive SKILL.md trying to do everything, sacrificing modularity, reusability, maintainability, and progressive disclosure in one go.
Skill library
Your maintained set of well-built, single-job skills — a permanent reusable asset base that compounds in value as more skill systems draw on the same components.
Subagent
An agent spawned with its own isolated context window and tool set to do focused work; only its final summary returns to the main agent. Used within skill chains to keep per-step context narrow and output quality high.
Sequential workflow orchestration
Anthropic's own term for the pattern Simon demonstrates: explicit step ordering, clear dependencies between steps, and validation at each stage — what a skill system implements.
Context management
In a skill chain, ensuring each step receives exactly the information it needs — no more — so the model's limited context window is used efficiently and quality stays high at every stage.
Human-in-the-loop checkpoint
A deliberate pause in a skill system where the human reviews and approves (or adjusts) the output before the automation proceeds to the next step — essential for high-stakes work like database writes or publications.
