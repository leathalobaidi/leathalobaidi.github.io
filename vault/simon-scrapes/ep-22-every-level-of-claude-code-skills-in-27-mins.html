Concept · Skills

A skill is an official Claude Code feature: a folder containing at minimum a SKILL.md file with YAML frontmatter (name, description) and a structured process body. Drop the folder into ~/.claude/skills/ and Claude Code can discover and invoke it by name. The YAML description is always in context; the body and any referenced files are loaded only when the skill fires — this is "progressive disclosure." Skills package reusable workflows (SOPs) in a form Claude can reliably invoke without re-prompting.

Applied economist lens

Think of each skill as a codified SOP from your research toolkit. A companies-house-extractor skill that Simon's progressive-disclosure model would handle: the YAML tells Claude "use this when the user asks to pull filed accounts for a UK company," the body walks the PDF-download and field-extraction steps, and the references folder holds your 96-firm schema so it's only loaded when writing the output row — not during the initial API call. That's the structural discipline this episode is selling.

Level 1 — understanding what a skill actually is

Level one is all about understanding what a skill actually is, and it's not what people have been telling you. At its core, a skill is just a folder of knowledge. That's it. But it's how you build the files inside of the folder and the relationship between those files that actually determines how successful your skills are. Let me break it down with a real example skill that I found online that I love. It's called the last-30-days skill in a running nutrition GitHub repo. The idea is it goes out and does trending research on topics of your choice on Reddit and X — collecting upvotes, likes, retweets, all of that from sources where traditional search cannot reach.

Now, every skill has one required file: the skill.md. This is the brain of the skill — a markdown file that tells Claude what to do, when to do it, and how to do it well. Think of it as an SOP or standard operating procedure. In our adapted trending research skill, the skill.md has step one: parse the request, understand the user's intent. Step two: run the research, collecting from the Reddit and X APIs.

Beyond the skill.md, you have three optional folders. A scripts folder for executable code — like an API-calling script that the trending research skill uses to actually pull Reddit and X threads. A references folder for documentation Claude can pull in only when needed, like a brief template that defines how research reports should look. And an assets folder for templates, fonts, or icons. Ultimately they're all just additional files of deep knowledge that the skill.md references in its instructions — and they're only pulled in as and when needed.

Concept · Progressive disclosure

Progressive disclosure is Anthropic's term for the three-tier loading model inside skills. Tier one: the YAML frontmatter — always loaded into your context for every installed skill (name and description only). Tier two: the body of skill.md — loaded only when the skill is activated. Tier three: reference files, scripts, assets — loaded only when a specific step in the body explicitly calls for them. The result is that Claude is never reading more than it needs, keeping the context window clean and performance high.

That three-tier system is really, really important because it means Claude isn't loading everything all at once. It's being smart about what information it brings in and when. So to use a skill, all you need to do is drop that folder directly into your .claude folder under a skills subfolder. Each skill has its own folder. As long as it has a skill.md, it's considered a skill and Claude can see and use it.

At level one you can take a skill that somebody else has built, install it, and Claude follows those instructions to produce better work than it would on its own.

Applied economist lens

The three-tier model maps directly to how you'd structure an AQuA-grade analysis SOP. Tier one (always visible): "use this skill when the user asks to QA a salary-survey submission." Tier two (loaded on activation): the full step-by-step method — parse the Excel, run year-on-year delta checks, flag outliers above two standard deviations, check for PII in free-text fields. Tier three (loaded only when needed): a references/outlier-rules.md with your specific thresholds and a references/pii-patterns.md with the redaction rules. This way a 25-year series check doesn't dump every reference file into context on startup.

Level 2 — lean, properly structured skills that actually trigger

You can install and use a skill — great. But here's where most people hit a wall. They start building their own skills and they think that more information equals a better output. That's exactly what level two is about solving. I found a really popular Reddit post called "I Was Wrong About Agent Skills and How I Refactor Them." The story is so common I had to share it. This developer started building skills the day they dropped, and within two weeks they had a CloudFlare skill at 1,131 lines, a Svelte UI skill at 850 lines, and a Next.js skill at 900 lines. Every time Claude Code activated multiple related skills, the context window exploded, loading 5,000 to 7,000 lines every single time.

That's because he put everything in one giant skill.md file so that Claude could access all that information up front — but that is completely wrong. The problem isn't lack of information; it's actually having too much irrelevant information. Skills share the context window with your conversation, so uncontrolled loading of unnecessary information is basically a direct performance tax on your conversation. If you build skills this way, you'd probably see slowness, drifting from instructions, and Claude ignoring the obvious parts of your instructions.

Concept · The 200-line golden rule

Anthropic's guidance is to keep the main skill.md body under 200 lines. This isn't arbitrary — it reflects how much context an LLM can efficiently scan to decide what to load next. Think of skill.md as a table of contents: it tells Claude where to look; the actual detailed documentation lives in the references folder, pulled bit by bit. Anything beyond 200 lines of process instructions should be abstracted into a reference file pointed to from the body.

Here's the fix, and this is the golden rule of skill building: keep your skill.md under 200 lines max. That 200-line limit effectively caps how much context an LLM has to scan to decide what to load next. We're keeping the entry point under 200 lines so that Claude can understand what the skill offers, decide which reference file it needs to read, and load just that file — another 200–300 lines only. The skill.md is a table of contents; the actual detailed documentation goes into your references folder, and Claude should only be pulling it when it needs it, bit by bit.

The second massive mistake people make is in the description field — the YAML frontmatter. This is how Claude decides whether to load your skill in the first place. If it's too vague, the skill will never trigger. If it's too broad, it triggers on everything. Studies apparently suggest that skill activation sits around 20% for skills downloaded from a marketplace — they activate only one in five times. Here's a simple three-step framework for writing it well: first, write the trigger (what event or situation causes the skill to be used, what keywords trigger it), second, write what does not trigger it (e.g. "does not trigger for general web browsing or simple URL searching"), and third, state the outcome the skill produces. For the trending research skill: "produces a research brief that other skills can consume, and researches what's trending in the last 30 days." If any marketplace skill you have installed doesn't have these three things, it can be improved massively.

At level two you know how to build skills that are lean, properly structured, and actually trigger when they should. That is a huge competitive advantage over most people building skills right now.

Applied economist lens

The activation-description problem is real in research pipelines. A vague description like "helps with company research" will fire on half your sessions for unrelated tasks; a description like "triggers when the user asks to extract filed accounts, revenue, or headcount for a named UK company from Companies House; does not trigger for general company background questions" will fire precisely. Spend ten minutes writing the trigger/not-trigger/outcome triple for each skill you install — it pays back in every session where the skill fires first time.

Just a wall of text that's going to load into your context window as soon as the skill is activated. And the most frustrating part is that the actual content is often really good. The business logic is solid, the step-by-step processes are valuable; it's just the structure that's the problem.

Here's what you can do. You can grab the skill, look at what's inside it, and apply everything you learned in level two by refactoring it using Anthropic's skill creator skill. The skill creator skill, effectively as it sounds, creates new skills, modifies and improves existing skills, and measures skill performance — this even applies to improving skill descriptions and activation rates.

Concept · Skill creator skill

The skill creator is Anthropic's own meta-skill for building and refactoring other skills. It takes an existing skill.md, analyses what belongs in the body versus the references folder, rewrites the YAML description using the trigger/not-trigger/outcome framework, and can generate an evals suite (see level five). It's the fastest way to restructure a poorly built marketplace skill without losing any of its domain logic.

I want to create a skill that can help my website be cited by AI answers like ChatGPT, Gemini, Claude. I found a really popular one by a guy called Corey Haines — a sequence of marketing skills with brilliant content, about 13,000 stars. It does have some references and even an evals folder, but the skill.md file is around 400 lines. Although it's packed with really useful information, we're probably loading too much into the context of Claude right away and can abstract some of this out.

So we grab the installation line, go to our terminal and install it locally. Then we ask the meta skill creator skill — which is an adapted version of Anthropic's skill creator skill — to refactor it and ensure it's under 200 lines. We said: "Take the new AI SEO skill and use the skill creator skill to refactor it. I want the skill to be max 200 lines and I want all the reference info inside the references folder."

It worked out what information to abstract into different reference files and told us exactly what changed. The skill.md was 400 lines and now it's 148 lines — a 60% reduction in the number of lines loaded every time this skill activates — and it created four new reference files: authority signals, AI visibility audits, content type optimisation, and others you can see inside the references folder. So we've got more things we're pointing to, but less information loaded at any one point.

The important bit about the references file and pointing to other files is that Claude Code can load and unload information out of its context. During step two it doesn't have a huge amount of information from the other references files — it's just focusing on step two and can load the AI visibility audit at that point, which is an extra 50 or so lines. As soon as it's finished with step two, it can offload it and therefore save token usage, but also improve the accuracy of the results we're getting.

When you think about it like this, you've now got this incredible ability to take someone else's domain expertise, clean it up, structure it properly, and have it working in your system in an afternoon. At level three, you're not limited to what you can build yourself. You can pull in expertise from anywhere and refactor it into a proper skill format — so Claude is basically a specialist running workflows that didn't even exist in your business in the morning.

Level 5 — measuring and testing skill quality with evaluations

Now you've got personalised skills that produce really good outputs — but here's the question that nobody's asked yet. How do you know they're actually producing good outputs? How do you know that version two you've made is better than version one? This is where we move from building to measuring.

Anthropic just released evaluation and benchmarking built right into that skill creator skill. Up until now it was completely vibe-based whether it was giving us high-quality outputs or not. You had no way to actually measure whether your changes made things better or worse. Say we removed a reference file — we wouldn't know if that was affecting quality apart from reviewing every single output and making a judgment call. Most good skills go through many iterations before they really start solving your problems in an automated fashion, and that's exactly what the eval function is designed for — to speed up that learning cycle.

Concept · Skill evaluations

The evaluations feature (built into Anthropic's skill creator skill) lets you define three to five measurable criteria, run the skill five or ten times against a test task, and score each output against those criteria automatically. It supports A/B testing: run the same task with a reference file included versus excluded, compare pass rates, and see whether that file actually improves quality or just costs tokens. The output is a structured report with per-run pass/fail results and aggregate scores.

Let's continue with the AI SEO skill to demo this. Instead of saying "run some tests on my SEO skill," we're going to say "run a new test using the skill creator evals optimised for making sure my content follows the content type optimisation patterns." I specifically looked through the reference files and figured out I want to optimise the results for this specific reference file. Is it even loading the reference file in the first place? And if it is, is it optimising for things like: does it have one clear target query per post? Does it have an author bio with relevant credentials? Does it have internal links that relate to product or feature pages? Those are some of the things that help rank in AI search because they show authority and content-pillar depth.

We gave it a blog page from my AI agency — written before I did any work on optimising content, not very long at all. The evals feature also has a great other feature: A/B testing. If I wanted to A/B test, I could take out the content type optimisation reference file and say "Run this same task five times, once with the content type optimisation and once without — which ranks better?" It comes back with a thorough before-and-after review and helps me effectively optimise this skill: does this reference file even feed into the results in the first place, or does it help enhance quality?

I'd highly recommend when you run these evals doing three to five criteria at a time, so you're not giving it too much to think about. It creates an assertions file, then spawns all five or ten runs in parallel and grades them against the criteria. What you get is a really nice overview of the different assertions and an aggregated results report hosted on your local environment.

You can read the full output from each variation, see exactly where it passed and failed, and give feedback directly into the report — which then feeds back into improving the skill and will retest to make sure the improvements actually fixed the results. In the benchmark, you get an overview of all different runs. With the skill and all reference files, the pass rate was 93%. Taking out some reference files, the pass rate was also 93% but the time taken and tokens were significantly lower. So what this tells you is: can we take away some reference files without compromising output quality? That is where you run the A/B test to see exactly which files are improving quality and which don't even matter.

At level five, you stop guessing whether your skills actually work. You can test them, score them, measure them, and improve them. That's how you go from a skill that kind of works to one that consistently delivers.

Applied economist lens

This is AQuA model QA, applied to Claude Code skills. Instead of "does this model produce the right variance analysis?" you're asking "does this skill reliably assess whether a page has a target query, an author bio, and internal links?" Define three to five concrete criteria — for a salary-survey QA skill that might be: does it flag every year where the year-on-year delta exceeds 15%? Does it check for PII in free-text fields? Does it produce a summary table in the standard format? Run five times, aggregate, score. Then A/B test whether your references/outlier-rules.md is actually being used or just consuming tokens. This is the difference between "I think it works" and "it passes at 93% on these specific criteria."

Level 6 — skills that learn and improve themselves

Now you can measure and improve your skills — but you're still the one doing all the manual tests for improvements. What if skills could actually improve themselves? Things get really, really cool now, and this is something I've been using for my own agentic OS that makes a massive difference. The concept is super simple: your skills should learn from every interaction and get better over time. But to do this, you need one critical piece that most people are missing — a feedback loop.

Let's stick with the AI SEO skill. Every time you run it and notice something, that observation gets captured in a learnings and/or rules file inside the skill directory. The meta skill creator adds a rule section when asked to create one. It says whatever feedback we add will go into these rules, and the rules will always be read when the skill.md is read. So maybe you're running the skill and the article it produces ranks on page one within a week. You can go back and note exactly what worked: "Articles that open with a direct answer to the search query in the first paragraph seem to get picked up in AI search more quickly." Or: "A comparison table in the middle of this article increased time on page." All of that can go into the rules file and/or a separate learnings file that sits within your context, documenting on a skill-by-skill basis what works and what doesn't.

Concept · Learnings files and feedback loops

A learnings file is a structured markdown file inside your skill folder (or referenced from your CLAUDE.md) that accumulates observations from past runs: what worked, what patterns emerged, what to always check. Because it's loaded as part of the skill's tier-three references, Claude reads it before each run and applies the accumulated knowledge. It's distinct from the AQuA-style eval (which scores against fixed criteria) — the learnings file captures qualitative, contextual patterns that build up over time.

The way this gets updated is with a wrap-up skill. At the end of every session you close the session, and any skill you've used and any feedback you've been given will be generated and input directly into this learnings.md file on a skill-by-skill basis. It's just a line that says something like "after completing a task, review the output against these criteria, and if you identify specific patterns or improvements, add them back to the learnings file."

Now every time you run a skill, it's actually checking its own work. Did the article answer the search query in the first 100 words? Did it include structured data patterns from the reference file? And if it doesn't do any of that, it logs it automatically. This connects straight back to level five because you can use your evals to validate that the learnings are actually improving output quality — run the same eval before and after the learnings are applied, and you'll see the difference in your benchmark scores. They'll start improving without you actively having to run evals.

But the key thing is keeping the learnings file under control. You can't just dump random notes in there forever, or it becomes its own context window problem — feeding in too much every single time. The idea is to keep it structured and prune the obvious stuff every so often. Every week, go through the learnings file and strip out any surplus information that won't contribute to quality.

At level six, your skills are not static anymore. They're actually learning from every piece of content they produce, accumulating knowledge over time, and getting better the more you use them. Your skill in three months from now will be dramatically better than the one you built on day one, and you didn't even have to manually run evaluations for it.

Applied economist lens

Build a learnings.md into your companies-house-extractor skill. After each extraction run, the wrap-up skill logs: which PDF layouts caused the revenue figure to be misread (e.g. "£000s units on balance sheet — always multiply by 1,000"), which company types use non-standard filing dates that shift comparability, which firms have changed their year-end mid-series. Over a year of running the skill across 96 firms, that file becomes a precise set of gotchas that no prompt you write from scratch would capture. The skill in month twelve will catch errors the skill in month one missed — automatically.

Concept · Skill chaining (usage pattern)

Skill chaining — where one skill invokes another in sequence — is a usage pattern rather than a named official Claude Code feature. It works because skills can reference other skills in their skill.md instructions, telling Claude to invoke a specific named skill at a given step. The result is a pipeline where each skill stays lean and focused while the overall workflow spans multiple specialist processes. The key discipline is still progressive disclosure: no skill in the chain should load more than it needs for its own step.

Applied economist lens

Your end-to-end research pipeline for the 96-firm database is exactly this: a companies-house-extractor skill that pulls and parses the filing, passes its structured output to a ratio-calculator skill that computes margin and headcount ratios, which passes to a db-appender skill that writes the row to your SQLite database, which triggers a salary-survey-qa subagent to check the new row against your AQuA outlier rules. Each skill does one thing, stays under 200 lines, and the system as a whole covers a workflow that would otherwise require four separate prompting sessions and manual copy-paste between them. The foundation cost is building the shared econ-context/ reference folder once — after that every new skill in the chain inherits your schema and QA rules automatically.

Try this (applied-economist exercises)

If it's over 200 lines, use Anthropic's skill creator skill to restructure it: move detailed documentation into a references/ subfolder and rewrite the YAML description using the trigger/not-trigger/outcome triple. Verify that activation improves by running it five times on realistic tasks and noting how often it fires without prompting.
  2. Build a contextualised extractor. Create a companies-house-extractor skill whose YAML targets "extract revenue, operating profit, and headcount from a filed UK company account." Put the extraction steps in a lean skill.md body (under 200 lines) and move your 96-firm schema and AQuA field-mapping rules into references/schema.md. Run it on five company numbers.   3. Run a three-criteria eval on your best skill. SKILL.md
The required file in every skill folder. Contains YAML frontmatter (name, description, trigger words) and a process body (the SOP steps). Must stay under 200 lines to maintain performance.
Progressive disclosure
Anthropic's three-tier loading model: YAML always in context (tier 1), body loaded on activation (tier 2), reference files/scripts loaded only when a specific step needs them (tier 3). Keeps the context window lean.
YAML frontmatter
The structured metadata block at the top of a SKILL.md: name, description, trigger keywords, exclusions. This is what Claude reads to decide whether to activate the skill — so its description must be precise.
Skill creator skill
Anthropic's meta-skill for building and refactoring other skills. Restructures bloated marketplace skills, rewrites YAML descriptions, and can generate an evaluations suite.
Progressive disclosure (15,000-character limit)
The total budget for all skill YAML descriptions loaded into context at startup is 15,000 characters (~5 pages). Installing too many skills exhausts this budget before any work begins.
Skill activation rate
The proportion of times a skill fires when it should. Marketplace skills average ~20% activation without description tuning. The trigger/not-trigger/outcome framework in the YAML description is the primary lever for improving this.
Evaluations (evals)
A testing feature built into the skill creator skill. Define three to five criteria, run the skill multiple times against a test task, score each output, and aggregate into a report. Supports A/B testing of reference files.
Learnings file
A structured markdown file in a skill folder that accumulates observations from past runs — what patterns worked, what errors to watch for. Loaded as a tier-three reference, it gives the skill a growing memory of what good looks like.
Wrap-up skill
A meta-skill run at the end of a session to capture feedback from all skills used and append it automatically to their respective learnings files. Closes the self-improvement loop without manual note-taking.
Agentic OS
Simon's community branding (not an official Anthropic product) for an opinionated assembly of production skills, shared context folders, evaluations, and learnings files into a structured personal AI system. The underlying Claude Code features are all official; the framing is creator-specific.
Skill chaining
A usage pattern (not a named official feature) where one skill invokes another in sequence, allowing multi-step pipelines where each skill stays lean and the workflow spans specialist processes without a human hand-off between them.
