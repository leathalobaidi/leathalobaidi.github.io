Concept · Skills

A skill in Claude Code is a SKILL.md file with YAML frontmatter (name, description) plus any bundled reference files. The key property is progressive disclosure: only the name and description sit in context by default; the body only loads when the skill is actually invoked. This means you can have dozens of skills registered without flooding the context window — until you call one. The flip side is that if the skill body itself is enormous (850–1,100 lines as Simon describes), loading several at once still overwhelms the window.

Applied economist lens

This "starts from zero every time" problem is exactly what Leath hits when each CH extraction session requires him to re-paste the 96-firm schema, the AQuA outlier rules, and the per-firm research brief. The fix is structural: define those artefacts once as shared reference files, then have every skill point to them by path at the relevant step. The extraction skill loads the schema, the QA skill loads the AQuA rules, the Substack-draft skill loads the positioning brief — none of them carry the full context permanently.

Pattern 1 — Context is milk: keep it fresh and condensed

We're going to start with the most fundamental pattern. There's a saying that's been going around the Claude Code community, and I absolutely love it because it's spot-on. AI context is like milk — it's best served fresh and condensed. And this is the pattern that underpins everything. The number one reason people get bad outputs from Claude Code is not bad prompts, it's not bad skills, it's context pollution.

Your context window is not unlimited, it's finite, and everything Claude is holding in memory at any given moment — your instructions, your skill.md files, your conversation history, the outputs it's just generated — all of that is competing for the same limited space. And when that space fills up with stale information, like your failed attempts, your errors, your old corrections, or irrelevant details, Claude is going to get progressively worse. It's drifting from your instructions, it's starting to slow down, and it's not because it's broken, it's because you've overwhelmed it. This is what people call context rot.

Concept · Context rot

Context rot is the gradual degradation of Claude's output quality as a session fills up with stale or irrelevant material — failed attempts, error messages, old corrections, repeated tool outputs. The model doesn't "forget" in a human sense; it just has to find the relevant signal inside an increasingly noisy window, and it gets worse at this as the window fills. Crucially, a large context window is not a defence: a 200k-token window filled with noise is still worse than a 50k-token window with clean, relevant content.

And almost everyone is doing this without even realizing, because it happens quicker than you think. Just because a context window of a model is large, it doesn't mean we can fill it and still expect the same performance.

There's a really popular Reddit post called "I was wrong about agent skills and how I refactored them." And the story is so common that I had to share it. This developer started building skills the day they dropped, and within two weeks he had a Claude first skill at 1,131 lines, a UI skill at 850 lines, and a Next.js skill at 900 lines. And every time Claude activated multiple related skills, the context window he was using just exploded. Loading five to seven skills was like loading 5,000 to 7,000 lines into the context window, flooding it immediately. Naturally, he was getting slow outputs, drifting instructions, and Claude flat-out ignoring parts of his skill files that were necessary.

So here's the principle. In everything you do with Claude Code, you want the right context at the right time, and nothing more. And this applies everywhere — it's not just skills. In your skill.md files, keep them under 200 lines. The skill.md should be just a table of contents, not an encyclopedia. All of the deeper knowledge should sit inside the reference files, and Claude pulls them in only when it needs them, bit by bit, not all at once. So each time Claude Code hits a process step — step two, let's say, on a skill.md file — it's able to just load in the most relevant context and all of the relevant reference files, and then unload it before it hits step three.

And in your CLAUDE.md, it's the same principle. Keep it lean, point to your reference files for the detailed stuff. And if your CLAUDE.md is 2,000 lines, Claude is ignoring half of it because the important rules are probably just getting lost in the noise.

Concept · CLAUDE.md

CLAUDE.md is a plain markdown file that Claude Code auto-loads into context at the start of every session. It's hierarchical: a user-global ~/.claude/CLAUDE.md applies everywhere; a per-project ./CLAUDE.md applies to that repository. It's the primary way to give Claude persistent project rules, conventions, and memory — but only the human curates it. Simon's point stands: a 2,000-line CLAUDE.md is self-defeating. The file should state principles and point to reference files, not contain them.

Then, even in your conversations, you should be using the /clear command between unrelated tasks, always. And when a conversation's running long, make sure to compact it too, using the /compact command. Don't wait until Claude auto-compacts it, because by that point, you've already been operating in degraded output mode for ages. Compact it proactively to keep it fresh.

Mid-session when output quality dips: /compact keep the schema, the AQuA rules, and which firms still need accounts pulled — pass specific instructions so the compaction summary preserves what matters.

Concept · Skill chaining

Skill chaining is a usage pattern (not an official named feature) where one skill's output file becomes the input for a subsequent skill. In practice, skill A writes a structured output to a known file path (e.g. output/research-brief.md), and skill B's first step reads that file. The result is a pipeline: the user invokes skill A, then skill B, without manually copying anything. Each skill still runs in its own invocation context; the handoff is mediated by files on disk, not by a shared in-memory state. Simon is not describing subagent orchestration here — this is sequential, file-mediated chaining.

And here's what connects this back to pattern one so well. Each skill that runs in a chain can operate in its own context space. Your research skill can go deep for thousands of tokens of data, do all of the heavy lifting, and then hand back a clean condensed summary to the next skill. The next skill's context stays fresh and it's not polluted at all by all that raw research data. And this is extra important for founders, because you wear every single hat — you run content, sales, operations, strategy, and there is massive overlap between all of those, so they should never live in isolation. Your marketing strategy should inform your content, your content performance should feed back into your strategy, and your operations should support both. When skills collaborate, one skill's output becomes another skill's input, and you stop being the bottleneck moving information around.

Glossary — terms from this episode

80% problem
Simon's label for the pattern where AI delivers most of what you want but the remaining 20% requires constant manual correction — a symptom of missing system architecture around Claude Code, not a model limitation.
Context pollution / context rot
The accumulation of stale, redundant, or irrelevant material in the context window that degrades output quality. Happens faster than most users expect. Fixed by /clear, /compact, and lean skill design.
Context window
The finite buffer of tokens (instructions + conversation + tool outputs) that Claude can "see" at once. Larger windows help but don't solve context rot — signal-to-noise ratio matters more than raw size.
CLAUDE.md
Auto-loaded memory file that gives Claude persistent project rules across sessions. Should be lean and pointer-based, not a 2,000-line encyclopedia. Boris Cherny's practice: add a rule every time Claude makes a mistake.
Skills (skill.md)

Business brain (shared reference files)
Simon's term for a set of short shared context files (tone, audience, standards) that every skill imports at the relevant step, rather than each skill carrying its own copy. The DRY principle applied to prompts.
Skill chaining
A usage pattern where skill A writes a structured output file and skill B reads it as input, creating a file-mediated pipeline. Each skill runs in its own context; handoffs are compact artefacts, not raw session state.
learnings.md

Slash command that wipes the current conversation and starts a fresh session. Use between unrelated tasks to prevent context pollution.
/compact
Slash command that compresses the conversation history to a model-written summary, freeing context space. Can be invoked proactively with guidance on what to preserve, rather than waiting for auto-compaction.
