Concept · CLAUDE.md

CLAUDE.md is a markdown file that Claude Code auto-loads into its context at the start of every session. It is the primary mechanism for persistent, human-curated context: project purpose, coding conventions, important file paths, house rules. There is a hierarchy — a user-global ~/.claude/CLAUDE.md and per-project ./CLAUDE.md files — so you can have universal preferences alongside project-specific instructions. Concept · MCP servers

MCP (Model Context Protocol) is an open standard for connecting external tools and data sources to Claude. An MCP server exposes a set of tools — for example, a Companies House MCP that Leath uses can query company profiles, directors, and filing history. Claude Code calls these tools the same way it calls built-in ones. MCP is how Claude reaches systems outside the local filesystem — databases, SaaS APIs, internal services — without bespoke plumbing for each integration.

What markdown is and why Claude Code uses it everywhere

Now that setup is complete, Simon pauses before the first real build to explain markdown, because you are going to see files ending in .md or .markdown absolutely everywhere. Your CLAUDE.md file, all your skill files, planning documents, slash commands — all of it is markdown. It sounds technical but is straightforward once you see it.

Markdown is a way of writing formatted text using nothing but plain text. A single hashtag is a top-level heading (heading 1). Two hashtags is heading 2. Three is heading 3. An asterisk on each side makes bold text. Dashes create bullet points. Square brackets plus parentheses make links. You can also do block quotes and inline code. That's basically the whole syntax.

To make this tangible, Simon copies a markdown snippet and pastes it into a Notion page. Notion renders it immediately: the heading 1 becomes the biggest header, heading 2 and 3 step down in size, bold text is bold, bullet points are bullets, the link becomes a clickable block, inline code appears in a code font. The conclusion: markdown is just a compact, symbol-based notation for something that humans already read naturally as formatted text.

There is another reason the whole ecosystem uses markdown: large language models read it extremely well. LLMs are trained on vast amounts of markdown-formatted text, so they understand the document structure — which sentences belong under which heading, where one section ends and another begins. When you hand Claude a CLAUDE.md file with headings, subheadings, and bold text, it doesn't just see a wall of characters; it sees a properly structured document it can navigate. You don't need to memorise any markdown syntax either, because Claude will start writing it for you and you'll pick it up by reading what it produces. Simon points to the README file in the course folder, where everything covered so far is represented as hashtags; the Preview tab shows it rendered as a proper formatted page.

Second, skills. Skills are folders of expertise you hand to Claude: process documents, standard operating procedures for recurring tasks. A content-repurposer skill knows your tone, your structure, and your best-performing posts as examples. A proposal-generator skill follows your exact format for client proposals. Once a skill exists, you don't have to re-explain that context every session. Claude picks it up automatically whenever you ask it to do the thing that skill covers.

Concept · Skills

A skill is a folder containing a SKILL.md file with YAML frontmatter (a name and description) plus any supporting reference files the skill needs. Claude Code uses progressive disclosure: only the name and description sit in the main context window. When you invoke the skill by name, the full body loads into context. This means you can accumulate dozens of skills without bloating every session — only the skills actually in use consume context space.

Concept · Subagents

Subagents are spawned via Claude Code's Task (Agent) tool. Each subagent gets its own isolated context window and tool set, does a focused piece of work — a web search, a code review, a sub-build — and returns only its final summary to the main agent. This is the mechanism that lets a complex job stay manageable: instead of one giant context filling up with intermediate results, you have lightweight workers that each carry a small, focused context, then hand off a clean result.

Fourth — and biggest — planning. The content-repurposing engine worked for the demo because the task was small: one transcript, three drafts. Try to scale it up to a real business use case — say, a multi-week content campaign that pulls source content from Notion, generates drafts for five different platforms, and schedules posts on those platforms — and it falls apart completely. That is too complex for Claude Code without a plan. Claude would make its own assumptions, fill the context window, and you'd end up doing more rework than if you'd planned it out first.

Simon explains context rot. Claude has a context window — a limited amount of information it can hold in its head at any one time. The more you ask it to do in a single prompt, the faster that window fills. You can watch it happen: there's a green bar at the bottom of the terminal showing how full the context is. As the window fills, quality degrades. By the end of a long, complex prompt — like building out a multi-stage campaign — Claude is making decisions based on context it can no longer actually locate. The fix is not to ask Claude to try harder. The fix is to give it a plan that feeds in context at the right time, breaking big projects into small focused tasks, each running in its own clean context. That is exactly what frameworks like plan mode and GSD do, and the course will cover both.

Concept · CLAUDE.md

CLAUDE.md is a plain-text (Markdown) file that Claude Code auto-loads at session start, inserting its contents into the model's context before any user message. Parent inheritance — one master CLAUDE.md that feeds all your projects

But when we think about it, most people don't just work on one project and you're probably not planning on adopting Claude Code for just one project either. I want you to understand this now while you're still learning because it's going to change how you think about using Claude Code day-to-day and what information is inherited from which project files.

Here is the core insight. Once you get comfortable with Claude Code, you're not going to stay in one project folder. You'll end up with lots of different project folders: your content engine, your client work, a personal brand folder, an Obsidian memory folder, admin like emails and invoices and finance tracking, experiments or apps you're building on the side. And suddenly you face the question: should I build a CLAUDE.md for every single one of these and repeat myself? Do I paste my voice rules into every project?

There's a much better way to do this. And once you understand it, you stop thinking of Claude Code as a tool you use project by project and you start thinking of it as your business operating system. Inside the school community we've built out what's called the agentic operating system which builds on this foundational concept. The key concept you need to understand right now is parent inheritance. Files inherit from their parent folders. When you run Claude Code in any folder, it doesn't just read the CLAUDE.md in that exact folder — it walks up the folder tree all the way to the top folder, combining every CLAUDE.md file it finds on the way. The file closest to the folder you're working in takes priority. So if there's a CLAUDE.md above your Claude Code course folder, it's going to inherit those rules, but any rules that contradict those in the project-level CLAUDE.md will take priority over the rules from higher up.

Concept · CLAUDE.md parent inheritance

Claude Code walks the directory tree upward from your working directory, loading every CLAUDE.md it encounters and merging them. The closer the file is to your working directory, the higher its priority — so a project-level CLAUDE.md overrides conflicting rules from a parent-level one. This means you can put global identity rules (voice, tool preferences, gotchas) in a master CLAUDE.md two or three levels up, then add project-specific overrides in each project's own file, without copy-pasting. The trade-off: every CLAUDE.md in the chain consumes tokens, so a very large parent file raises context cost for every sub-project.

Claude Code distinguishes two scopes for configuration: global (~/.claude/) and local/project (the project directory and its ancestors). The same artefact placed in a project folder applies only in that project (and its subfolders via inheritance). When the same rule is defined at both levels, the closer (more local) definition wins. Concept · Context rot

Context rot is the gradual degradation in output quality that happens when a model's context window fills up with irrelevant or stale information. Instructions given early get crowded out, the model starts confusing earlier instructions with later ones, and quality slips. Skills are a structural defence: only the front-matter description is permanently in context; the full instruction body loads only when the skill is invoked, then clears. This keeps the working context lean and focused throughout long sessions.

At its essence, what we've already seen is that skills are just a folder of: effectively a process document (which is our skill.md), any scripts or reference files, and any assets too. You don't need to have any of those supplementary items — they just supplement the main skill.md file. We're trying not to overload the context of Claude. So as a rule of thumb, we keep our main skill.md file under 200 lines. We sometimes slip a little bit over. But the same as CLAUDE.md — any information you want to include that doesn't belong in the instruction-manual steps should go inside reference files (like examples or API guides or, in our case, platform formatting rules). Those don't need to be in the skill.md. They just need to be loaded in at a specific step.

Say, for example, in our content-repurposing skill one of the steps is "produce it for LinkedIn" — and at that point it loads in the platform formatting rules for LinkedIn, then offloads that context once it moves to the next step where those rules aren't needed anymore. It's all about loading context in at the right time and then offloading it: one, to save money because tokens cost a lot; and two, to improve output quality because we don't want context rot from overloading Claude.

Concept · YAML Front Matter in skill.md

The front matter block (delimited by --- above and below) at the top of a skill.md contains the name and description fields that Claude reads to decide whether to invoke this skill. These two fields are always loaded into Claude's context — they are the "always on" signal. The body of the file (everything after the closing ---) is the instruction set and is loaded only when the skill triggers. Simon notes this is "front matter" in Markdown convention; the official name is YAML front matter.

The most important concept you need to understand — and this relates back to loading the right context at the right time — is that skill.md files have this front matter. Inside the skill.md we have the name and description fields and then underneath we have the instruction steps. So let's go into our skill.md file and start building it out.

For a content-repurposing skill the description might start as: "Use whenever the user asks to repurpose, adapt or rewrite content for LinkedIn, X, Instagram, a newsletter or any social platform." The art of building a good description that Claude invokes reliably is being very specific about three things. First, the trigger: say exactly what words or requests should fire it. Second, what it does — not just "use whenever" but what it actually produces. Third — and this is often missed — when it should not trigger. So you might extend this to: "Do not trigger when the user asks for original copywriting or ad copy. This skill is specifically for repurposing from a source transcript."

It's easy to get Claude to help you build these descriptions. You can just say: "Improve this YAML front matter for this skill. Progressive Disclosure — The Architecture Behind Skills

Now here comes the trick in how you use skills, and then I'll explain how context is loaded in one piece at a time. You can still invoke a skill using a slash command: if I type /content-repurpose, you can see that the description now appears in the slash-command picker — "Trigger when the user says repurpose…" — so I can invoke it explicitly. But the real power of skills is that you don't need to invoke them explicitly. I can just say "Repurpose transcript.txt" and what we're going to see is that Claude reads through our installed skills, finds the content-repurpose skill whose description matches, reads that skill.md file, and now goes through each step the skill.md specifies to produce the outputs.

Getting the description right is critical, because you'll have lots of skills that are similar — multiple content skills, not just repurposing. The trigger words, the what-it-does, and the do-not-trigger-when are all essential to loading the right skill at the right time. Once Claude is working through the skill you can see it loading the right reference context at the right time — which brings us to the concept of progressive disclosure.

Concept · Progressive Disclosure

Progressive disclosure is the pattern by which skills reveal context in layers, only when needed. Layer one: the YAML front matter (name + description, ~100 words) — this is always in context for every installed skill. Layer two: the skill.md body (≤200 lines) — loaded only when the skill triggers. Layer three: reference files — loaded individually, one at a time, only at the step that references them, then unloaded. This three-layer architecture is what keeps context lean despite having many rich skills installed.

Progressive disclosure is just disclosing the right context at the right time. Layer one is the description — around 100 words, the YAML front matter name and description. This is always loaded into your Claude Code instance. So whenever you have skills installed — whether in your local project folder or your global folder, remembering they cascade down — all of those descriptions are in context. If you've installed a package of 200 skills and they have overlapping purposes, all of their descriptions are loaded in, which means: one, you're using more tokens; and two, Claude Code might have trouble picking between certain skills. So it is critical to get those descriptions nailed so that the right skill activates at the right time.

Once Claude decides it should load a skill, it then loads in the skill.md body — around no more than 200 lines — and this is loaded only into Claude's context when the skill triggers. This is all the process steps. Then you can have unlimited reference files. In a fairly simple skill we've already got content-angles and platform-formatting. We could also have examples per platform and a brand-voice guide. We reference them in the skill.md and only bring them in when needed. Claude knows to load the platform-formatting reference only at step five, when it's applying voice and quality checks — and then unload it before step six.

Platform formatting alone is 94 lines of context. By loading and unloading it at only one step, I've saved myself and Claude from carrying 94 lines through every other step of the skill. The point with skills is that Claude pulls in exactly what it needs, no additional context, and pulls it in at the right time. And we can see that our skills have now successfully executed — and in the drafts folder we have outputs saved for LinkedIn, newsletter, and X.

Applied economist lens

Think about your Wenta RCT design work. A rct-power-calc skill could have three reference files: references/outcome-definitions.md (loaded at step 1 only), references/assumption-ranges.md (step 2), and references/pre-reg-template.md (final step only). Each is 50–80 lines. Loaded one at a time, you avoid the ~200-line overhead that would otherwise sit in context for the entire session. Planning frameworks — matching the tool to the size of the work

Now, we already talked about planning earlier, but let's build out why planning actually matters and some frameworks apart from the built-in plan mode that you can use to build out better plans. And I'm going to reiterate the context of why planning is so important with an example of how you used to interact with chatbots and ChatGPT. You used to ask a chatbot a question and you'd get a single response back — there was effectively nothing to plan. It was going out and answering a query. You typed, you got an answer, you moved on to the next query.

But because Claude Code is an entirely different beast, it can take actions. It can build entire applications and wire up your MCPs to your Notion etc., and even run for hours on a single given task. The more capable the tools get, the more we have to invest in upfront planning so that we don't hit the issue of context rot or context degradation as we go through. So we can pass in the right context at the right time. That's why we plan. We want to keep Claude focused, keep it on the right task at the right time. And that's all done through context planning or planning frameworks. It's all about giving Claude smaller, well-defined tasks at the right time, so it runs those tasks with a clean context. A written plan survives when Claude's memory does not survive.

Concept · Context rot / context degradation

"Context rot" is a community label (not an official Anthropic term) for the gradual degradation in output quality as a long session accumulates prior turns, tool outputs, and partial results. The model's attention is finite — eventually distant instructions lose influence, earlier tool outputs bloat the window, and Claude drifts from the original goal. Planning frameworks combat this by externalising state into files (a PRD, a phase plan, a state document) so that the model can reload what it needs even after a context reset, rather than relying on the conversation window alone.

There are kind of three levels of planning. The first one we've already touched, which was plan mode. I'd use this for anything under an hour — it's built into Claude Code like we saw earlier, Shift+Tab twice. You see "plan mode is on" down here. Now, level two — and these, by the way, are just opinions; you can use whatever framework you want — level two, I would use something called a PRD generator. It's going to generate a structured requirements document, which you can use for multi-hour projects. And then level three, my favourite planning framework is the GSD framework — Get Shit Done — which I've talked about in previous videos too. I love this framework, and I think it's the most comprehensive and best way to plan, but you pay for it in the amount of tokens it uses and also the time it takes to execute each phase. So you're basically matching the framework to the size of the work: plan mode for small tasks, PRD generator for multi-hour projects, GSD framework for full application builds, multi-day builds basically.

Worth flagging

GSD ("Get Shit Done") is a community-created planning framework — a third-party GitHub repo, not an Anthropic product. Simon genuinely uses it and recommends it, but the repo, its commands, and its token costs all change independently of Claude Code itself. Treat the specific workflow Simon describes (three-command loop: plan → execute → verify) as a useful pattern, not a stable API.

Now, a PRD document is a product requirements document. It's effectively used by product managers, product teams, and engineers. Glossary — terms from this episode

CLAUDE.md
A Markdown file Claude Code reads automatically at session start, acting as a persistent system prompt for the project. Leath should keep one per project (e.g. one for the 96-firm database, one for the SPE survey) and a global one at ~/.claude/CLAUDE.md for cross-project rules.
Parent inheritance
When Claude Code starts in a subfolder it also loads the CLAUDE.md files of every parent directory up to the global one. This means shared rules (PII policy, output conventions) defined once at the root cascade automatically to every child project without copy-pasting.
Skill / SKILL.md
A reusable workflow packaged as a folder containing a skill.md process document plus optional reference files and scripts. Claude Code loads only the name and description (the YAML front matter) until the skill is invoked — a key scaling property called progressive disclosure.
Front matter
The YAML block between --- delimiters at the top of a skill.md, containing at minimum name: and description:. Claude Code indexes these fields to decide when to load the full skill body, so a well-written description is what determines whether the skill activates correctly.
Progressive context loading
The practice of splitting a skill's supporting material (platform rules, field maps, example documents) into references/ files that are loaded only at the step that needs them and then released. Keeps token cost low and prevents "context rot" — degraded output quality from an overloaded context window.
Context rot
Simon's term for the quality degradation that results from stuffing too much text into the active context window. Avoided by keeping CLAUDE.md under ~200 lines, keeping skill.md under ~200 lines, and using reference files loaded on-demand.
Plan mode
A built-in Claude Code mode where the model uses only read-only tools to investigate a codebase and draft a proposal before being allowed to make changes. Invoked with shift+tab or the --plan flag. Simon's minimum recommendation for any non-trivial task.
PRD Taskmaster
A community skill (installable via git clone) that walks Claude through 12 discovery questions to produce a phased Product Requirements Document, then executes each phase against that plan so a context reset does not lose state. Sits between built-in plan mode and the full GSD framework in complexity.
GSD framework
"Get Shit Done" — a community three-command workflow (/gsd) that generates a requirements document, breaks work into phases each with their own plan, and tracks state in a project folder. Appropriate for multi-day builds; uses substantially more tokens than plan mode. Not an Anthropic official feature.
Subagents
Specialist Claude Code instances with their own isolated context window, handed a self-contained task and returning only their final result to the orchestrating agent. Anthropic ships built-in planning and explore subagents; custom ones live in .claude/agents/.
/loop & cron
/loop runs a prompt on a recurring interval inside the current session (expires after 3 days when the terminal closes). The cron skill schedules persistent background runs with a job ID that can be cancelled via cron delete <job-id>. Together these enable unattended scheduled workflows.
Remote control (QR code)
A Claude Code feature that starts a web session and exposes it as a QR code (/remote-control), allowing instructions to be sent from a mobile phone. Claude executes on the local machine; the session can be closed from the phone when done. Useful for long-running tasks where the user is away from the keyboard.
claude --resume
A CLI flag that displays a numbered list of past Claude Code conversation histories so any prior session can be reopened and continued. Equivalent of a chat-history browser; all sessions are stored in the global .claude folder.
Skill marketplace (skillsmcp.com)
A community-run directory of downloadable skills (800k+ listed at time of recording). Simon cautions that many lack proper reference files or process steps and should be inspected and customised before use — not an official Anthropic resource.
