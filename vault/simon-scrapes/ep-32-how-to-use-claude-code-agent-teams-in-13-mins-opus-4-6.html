Concept · Context rot & compaction

Context rot is the empirical finding that LLM reliability degrades as the context grows — not because information disappears, but because the model struggles to retrieve the right needle from a bigger haystack. Claude Code's built-in response is automatic compaction at around 95% fill: the conversation history is replaced by a model-written summary, and only that summary enters the next exchange. The risk is that a lossy summary silently drops a binding rule or a critical schema detail. The manual /compact command lets you invoke compression at a chosen moment and append instructions about what must survive.

This is the exact failure mode that breaks your 96-firm accounts-extraction sessions. You load the Companies House MCP, paste your firm schema and QA rules, start pulling PDFs — and by firm 40 the AQuA outlier thresholds you stated at the start are the first casualty of compaction. The fix isn't a longer prompt; it's architecture. Pin your schema and rules in a short CLAUDE.md so they reload on every exchange regardless of compaction. Then consider whether firms 1–50 and firms 51–96 should be separate sessions — or separate agents — rather than a single sprawling context.

Why subagents alone aren't enough — the hub-and-spoke bottleneck

Single agents get overwhelmed, they forget, and actually experts in the community spotted this problem early, and they started building out their own subagents. So, you'd only use subagents for two reasons with Claude Code. The first is to improve output quality. Think hiring a specialist versus hiring a generalist. We can have one Claude that's just delegating to focused, specialised instances in the same session that have their own context window, and therefore we can improve the output quality and also reduce the cost because we're not feeding all of the context to those specialised subagents. And the second reason is to get things done faster. So, instead of having them feeding back and forth between each other, we would have them in parallel — two terminal windows open taking on non-dependent tasks. Like for example, in terminal one we could have it writing our LinkedIn post, in terminal two at the same time in parallel, we could have it working on Instagram posts which are often unrelated.

Concept · Subagents

A subagent is a Claude Code instance spawned by the main (parent) agent to do focused work in its own isolated context window. Only its summary or output is returned to the parent — the parent's context is not visible to the subagent. This isolation is the feature: each specialist starts fresh, so it doesn't inherit the noise that has accumulated in the parent's window. Subagents can run sequentially (chained) or in parallel (separate terminals), but they have no direct channel to each other — all inter-agent communication must flow through the parent.

People like Boris Cherney, the creator of Claude Code and other power users in the community, were tackling this by building their own subagent system. Boris even said he ran up to 15 agents — five in parallel in his terminal, and then five to ten Claudes online at the same time. And this is great in theory, but it broke down when the agents actually had to communicate with each other.

Subagents can only report back to the parent. So if for example we had a content research team with a researcher as subagent one, the writer as our main instance, and the reviewer as subagent two — the subagents, the researcher and the reviewer, can only feed information back to the main instance, the writer. The reviewer can't review the research, and the researcher can't see any outputs of the reviewer; the writer handles all of that. So with subagents, what you actually end up with is the main agent becoming the bottleneck. There's only two-way communication between that main agent and every single specialist subagent.

So when we were spinning up subagents, each subagent has its own context, and the results are returned to our main agent — in our case the writer — and we'd have the reviewer and the researcher as subagents there. But the main agent is coordinating all of that work. So if multiple tasks and multiple subagents feed into that main agent, that becomes the clear bottleneck. And this only works if you don't need the researcher to interact with the reviewer and vice versa. So subagents acted like a hub-and-spoke model, and they gave us delegation, but they didn't give us collaboration. And that's the gap that agent teams actually fills.

What agent teams add — the shared task list

And they do this through a shared task list. Concept · Agent teams & the shared task list

Agent teams are Claude Code's experimental multi-agent architecture (as of early 2026) in which multiple Claude instances run as "teammates" — full Claude Code sessions in their own right — coordinated by a shared task list rather than a parent–child delegation tree. Unlike subagents, teammates can pull tasks from the shared list and message each other directly, enabling genuine peer-to-peer collaboration. The cost is that every teammate's context must be seeded with the full setup (CLAUDE.md, MCP configs, tool permissions), making teams more token-intensive than subagents for the same breadth of task.

Limitations — cost, file conflicts, and context inheritance

But the important thing is to understand which tasks actually are suitable for a team and which should just be using subagents or single agents. Let's quickly talk about some of the potential limitations of using this team approach.

First is no context inheritance. Teammates spun out don't get that conversation history from your main workflow. So, we need to actually make sure that Claude is passing them a generous amount of the context so that they can do their job.

Permissions do propagate, so if we start with bypass permissions on, they will be passed to all of the teammates. So, say we have a reviewer agent that we don't want to be actually editing — we're going to need to actually ask Claude to change the permissions of that.

Concept · Context inheritance & permission propagation

When agent teams spawn teammates, each teammate starts with a fresh context — it does not inherit the main agent's conversation history. The main agent must explicitly pass necessary context (schema, rules, prior findings) in the teammate's initialisation prompt. In contrast, permissions do propagate automatically: if the orchestrating session is running with --dangerously-skip-permissions, all teammates inherit that. This asymmetry matters: you control what context a teammate receives, but you cannot easily restrict a teammate's permissions relative to the parent without explicitly instructing Claude to do so when it spins up that agent.

Glossary — terms from this episode

Agent team
Claude Code's experimental multi-agent mode (enabled via a settings.json flag) in which multiple Claude instances run as teammates sharing a task list and able to communicate directly with each other — not just with the parent agent.
Subagent
A Claude Code instance spawned by a parent agent to do focused work in its own isolated context window. Returns its output to the parent; cannot directly communicate with sibling subagents. The hub-and-spoke predecessor to agent teams.
Shared task list
The coordination mechanism in agent teams: a common list of tasks that the main agent populates and all teammates can pull from and update, enabling peer-to-peer collaboration without routing everything through the parent.
Context rot
Degradation in a model's reliability as context grows — not because content is deleted, but because the model becomes worse at finding the right information amid the noise. The core reason long single-agent sessions drift.
Compaction / auto-compact
Claude Code's built-in response to a nearly-full context: it replaces conversation history with a model-written summary. Automatic near the limit; manually invokable via /compact with optional instructions about what to retain.
Hub-and-spoke model
Simon's term for the subagent architecture: a central parent agent delegates to specialist subagents and all communication flows in and out through that hub. Efficient for delegation; a bottleneck when specialists need to coordinate with each other.
Orchestrator
The main agent acting as a coordinator — initialising teammates, seeding their context, and managing the shared task list — rather than doing all the work itself.
Context inheritance
Whether a spawned agent receives the parent's conversation history. In agent teams, teammates do not inherit conversation history; the orchestrator must explicitly pass any required context in the teammate's initialisation prompt.
Permission propagation
Agent team teammates automatically inherit the permission level of the session that spawned them (e.g., --dangerously-skip-permissions). Restricting a specific teammate requires explicit instruction at spawn time.
tmux
A terminal multiplexer (installed via brew install tmux) that splits a terminal into multiple independent panes. Used alongside agent teams to give each teammate its own visible window rather than interleaving all output in one stream.
settings.json
Claude Code's configuration file controlling tool permissions, environment variables, hooks, and feature flags. The claudeCodeExperimentalAgentTeams: true flag enables agent teams; changes take effect after restarting Claude.
Two / six / eight out of ten
Simon's informal scoring rubric for how much inter-agent collaboration a task genuinely needs. Two-out-of-ten tasks (independent parallel outputs) are better served by subagents; eight-out-of-ten tasks (tight mid-run coordination) justify the overhead of agent teams.
