Stop Using Claude Code in Terminal (It's Holding You Back)

16:32 runtime 140K views Published 2 Apr 2026 youtu.be/uhMCy25NBfw

The bottleneck in multi-agent Claude Code work isn't the agents — it's the terminal interface itself, and the fix is a goal-first dashboard that abstracts away session management entirely.

What this episode covers

Simon diagnoses a problem that emerges once Claude Code agents get genuinely capable: running five parallel terminal sessions creates so much cognitive overhead — switching, re-reading context, tracking which task is where — that you spend more time managing agents than benefiting from them. He surveys the existing landscape (Tmux, the Claude desktop app, Vibe Kanban, Paperclip, and a cluster of coding-session dashboards) and explains why each one fails business users. He then unveils the "command center" he built on top of his Agenta OS: a Kanban-style web dashboard that manages goals, not sessions, with built-in business context, scheduled tasks, skills management, and multi-client support.

Why it matters for an applied economist

Your research pipeline — querying Companies House for 96 firms, running AQuA QA passes on 25 years of salary-survey data, generating Substack drafts, chasing DCC board-reporting deliverables — is exactly the multi-goal, multi-agent workload Simon is describing. Each of those jobs benefits from running in parallel isolated sessions, but flicking between five terminals destroys the productivity gain. The goal-oriented dashboard pattern — where you describe a business outcome and the system routes it to the right agent with the right context — maps directly onto how you already think about analytical projects: define the deliverable, delegate the execution, supervise the outputs.

Concept · Multi-agent parallelism

Claude Code supports running multiple independent agent sessions simultaneously — each in its own terminal or worktree, each with its own context window. Because each session is isolated, you can have one agent extracting data, another drafting text, and another running QA checks, all at once. The bottleneck has shifted from agent capability to human coordination: keeping track of which session is doing what, and switching intelligently between them.

Concept · Tmux

Tmux (terminal multiplexer) is a Unix tool that lets you split one terminal window into multiple panes, and keep sessions alive even if you disconnect. It's a common power-user move for running several Claude Code sessions side by side. The design principle: start from goals, not sessions

And this is the gap that none of these tools fill. So, let me explain what it's actually missing. Every tool I just showed started from the bottom up. So, they start with the session in the terminal. They start with the code, and then they try to add a project management layer on top of it. But, what we actually need is the opposite. We need to start from the top, and work our way downwards. So, we need to start with the goal, like, "I want to build a lead generation system." And then, let the system figure out what sessions to spin up, how in-depth the planning should be, how many agents are needed, what skills to use, and how to get it done — i.e., all the technical overhead.

And it's much like hiring a competent employee. You give that employee a goal. You say, "Here's the deadline. Go and figure it out, and update me when you've made some progress." That's the level of abstraction as business owners that we need. And there's also one more thing that none of these tools handle at the moment. None of them have the operating system in the back end that stores your business context, your brand voice, your client details, your content strategy, your target audience. None of these Kanban tools show anything about your business. They're just managing coding sessions in a complete vacuum. So, what we need is a tool that thinks in goals, not sessions. It knows your business context, and lets you manage everything from one place with minimal interaction with the terminal.

Concept · Business context / CLAUDE.md as memory

In Claude Code, persistent context lives in CLAUDE.md files — project-level (in the repo root) and global (in ~/.claude/). These files are automatically loaded at the start of every session, so they act as institutional memory: your working standards, domain knowledge, schemas, naming conventions, and QA rules. Simon's Agenta OS extends this by also storing brand voice, client profiles, and content strategy — so every agent that spins up already knows your business context, without you needing to re-paste it each time.

Concept · Auto-mode / permission levels

Claude Code can run in interactive mode (asking permission before each action) or in auto/bypass mode (executing without pausing). The settings.json permissions field controls which tools and actions are pre-approved. "Dangerously skip permissions" means allowing all actions without prompts — appropriate for well-scoped, low-risk tasks in a controlled repo, but risky for anything touching production systems or sensitive data. For most analytical research work, a middle ground — pre-approving read and write to your research directories, but requiring confirmation for any network calls or deletions — is safest.

Concept · Scheduled tasks in Claude Code

Claude Code can interact with your operating system's scheduler (cron on macOS/Linux, Task Scheduler on Windows) to fire agent sessions on a recurring schedule. A scheduled task is defined in a file that specifies the trigger interval and the command to run. Simon's implementation stores these as files that the dashboard can activate, deactivate, test-run, or delete — with each run producing a log file. This is a real Claude Code capability built on OS-level scheduling, not a platform feature unique to the command center.

Concept · Skills (SKILL.md files)

In Claude Code, a "skill" is a Markdown file (conventionally SKILL.md) stored in .claude/skills/<skill-name>/ that provides Claude with a structured prompt, workflow, and any reference materials for a specific repeatable task. When you invoke a skill (via a slash command like /companies-house-extractor), Claude reads the SKILL.md and follows its instructions. Skills are how you encode domain knowledge — your extraction schema, your QA checklist, your report template — so you don't have to re-specify it each time. The command center's skills browser is a rendered UI on top of these raw Markdown files.

A Unix terminal multiplexer that lets you split one terminal window into multiple panes and keep sessions persistent. Used by developers to monitor several Claude Code agents side by side, but doesn't solve the higher-level goal-tracking problem.

Recurring agent sessions triggered by the OS scheduler (cron on macOS/Linux). Claude Code can read from and write to cron via its computer-use capability, letting you set up daily QA runs, weekly digests, or monthly consolidation jobs.
Auto-mode / permission bypass
Running Claude Code without pausing for approval of each tool call. Configured in settings.json under permissions. Efficient for well-scoped tasks; requires careful judgment about what actions are safe to pre-approve.
Goal vs session abstraction
The design principle Simon advocates: users should specify a business outcome, and the system should handle which agents to spin up, what skills to use, and how to coordinate them. The inverse of starting from "open a terminal."
CLAUDE.md (project context)
The Markdown file(s) Claude Code reads at the start of every session to load persistent context — your schemas, QA rules, working conventions, domain knowledge. The primary mechanism for giving every agent session institutional memory without re-pasting.
