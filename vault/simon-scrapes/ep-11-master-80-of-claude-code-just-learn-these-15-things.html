Concept 1 — What Claude Code actually is

If you're using Claude Code at any level, then this video is for you because in the next 20 minutes, I'm going to teach you the 15 things that cover about 80% of how Claude Code is actually used. This is all based on what I see across thousands of real users inside my Agent Academy doing day-to-day. Master these and you'll be better than most people at Claude Code. You can think of these as short, focused examples that show you what's possible and point you toward where to go deeper next.

Let's jump into it with what Claude Code actually is. Using ChatGPT is like asking for advice, but Claude Code on the other hand is not a chatbot in the terminal — because it can not only read your files, it can actually write to them too, and run commands on your machine. Give it a brief with the right context, you let it work, and then you review what it's done. Think about website building, content creation, file sorting, researching — pretty much anything is possible inside Claude Code. It takes the actions, not only gives you advice.

Concept · Claude Code as an agent

Claude Code is Anthropic's official agentic CLI. Unlike a chat interface, it has built-in tools — file read/write, bash execution, web fetch, search — and can chain those tools together to complete multi-step tasks without you shepherding each step. That "takes actions" framing is the key distinction: Claude Code is an agent, not an advisor.

Applied economist lens

The "brief, let it work, review" loop matches your natural analytical workflow — you scope a task (extract all turnover figures from 20 Companies House filing PDFs), set it running, and return to QA the output rather than typing in every intermediate step. The key shift is treating Claude Code as a contractor who can access your file system, not a search box. That reframe changes how you write prompts: less "explain X", more "do X and write the result to output/ch-extract-2024.csv".

Concept 2 — Plan mode: read-only thinking before writing

Now, because anything is possible, it means Claude Code is powerful out the box, but you need the right framework still on how to use it to get the most out of it. Plan mode is genuinely one of the most underused features in Claude Code, and it comes straight from Boris Cherney's own workflow. Boris Cherney is the guy who actually built Claude Code, so he knows a little bit about it — it's safe to say.

Hit Shift+Tab twice on your keyboard, and Claude will flip into plan mode. And I know we just said we wanted Claude to build stuff for us, and plan mode is actually going to do the opposite — it's going to switch it into a read-only mode. But what it's actually doing in the background is creating a refined set of specific contexts so that we can actually improve our outputs later on. When we get it to plan, it's going to actually go through and ask us a bunch of different questions and create, as it sounds, a plan that we can approve or actually make changes to. The reason this matters so much is that it forces Claude to break the work down before it touches a single file. It's all about context, and you'll hear me say that again and again in this video.

A clean plan written to a file lets Claude Code just load that relevant chunk of the context and the information instead of trying to hold everything in its head at once — the same way we as humans plan out tasks: we break them down into subsets of context so that we can go back, revisit, and build from that plan. The order of working should always be: plan first, execute the build second. I'd literally always do this if it's more than just a simple 10-minute task.

Concept · Plan mode

Plan mode (Shift+Tab twice in the Claude Code terminal) puts the agent into a read-only, no-file-writes state. Claude can read, search, and ask clarifying questions, but cannot take any action that modifies the file system. This forces a planning pass — usually a written plan file — before execution begins, which dramatically reduces mid-task direction changes and wasted tokens.

Applied economist lens

Before starting any new Companies House extraction batch, open Claude Code in plan mode and paste your brief: "We need to extract turnover, operating profit, and headcount from the latest filed accounts for these 20 company numbers." Claude will ask clarifying questions — output format? how to handle restated figures? what if headcount is missing? — and produce a structured plan you can approve. That approved plan then lives as a file (plan.md) and becomes the context anchor for the entire run, meaning you can claude --resume mid-session without re-explaining the method. This is the AQuA quality-assurance discipline — agree the method before touching the data — re-expressed in Claude Code terms.

Concepts 3 & 4 — Permissions and notifications

When you actually do get to the building stage, a lot of people just tell you to start with the --dangerously-skip-permissions flag, which is just going to go through and auto-approve everything. Even as it sounds, file deletion is auto-approved. When you start with Claude Code, I would advise you to start more cautiously. Try to understand what it's asking for approval for as you go through, and then you can start setting up your own permissions file later on in the settings.json file.

But here's the real time-saver: install a notification extension like CC-Notify, and it's going to ping you the second Claude needs approval or finishes a task. That one is going to speed up your workflow massively. It means you can literally set a task off, go away from the terminal, do other stuff, and come back only when Claude actually needs your inputs. It even works if you're running five or so tasks at once — you'll get the notifications from each session window. You're basically letting Claude run for longer stretches without stopping for every little thing. And by the time you're watching this video, there might even be more autonomous modes available inside Claude Code surrounding permissions, because Anthropic keeps on shipping more and more updates in this space.

Concept · Permissions and settings.json

Claude Code prompts for user approval before taking potentially dangerous actions (writing files, running shell commands, deleting). The --dangerously-skip-permissions flag bypasses all these prompts — useful for trusted automated runs but risky in development. The safer path is to build an allowlist progressively in your project's .claude/settings.json, naming specific commands and file-write patterns that are pre-approved.

Worth flagging

Never run --dangerously-skip-permissions in any directory that contains live data, filed accounts, or master workbooks. Claude Code with blanket auto-approval can overwrite or delete files without pause. Build your settings.json allowlist instead — it gives you the same low-friction experience with an explicit, auditable scope.

Applied economist lens

For your SPE salary survey compilation — where the master workbook is precious and irreplaceable — set up a settings.json that whitelists only the specific output directories Claude is allowed to write to, and keep the master workbook folder outside that scope. Then use CC-Notify to monitor overnight extraction runs across your 96-firm database: start five parallel terminal sessions (one per industry cluster), go away, and get pinged when each one needs approval or finishes. That's the same throughput as a junior analyst batch, without the coordination overhead.

Concepts 5 & 6 — Context window and context rot

Now you've got the foundations; this next section is all about achieving the highest quality outputs, which can't be done without understanding and managing your context effectively. The context window is just the amount of text that Claude can hold in its head at once per LLM model. Opus has a different context window to Sonnet, for example. And context windows generally are actually getting bigger and bigger — they've become huge, hundreds of thousands of tokens, even in some cases one million tokens, which sounds like loads, right?

But compare it to what you're actually loading. If you're just asking a back-and-forth question, it might be about a paragraph, and then you go back and forth a few more times and start adding to the context. Now you think about loading in code, files, documents, transcripts from videos, brand guides, client briefs — suddenly you start piling in thousands of lines really, really quickly. The window then is going to start to fill up much quicker than you think. And each line is costing you in usage limits and in the effectiveness of how Claude responds. And that's context rot.

Context rot is a real problem. The more tokens you load into Claude or any LLM, the more forgetful it gets. Even though the models do keep getting better, this rule still holds. There's a graph showing the different models with number of tokens loaded in — you can see that as the number of tokens increases, the rate of recall drops significantly. At roughly 7,500 words or around 10,000 tokens, you've lost about 50% of the context for most models.

Concept · Context window

The context window is the total number of tokens — a token is roughly ¾ of a word — that a model can "see" in a single session. Everything in that window (system prompt, CLAUDE.md, conversation history, files read, tool outputs) counts against the limit. Claude models currently range from roughly 100k to 200k tokens. Once the window fills, older content is either dropped or the model's ability to attend to it degrades sharply.

Concept · Context rot

Context rot is the empirical phenomenon where recall accuracy drops as context length grows. Research shows that models perform disproportionately well on content near the start and end of the context, but increasingly miss material buried in the middle. It is not a bug — it is a property of attention-based models. The practical effect: instructions you gave early in a long session get quietly ignored as the session grows.

Applied economist lens

Context rot is why your AQuA rules disappear 45 minutes into a salary-survey QA run — the rules are near the top of the context, the current extraction task is at the bottom, and everything in between (intermediate tool calls, previous company outputs) is filling the middle where attention is weakest. The fix is not a bigger context window: it is keeping each session focused on one company cluster at a time, loading only the relevant schema, and starting a fresh session for each batch. Think of it like running a tight scope on a regression: more irrelevant controls do not improve the estimate.

Concepts 7 & 8 — /clear, /compact, and claude --resume

So, what can you actually do about context rot? Well, there are three things at first, all built into Claude Code. First, you can use plan mode — we've spoken about this already; it basically keeps the important context for your plan safe so we can recall it at the right time. Next, you can use the /clear command between unrelated tasks in the same conversation window. Every single time you start a new unrelated conversation, you want fresh context, fresh output — just run that /clear command.

And if you want to consolidate the information after a long conversation — a long session where you haven't switched topics — then you can use the built-in /compact. It's basically going to summarize everything so far so Claude keeps the important stuff, and it's going to get rid of all the noise in that conversation. You can even tell it which specific context to keep. Now, those three aren't perfect — you still have some of the same issues — but you can get much better context management through external memory systems, which we'll cover later.

It's also important to know that Claude saves the context from the session. Every conversation you have is being stored in the memory of Claude sessions. And the --resume flag is all about restoring that context. You can run claude --resume, pick the conversation you were literally on yesterday, and then you're right back in that conversation. Claude's going to remember exactly what you were building, what you told it, and also what activities are still left — especially if you have a plan file that coincides with that. This is all about speeding up your workflow: you don't need to re-explain again and again everything.

Concept · /clear and /compact

/clear wipes the conversation history entirely — useful when switching to a completely unrelated task in the same terminal session. /compact calls the model to summarise the current conversation into a condensed version, keeping only what matters for continuing the work. You can add an instruction like /compact keep the schema and QA rules to preserve specific context. Neither is a substitute for starting a fresh, tightly scoped session, but both buy you more usable runway within a session.

Concept · claude --resume

claude --resume reopens a previous session from Claude Code's stored session history. Claude Code persists every conversation locally; --resume lets you pick from a list and continue exactly where you left off, with the same context window state. This is distinct from /compact: --resume restores a session, it doesn't compress it.

Applied economist lens

For a multi-day Companies House deep-dive on a single firm — reading multiple filing years, cross-referencing notes to accounts, building the ratio table — start each morning with claude --resume to pick up yesterday's session rather than re-pasting the company brief and schema. Use /compact keep the filed-accounts schema and the current company's extracted rows before you end each day so the session stays lean. Use /clear when you pivot from that firm's financials to writing the Substack section — that's a genuinely different context that should start clean.

Concept 9 — CLAUDE.md: table of contents, not a bible

Now you're probably starting to see a little bit of a pattern: everything in Claude Code comes back to context. CLAUDE.md — the main file that's often referenced — is loaded at the start of every session as part of Claude Code's system prompt. So whatever you put in the CLAUDE.md file, it's always going to be read, which means it has to be concise to not bloat your context.

Your CLAUDE.md should not be a 2,000-line bible of everything you've ever thought about this specific project. It should actually be more like a table of contents that points to other files. You're going to put in there specific rules, and then when you've got additional context or additional information, you can just point to a reference file elsewhere that Claude can load in and unload — importantly, whenever it actually needs them. It will effectively load in less than 200 lines and then be able to load additional context exactly when it needs it based on your question.

Concept · CLAUDE.md

CLAUDE.md is a markdown file placed at the root of a project (or globally at ~/.claude/CLAUDE.md) that Claude Code automatically reads at the start of every session. It functions as the persistent instruction layer — your coding standards, project structure, tool preferences, and domain rules live here. Because it is always loaded, it counts against your context budget on every turn, which is why the table-of-contents pattern (short rules, pointers to reference files) is critical: you get the guardrails without bloating the context.

Applied economist lens

Your consultancy-research project CLAUDE.md should be around 50 lines: the 96-firm database schema location, a pointer to docs/aqqa-rules.md for QA criteria, the Companies House MCP connection name, and the output file convention. Keep the AQuA rules in a separate reference file — Claude only loads it when the task is a QA pass, not on every extraction turn. Similarly, your SPE salary survey CLAUDE.md can point to schema/survey-schema.json and docs/pii-handling.md without embedding those full documents in the session from line one. This is the same discipline as keeping your regression specification in a separate dofile rather than at the top of every script.

In practice

Global rules (your name, working style, output format preferences) go in ~/.claude/CLAUDE.md. Project-specific rules go in ./CLAUDE.md at the repo root. Both are loaded; the project file is read last and takes precedence on conflicts.

Concept 10 — Slash commands: manual, repeatable prompts

That's foundations and context sorted. Now, let's talk about the four ways you can actually personalize Claude Code to how you work. Let's start with slash commands, and slash commands are for things that you do again and again and want to manually invoke. As they sound, they're kept in a commands folder and invoked using a slash with the shortcuts you can see. For extracting YouTube transcripts from content, for example, I've got a /tool-youtube. I paste in a YouTube URL and it's going to spit out the transcript without me having to write again and again what I'm actually looking for.

If you find yourself giving the same context regularly, that's a sign to convert that into a slash command. You can think of the slash command as just a set of prompt information where you can put in dynamic arguments — like a YouTube URL, for example.

Concept · Slash commands

Custom slash commands are markdown files stored in .claude/commands/ (project) or ~/.claude/commands/ (global). Each file becomes a slash command: a ch-extract.md file becomes /ch-extract. The file contains the full prompt template; you can embed $ARGUMENTS to pass dynamic values at invocation. Slash commands are explicitly invoked — Claude does not call them automatically.

Applied economist lens

Build a /ch-accounts slash command whose template reads: "Given company number $ARGUMENTS, use the Companies House MCP to retrieve the latest filed accounts, extract turnover, operating profit, and headcount, and append a new row to data/ch-ledger.csv using the schema in CLAUDE.md." You run /ch-accounts 12345678 for each firm in your 96-company list — same context, same output format, zero re-prompting. A separate /aqqa-check slash command can trigger your outlier-detection QA pass on demand.

Concept 11 — Skills: progressive disclosure of deep context

Skills are effectively slash commands that you can invoke, but with more — you can probably guess — context. The big difference is they let you load in more context as long as it's separated properly into reference files. Skills get loaded into context using a progressive disclosure system. The name and description are always loaded in first, and Claude is going to use that to pick the skill out when it thinks that skill is relevant. And the other thing here, compared to slash commands, is that we don't need to use the slash, but we actually can still invoke it with the slash if we want to.

In practice, I just paste in a YouTube video URL and say "transcript" and it would know to use the tool-youtube skill automatically based on the description of the skill. Once it knows it wants to actually use the skill, it's going to load in the full SKILL.md file, which should be kept again less than 200 lines to preserve your context window. You can think of the SKILL.md as effectively a set of process steps — step one, step two, step three. Think of it as an instruction book on how to follow that specific task.

Now, 200 lines sounds quite small, right? But you can again point to additional reference files with additional examples, additional context in a references folder that's kept separately. For example, for our YouTube skill, it might have: step one, retrieve the YouTube video; step two, send it to a transcription tool; step three, write the transcript. And in the write-the-transcript step, we might have a specific way we like writing transcripts, which we could offload to a reference file. So when you package this all up — you've got a short SKILL.md file that gives Claude the overview of the process, with deeper reference files that are loaded into context and offloaded only at that specific step in the process. You get all of the depth of the context without bloating the context window with all the information at once. That's what makes skills so powerful.

And by the way, don't write skills from scratch. Use Anthropic's skill creator skill because it generates them properly the first time. You still have to give it a bit of guidance about keeping it less than 200 lines and offloading references, but it's effectively going to help you build that with references and examples baked into your skill there. All you need to do is describe the skill you want, and it builds out the whole thing for you. Just make sure you understand what process document you actually want — the inputs, the outputs, etc.

Concept · Skills and progressive disclosure

Skills are structured prompt packages stored as SKILL.md files, typically under ~/.claude/skills/. Claude Code auto-discovers them and uses the name plus first-paragraph description to decide when to invoke a skill without an explicit slash command. The "progressive disclosure" pattern means only the short SKILL.md (under 200 lines) is loaded initially; deeper reference files in a references/ subfolder are loaded on-demand only at the step that needs them. This keeps the context window lean while supporting arbitrarily deep workflows.

In practice

Invoke the skill creator: /anthropic-skills:skill-creator (or simply describe your skill in a new session and Claude will offer to build it). Give it your process description, expected inputs, output format, and a hint to keep the SKILL.md under 200 lines with references offloaded.

Applied economist lens

A ch-accounts-extractor skill — with a 150-line SKILL.md covering the extraction steps and a references/output-schema.md for the CSV column definitions — is more reusable than a slash command because Claude will also invoke it automatically whenever you describe a CH extraction task in conversation. Chain it with a separate aqua-qa-checker skill to build a two-step pipeline: extract, then verify. Each skill runs in its own focused context, passing only a summary result to the next — far less context rot than trying to do both in a single long session.

Concept 12 — Hooks: deterministic, token-free event triggers

Now, compared to skills and slash commands, hooks are non-AI, deterministic, token-free actions — importantly — that happen on a specific event every single time. There's no randomness, there's no guessing whether the LLM's actually going to call it. Hooks happen every time. And Claude Code has some built-in hooks like session start. And this is where it gets really useful for memory frameworks, which we'll come to later.

Session start means that you can force a specific file to load in every time a session starts, rather than relying on the CLAUDE.md where Claude is deciding using its judgment whether to read that reference file. You just guarantee it loads by using a hook instead.

Concept · Hooks

Hooks are scripts or commands wired to Claude Code lifecycle events — currently PreToolUse, PostToolUse, Stop, and Notification among others. They run deterministically outside the LLM: no tokens consumed, no model judgement involved, guaranteed execution. A session_start hook can inject a file (e.g. your memory corpus) into the context before any user message is processed, bypassing the "will Claude choose to read this?" uncertainty you have with CLAUDE.md references.

Applied economist lens

Wire a session-start hook to automatically load your 96-firm schema summary and your current-round extraction status into every new session. That way you never start a fresh session having to re-explain the project context — it is injected before your first message. A PostToolUse hook on file-write events could simultaneously log every new row appended to ch-ledger.csv to a separate audit trail, giving you a free AQuA-style change log with no extra prompting.

Concept 13 — MCP servers: connecting Claude to your tools

It brings us next to concept 13, MCP. All of the concepts so far are completely useless without being able to actually connect to the apps that you use day-to-day, and that's where MCPs — or model context protocol — actually come in. MCPs are basically a standardized way to connect to your tools and apps. That is it. They mean you can work inside the apps you actually use every day instead of being stuck in this terminal window. In a previous video, I showed Claude reading data from Notion and writing data back — that's an MCP in the background just doing the connection work and making sure it does it in a standardized way. Same goes for connecting to Slack, Google Drive, HubSpot or another CRM, your database, whatever. They're all connected by MCPs, or can be.

Concept · MCP servers

The Model Context Protocol (MCP) is an open standard that allows Claude Code to connect to external services — databases, APIs, SaaS apps — via a local server that translates tool calls into API requests. Each MCP server exposes a set of named tools (e.g. search_companies, get_filing_history) that Claude can call just like its built-in tools. MCP servers are configured in ~/.claude.json and can be scoped globally or per-project.

Applied economist lens

You already have the Companies House MCP connected — it is the foundation of your entire 96-firm ledger workflow. But MCP coverage is the multiplier: the Gmail MCP lets you draft Substack emails without copy-pasting; the Notion MCP lets Claude update your research database directly from an extraction session; and a custom SQLite MCP pointing at your local consultancy database lets you run parameterised queries mid-session without any file-handling friction. Each MCP connection is a tool Claude can call in the same session — chain CH extraction, Notion update, and a drafted summary email in a single run.

Concept 14 — UI layers: start in the terminal, extend from there

Everything we've seen so far has been inside the terminal, but does this mean you need to be technical to run Claude Code? Well, not really, because there are actually a ton of options. You can run Claude Code in the terminal or inside VS Code — it just takes a little bit of learning. You can use Claude Desktop or other UI layers to sit on top of it. Or Anthropic even brought out Claude co-work, which is effectively just another UI layout on top of Claude Code with a bit less functionality, but access to your local files. And none of these, by the way, is the one best solution — but I highly recommend starting inside something like VS Code or the terminal, because once you understand the foundations there, everything else becomes easier. Everything else is a layer on top.

Inside the terminal, you see your files, you see Claude's edits, you see exactly what's happening underneath the hood. And once you've got those basics, you can start looking at external tools that focus on solving specific jobs. For example, we've built out our own command center that's part of our Agent Academy — it's going to take away all the overhead and focus you only on business goals. So basically you can stay working at a high level, see all of the agents working in parallel in one place, and then if you want to, you can dig into individual tasks when you need to. Or look at Vibe Kanban, which is like a Kanban board for your agents, or Paperclip, which is more about setting up your agent organisation with traditional business roles — like CTO replaced by an agent. But the point is: understand the concepts first, and then work out how you extend your own capabilities with a cleaner interface on top after you've understood those foundations.

Worth flagging

Similarly, "Agentic OS" (below) is a community framework, not a built-in Claude Code feature. They are real and useful, but they come and go — always check current availability and maintenance status before wiring them into a production workflow.

Applied economist lens

For your work, the terminal or VS Code integrated terminal is the right starting point — you need to see what files Claude is writing and when, especially for any session touching your master workbooks or ledger CSVs. Once you are confident about permissions and hook setup, adding a UI layer like Claude Desktop for lighter tasks (drafting Substack sections, answering one-off CH questions) makes sense. But keep the data-extraction and QA runs in the terminal where you have full audit visibility of every file write.

Concept 15 — Agentic OS, skill chaining, scheduling, and memory systems

You've got the foundations, you've got Claude Code context management, you've got Claude Code personalisation. Now let's take it a step further — and this is where you go from using Claude Code to actually running your business on Claude Code. And it's where something like the Agentic Operating System comes in. The idea is dead simple: you build one shared business context folder that every skill references part of. You've got your brand voice, your audience, your client positioning, all the client details — all stored in that business context folder. This means you only need to update it once, and every skill is going to understand and get those updates. It's like setting up one source of truth for all of your data, but in this case it's applied to the context we're loading into Claude Code. By setting up a business operating system or Agentic Operating System, every output is going to get better because every skill knows your business, and every time you interact with Claude, it's going to inject the right context at the right time.

So you can actually take some of those jobs and turn them into scheduled tasks. This one is more of a mindset shift than an actual feature of Claude Code. I want you to stop thinking about individual process documents when you're looking at creating skills. Start thinking about, inside your business or inside your day-to-day work, what are the jobs to be done? For example, you might do a weekly content digest. That shouldn't just be one process skill that goes through all the steps of a weekly content digest — it should be a research skill, a copywriting skill, maybe a formatting skill, and then possibly even a skill to review, all chained together. Kept as separate skills, built for modularity so that you can use them in other skill combinations, but all combined together with one set prompt on a schedule to achieve a job to be done. And that's exactly how we build skill systems inside the Agentic OS, and how I'd recommend building them for yourself too.

Don't just go installing skills off the marketplace because you think they'll be useful. Think about how that skill is going to actually chain with other skills and pass it the proper outputs to create workflows instead of isolated process documents as your skills. And then when it comes to actually scheduling, you can just use Claude Code's built-in scheduling features, which are mostly inside the desktop app, so you can run these jobs on a recurring basis — your weekly content digest, your Monday lead reports, your daily check-ins, whatever your business actually needs. This is all about keeping a human in the loop, chaining multiple skills together, and not running things autonomously without supervision, because we're effectively trying to achieve high quality but reduce the amount of manual admin tasks that you need to do on a day-to-day.

Which brings us to possibly the most important concept. This last one, if you've not considered it yet, is going to totally level up your game with Claude Code. It's all about picking a memory system to make the context you load in the best it can be. Again, we've circled round full loop back to context, because it is so incredibly important. Claude's built-in memory is absolutely fine for the basics, but if you want to start doing complex projects or anything serious, you're going to need a memory system that's better than what ships out the box.

I have a complete separate video guide on this — a memory video I broke down into six levels, from the basics all the way up to advanced. Here's a quick primer: level one is the static rules and brand info that should be loaded into your memory every single time — loaded into that context. Level two is all about using hooks to force the context to actually load in reliably. Level three makes everything you've ever told Claude searchable, so it can find the right bit of information by meaning, not by keyword. And that there is the 80/20 for most business owners. And then you've got levels four, five, and six for specific use cases: word-for-word recall of past conversations, building a proper knowledge base, or sharing memory across all your different tools — ChatGPT, Claude, and your other AI tools too. It's all about picking one that matches what you're trying to remember and the way you like working.

Concept · Agentic OS (community pattern)

The "Agentic Operating System" is a community pattern — popularised by Simon and others — not an official Claude Code feature. The core idea is a shared business-context/ folder containing your brand voice, domain rules, and project-specific reference files, which every skill in your library imports from. This creates a single source of truth: update one file and every skill that references it immediately benefits. It is architecturally equivalent to a shared configuration layer in a software monorepo.

Concept · Memory system levels

Claude Code's built-in session history (accessible via --resume) is level zero. Simon's six-level memory framework goes further: level 1 = static CLAUDE.md rules; level 2 = hooks to force-load context; level 3 = semantic search over past interactions (e.g. using a vector store via MCP so Claude can retrieve relevant past notes by meaning rather than keyword); levels 4–6 cover verbatim recall, shared cross-tool knowledge bases, and more. The right level depends on how much context needs to persist and across how many tools.

In practice

Skill chaining pattern: define each modular skill with explicit inputs and outputs in its SKILL.md. The orchestrating prompt (or a parent skill) passes the output of skill A as $ARGUMENTS to skill B. For a weekly data digest: /ch-refresh → outputs updated CSV → /ratio-analysis → outputs analysis file → /draft-substack-section → outputs draft markdown. Schedule the whole chain in Claude Desktop's built-in scheduler to run each Monday morning.

Glossary — terms from this episode

Plan mode
A read-only Claude Code mode (Shift+Tab twice) that prevents file writes while Claude thinks through and documents a structured plan. Forces break-down of work before execution — equivalent to agreeing the method before touching the data.
Context window
The total token budget a model can process in one session, including system prompt, CLAUDE.md, conversation history, and file reads. Currently 100k–200k tokens for Claude models; sounds large but fills faster than expected once files and tool outputs accumulate.
Context rot
The empirical degradation of recall as context length grows. At ~10,000 tokens most models have lost ~50% recall accuracy for content in the middle of the window. The practical fix is tightly scoped sessions, not a larger window.
/clear
Built-in slash command that wipes the current conversation history entirely. Use between unrelated tasks in the same session to reset context to the system prompt and CLAUDE.md baseline.
/compact
Built-in slash command that calls the model to summarise the current conversation, discarding noise and keeping key facts. You can specify what to preserve: /compact keep the schema and QA rules.
claude --resume
CLI flag that lets you reopen a previous Claude Code session from local storage and continue it with the same context window state. Useful for multi-day analytical tasks where re-explaining the brief each morning is wasteful.
CLAUDE.md
A markdown file automatically read at the start of every Claude Code session as part of the system prompt. Should be a concise table of contents (under 100 lines) pointing to reference files, not a monolithic instruction document.
Slash commands (custom)
Markdown prompt templates in .claude/commands/ or ~/.claude/commands/ that become /command-name shortcuts. Accept $ARGUMENTS for dynamic inputs. Explicitly invoked — Claude does not call them autonomously.
Skills
Structured prompt packages (a SKILL.md under 200 lines plus reference files) that Claude auto-discovers and can invoke without an explicit slash command, using progressive disclosure to load only the context needed at each step.
Hooks
Deterministic, token-free scripts triggered by Claude Code lifecycle events (session start, tool use, stop). Guaranteed to run regardless of model judgement — use them to force-load critical context or write audit logs.
MCP (Model Context Protocol)
An open standard for connecting Claude Code to external tools and services via local servers. Each MCP server exposes named tools Claude can call like built-in tools. Configured in ~/.claude.json.
Agentic OS
A community pattern (not an official Anthropic feature) where a shared business-context/ folder acts as a single source of truth for all skills in a project. Skills import only what they need, keeping each session's context lean and consistent.
Progressive disclosure (skills)
The loading strategy for skills: name and description always in context, full SKILL.md loaded only when the skill is invoked, deeper reference files loaded only at the specific step that needs them. Maximises context efficiency.
settings.json
Claude Code's configuration file at .claude/settings.json (project) or ~/.claude.json (global). Controls permissions allowlists, hook definitions, MCP connections, and other runtime behaviours.
