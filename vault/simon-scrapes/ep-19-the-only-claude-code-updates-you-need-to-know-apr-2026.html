Solving the permissions problem: auto mode and allow/deny rules

Here's the frustration that everyone had. Claude Code would stop and ask permissions for everything — every file write, every API call — and if you walked away, it just sat there waiting for you to come back. You had two choices originally: babysit it (sit there clicking approve on absolutely everything), or use the --dangerously-skip-permissions flag and hope it didn't delete anything. Neither was great, and we needed a middle ground. This quarter gave us three solutions.

Auto mode dropped on 24 March and is the one Simon is most excited about for practical daily use. It's a completely new permissions mode where a classifier reviews every single action before it runs and decides whether it's safe or not. Say you ask Claude to build out a new landing page. In the default mode, Claude would stop and ask: "Can I create this file?" Yes. "Can I write to this file?" Yes. "Can I run npm install?" Yes. "Can I start the dev server?" Yes — over and over. It turns out, from Anthropic's research, developers approve 93% of those prompts anyway.

In auto mode, Claude creates files, writes code, installs packages, and runs your tests automatically. But if it tries to do something riskier — delete a bunch of files, push to your main branch, or send data somewhere unexpected — the classifier catches it and blocks it. Claude then tries a different approach, or if it really insists, it comes and asks you. It's designed to reduce friction while maintaining safety.

In practice

Start Claude Code with auto mode: claude --enable-auto-mode, then press Shift Tab to cycle through permission modes until you land on "auto." Note: as of late March 2026, auto mode is team-plan only — Pro/Max users won't see the option yet.

Concept · Auto mode

Auto mode is an official Claude Code permissions mode (team plan, March 2026) that sits between "approve everything" and "skip all permissions." A built-in classifier evaluates each proposed action at runtime and approves it automatically if it judges it safe, or blocks it and asks the user if it judges it risky. This is distinct from the manual allow/deny rules in settings.json, which are static pattern-matching — auto mode reasons about the action in context.

If you're on Pro or Max — and therefore can't access auto mode yet — there is a workaround. You can set up your own allow and deny rules in your settings file. You allow Claude Code to read files and folders, run dev-server tests, do formatting, edit and write files. But you stop it from installing packages, deleting files, sending data externally, and reading sensitive files and secrets. It's definitely not as smart as auto mode's classifier — it won't reason about whether an action is safe; it's a blanket yes/no to certain commands — but it'll get you 80% of the way there, and you can use it right now on any plan.

Concept · Settings / permissions (allow/deny rules)

Claude Code's settings.json (in .claude/ or your user directory) lets you define explicit allow and deny rules for tool calls by pattern. A rule like {"allow": ["Read(*)", "Write(*)"]} means Claude Code can read and write files without asking — but anything not matched still requires approval. This is static and literal: it cannot reason about context, only match patterns. Use it as a baseline safety net that complements auto mode when auto mode is unavailable.

Applied economist lens

Deny: any shell command that touches external network calls beyond the Companies House API, any deletion of the master workbook, and reading of .env files containing API keys. This lets the pipeline run unattended overnight without needing auto mode — and the deny rules are your AQuA-style guardrail against accidental data loss.

The other two key updates for running tasks autonomously are loops and scheduled tasks, which Simon covered in a previous video. Loops let you run recurring prompts inside your current session for up to 3 days — anything you want to auto-repeat within a running session. Scheduled tasks started as local only inside the Claude Code desktop app, but this quarter Anthropic also shipped cloud-based recurring tasks. You can now set up a repo, a schedule (daily, weekly, monthly), and a prompt — and it runs whether your machine is awake or not. Combine auto mode, loops, and scheduled tasks and you can run real work without watching: Claude Code handles the busy work.

Concept · Loops

Loops is a Claude Code feature that re-runs a prompt on an interval inside the current active session (maximum 3 days). Auto Dream: fixing Claude's decaying memory

Claude Code can now run on its own, but there's a problem that builds up the longer it runs — and you might have faced this if you've used Claude Code for a while. That problem is memory. The more sessions you have, the messier Claude's notes about you and the work you've done get.

Previously, Claude Code had something called auto memory. As you talk to Claude, it starts understanding your project preferences, build commands, your architecture decisions, and debugging solutions, and it captures that memory for about a month of previous sessions. That's been brilliant — until you've done about 50 sessions, and those notes are full of contradictions, stale references from 3 weeks ago, and the information keeps changing over time.

Concept · Memory tool / auto memory

Claude Code's memory tool writes observations across sessions to a memory.md index file (and associated files), auto-loaded at startup. This is distinct from CLAUDE.md, which you curate manually. Auto memory populates and updates these files automatically as you work. The tradeoff: it accumulates contradictions and stale entries over time — hence the need for Auto Dream (below).

To tackle that, Anthropic shipped something called Auto Dream. The name is deliberate — it's modelled on how your brain consolidates memories during sleep. Between your sessions, a background subagent kicks in and reviews all of Claude's memory files. It does four things: it converts relative dates to absolute dates (so "yesterday" becomes "on 30 March"); it deletes contradicted facts; it merges duplicates; and it prunes stale notes that reference files or bugs that don't exist any more.

It keeps your memory.md index under 200 lines, because that file loads at startup and every token spent on memory is a token you can't spend on actual work. Auto Dream is designed to keep it lean so you can reference a small amount of context and keep it manageable. It triggers automatically after 24 hours and five sessions — you don't even have to do anything to benefit from this.

Concept · Auto Dream

Auto Dream is an official Claude Code feature (Q1 2026) that runs a background subagent between sessions to consolidate memory files. Its four operations — absolute-date conversion, contradiction deletion, duplicate merging, stale-entry pruning — are designed to keep the memory index lean (under 200 lines) and accurate. It triggers automatically after 24 hours and 5 sessions. There is a /dream slash command for manual invocation, but as of late March 2026 it is not reliable for all users; the natural-language instruction "Consolidate my memory files" runs the same process.

Applied economist lens

Without Auto Dream-style consolidation, entries from a March session ("Baringa HQ is Paddington") can sit alongside a newer November entry ("Baringa moved to Farringdon") — and Claude will be confused. With Auto Dream running between sessions, contradictions get resolved, and the memory the pipeline uses to match firms to their Companies House records stays clean. The practical habit: after a major pipeline run, also say "Consolidate my memory files" to force an immediate pass.

Concept · MCP (Model Context Protocol)

MCP is an open standard (backed by Anthropic) for connecting external tools and data sources to Claude Code. An MCP server exposes typed tools (functions Claude can call) so Claude can reach systems beyond the filesystem: databases, APIs, email, calendar. The Google Workspace CLI is a single MCP-compatible CLI that bundles Google Drive/Gmail/Calendar/Docs integrations with pre-built recipes, replacing the need to wire up separate MCP servers for each Google service.

Skills 2.0 — The skill creator now includes a built-in evaluation system. You can test your own skills against specific criteria and improve them at a much faster rate. You get scored results back and can AB test with different versions against each other, or test with certain reference files removed from your skills. Concept · Skills & Skills 2.0 evaluation

A skill is a SKILL.md file with YAML frontmatter (name, description) plus optional bundled scripts and reference files. It packages a reusable workflow Claude invokes by name. Progressive disclosure means only the name and description sit in context until the skill is invoked; the body loads on demand — the key scaling property for large skill libraries. Skills 2.0 adds an evaluation harness: you define test cases and scoring criteria, run the skill against them, get scored results, and can A/B test different skill versions automatically.

Concept · Context window & model selection

The context window is the total number of tokens (roughly: words + code) Claude can hold in active memory for a single session. A 1-million-token window for Opus 4.6 means a 750,000-word document (or a very large codebase) can fit in a single session without chunking. In practice, performance degrades at very high utilisation — Auto Dream and lean CLAUDE.md files exist precisely to avoid burning context on stale metadata. Use the large window for genuine one-off deep analysis; for routine work, keep context lean.
