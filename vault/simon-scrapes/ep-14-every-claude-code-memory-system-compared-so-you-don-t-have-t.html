Concept · Storage vs. retrieval — the two axes of memory

Simon's framing is the most useful lens in the episode. Every memory system varies on (1) storage format — plain markdown files you can read, or dense vector embeddings you cannot — and (2) retrieval mechanism — always-on injection at session start, hook-triggered injection at prompt-submit, or explicit on-demand search by the model. Higher levels improve the retrieval sophistication; the storage format determines portability and readability. Choosing between systems is really choosing where on these two axes you need to be.

Applied economist lens

Think of these two axes against your own work. Your AQuA QA rules and the 96-firm database schema are small, stable, high-priority facts — they belong in CLAUDE.md (always-in-context, plain markdown, readable). Your per-company notes from Companies House extractions are numerous, irregular, and only occasionally relevant — they belong in a hook-injected semantic store (level 3). Your archived salary-survey decisions from 2019 are verbatim-critical (you need the exact wording of a methodological choice) — that is a level 4 / Mem Palace use case. Mapping your data types to these axes before choosing a system will save you installing something heavier than you need.

Level 1 — the native Claude Code memory stack: CLAUDE.md and the memory tool

You can think of level one as the native stuff that ships inside Claude Code. Most of you have already heard of the CLAUDE.md file; some of you have heard of memory.md; but very few people are actually using both together properly. So let's do CLAUDE.md first. As most of you know this one already, I'll keep it quick: it's effectively a plain markdown file inside your project folder where you store your rules, stuff about your brand, things about how you work with clients, coding style. It's always loaded into every single session you pull up inside the terminal. You can think of it like a note that sits at the start of the context — almost like a system prompt that you're giving to Claude.

You can actually have this at the Claude project folder level, to pass to all of your Claude instances. You can have it slightly lower at the root folder level if you have multiple projects that sit within the same folder. Or you can even have it at the individual project level for specific project instructions. The lower down the folder structure you go, it still inherits the parent CLAUDE.md, but any conflicting rules — it will always take the local CLAUDE.md as gospel for specific project-level instructions.

Concept · CLAUDE.md hierarchy

CLAUDE.md files are hierarchical: user-global (~/.claude/CLAUDE.md), then project-root, then sub-folder. Each level inherits its parent and can override with local rules. This lets you share universal preferences (coding style, safety rules) globally while keeping project-specific instructions scoped — a clean separation that prevents one project's schema from leaking into another. The hierarchy is not a plugin or add-on; it is how Claude Code works by default.

But here's the thing that most people actually get wrong with their CLAUDE.md — and I've seen this a lot — which is stuffing too much into it. So putting in your full brand guide, your tone of voice document, your complete client list. Think about when Claude actually starts a session every single time: it's going to burn through that context window reading all of that. So yes, it's important context, but only when you load it in at the right time. And this is exactly the problem we're trying to solve, called context rot — the inability for AI and LLM models to actually recall 100% of the information that we've loaded in, as we increase the amount of context that's loaded in the first place.

So as a general rule of thumb, keep your CLAUDE.md under 200 lines. And then if you've got more context — like a brand voice document — you can put that in a separate external file that's referenced from the CLAUDE.md, and therefore it will only pull it in at the right time when it needs it.

Concept · Context rot

Context rot is the empirical finding that an LLM's reliability at retrieving the right instruction falls as the input grows — not because information is deleted from the window, but because the signal-to-noise ratio degrades. The practical rule Simon offers: treat CLAUDE.md as an index pointing to separate files, not a dump of everything you might ever need. Quality and brevity of loaded context beats volume.

Applied economist lens

Your CLAUDE.md for the consultancy-research repo should contain the 96-firm database schema (a compact table definition), the five AQuA QA rules you apply to every extraction, and a reference to .claude/memory/companies-house-protocols.md — not the full protocol document itself. When Claude needs the detailed extraction procedure it will load the protocol; every other session just carries the lightweight index. This keeps your daily extraction sessions under the 200-line threshold while ensuring the full methodology is available on demand.

Concept · Hooks (session-start)

Concept · Short-term / long-term memory architecture

The OpenClaude three-layer design maps intuitively to human memory: daily notes are short-term (recent, loaded automatically), memory.md is long-term (durable, always present), and the "dreaming" process is consolidation (promoting what recurs, discarding what ages out). This structure avoids the failure mode of a flat markdown file that grows indefinitely — instead the long-term store stays compact because promotion requires repeated signal. MemSearch ports this architecture into Claude Code while adding semantic vector search on top.

And that's exactly what this is designed to do. So they've extracted OpenClaude's memory architecture into a standalone library that plugs straight into Claude Code with a two-line install.

So it's got all the benefits of the OpenClaude method, which is the markdown-first philosophy — everything we can go and see and read through and port to different systems. It's got the same chunking strategy as OpenClaude, the same file structure with the memory.md and the daily notes. But now it works inside Claude Code too. Concept · Semantic vector search vs. keyword search

Keyword search finds a document if it contains the exact words you type. Semantic (vector) search converts both the query and stored text into embedding vectors — high-dimensional numbers representing meaning — and retrieves documents that are semantically close even if they use different words. In practice: a keyword search for "revenue trend" will miss a note that says "top-line growth trajectory"; semantic search will find it. This is the core upgrade from level 2 to level 3, and why it handles months of accumulated, varied notes much better than a flat markdown index.

Concept · MCP-based retrieval vs. hook-injected retrieval

Claude Mem uses MCP (Model Context Protocol) tools for retrieval — meaning Claude must actively call a tool to search the memory store. Glossary — terms from this episode

CLAUDE.md
A plain markdown file auto-loaded into every Claude Code session as persistent context. Hierarchical: user-global, project-root, sub-folder. The primary mechanism for giving Claude standing instructions, schemas, and house rules. Keep under ~200 lines; use it as an index pointing to external files, not a dump of everything.
Memory tool (auto memory)
The model-managed memory system built into Claude Code, distinct from CLAUDE.md. Claude writes to it autonomously, storing session feedback and project notes in a hierarchical directory rooted at .claude/memory/. Inspectable via /memory in the terminal.
Context rot
The empirical degradation in an LLM's ability to retrieve the right instruction as the context window grows — not because information is deleted, but because the signal-to-noise ratio falls. The root problem all six memory levels are designed to mitigate.
Hooks
Shell commands the Claude Code harness runs automatically at lifecycle events (SessionStart, PreToolUse, PostToolUse, Stop, etc.), configured in settings.json. Deterministic: the model cannot skip them. Used in levels 2 and 3 to auto-inject memory indexes and semantic search results.
MemSearch
A Claude Code plugin by Zilliz that ports OpenClaude's memory architecture (memory.md long-term + daily notes short-term + dreaming promotion) into Claude Code, adding semantic vector search and a user-prompt-submit hook for automatic injection of top-3 relevant memory chunks before every query.
Semantic vector search
Retrieval by meaning rather than keywords: both the query and stored documents are converted to embedding vectors, and the system returns semantically similar results even if the exact words differ. The key upgrade from level 2 (keyword) to level 3 (semantic) search.
Mem Palace
A local RAG system that stores every conversation verbatim — in a symbolic AAAK index plus SQL and ChromaDB backends — and retrieves exact text on demand via nested wing/room/drawer pointers. Highest recall fidelity; opaque to direct inspection unlike markdown-based systems.
RAG (retrieval-augmented generation)
The pattern of retrieving relevant documents from an external store and injecting them into the model's context before generating a response. Mem Palace and Open Brain are both RAG systems; they differ in what they store (verbatim conversations vs. all-tool memories) and where (local vs. Postgres).
LLM Wiki (Karpathy pattern)
A community pattern for building a living markdown wiki on a topic: a human-curated raw/ folder of source documents plus a Claude-owned wiki/ folder of synthesised, cross-linked entries. A knowledge-base tool (for content you consume), not operational session memory.
Open Brain
A community cross-tool memory pattern by Nate Jones: a single Postgres database on Supabase with one "thoughts" table, accessed by any AI tool via an MCP server. Enables ChatGPT, Claude Code, Cursor, and future tools to share the same memory. Owner-controlled; ~£0.30/month on Supabase free tier.
Mem0
A funded, production-grade cross-tool memory SaaS (mem0.ai). Sub-minute setup; universal self-improving memory layer; but data lives on their servers rather than a user-owned database — the key tradeoff vs. Open Brain.
Dreaming (OpenClaude)
A background process in the OpenClaude architecture (ported by MemSearch) that reads through daily notes, scores recurring themes, promotes them to long-term memory.md, and discards stale material. Keeps the long-term store compact and signal-dense over time.
Kairos
An unreleased Anthropic-internal daemon referenced in leaked Claude Code source — described as an always-on background process that continuously monitors a project, decides what to remember, and consolidates notes. Not available in any public build as of April 2026.
