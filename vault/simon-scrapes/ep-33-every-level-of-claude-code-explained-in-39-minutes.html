Every Level of Claude Code Explained in 39 Minutes

39:22 runtime 173K views Published 7 Feb 2026 youtu.be/Y09u_S3w2c8

The seven levels of Claude Code aren't about using the tool harder — they're a systematic progression from "type and hope" to fully autonomous pipelines, and knowing which level you're actually at is the first step to reaching the next one.

What this episode covers

Simon walks through all seven levels of Claude Code proficiency, using a social media content creation system as a running example. Level 1 is planning before building (plan mode + ask-user-questions). Level 2 is giving Claude persistent rules via CLAUDE.md. Level 3 introduces slash commands, skills, and hooks — reusable prompts, background context, and deterministic triggers. Level 4 connects Claude to external apps via MCP servers. Level 5 adds the GSD framework for phased, state-tracked projects. Level 6 runs teams of subagents in parallel — for quality and speed. Level 7 is full autonomy: a Ralph loop running an unattended pipeline until acceptance criteria are met.

Why it matters for an applied economist

Most analytical work sits at levels 1–2 by default — a single context window, no persistent rules, no reuse. This episode maps a clear upgrade path for Leath's real workflows: CLAUDE.md pins AQuA rules across sessions; slash commands package the repetitive "extract this year's survey data" prompt; the Companies House MCP reads filings directly; subagents parallelize the 96-firm extraction while keeping context clean; and the GSD verify gate is structurally identical to AQuA model assurance. By level 7 you have a research pipeline that runs while you sleep.

Concept · The seven-level framing

Simon's seven-level map isn't an official Anthropic taxonomy — it's his synthesis of Boris Cherny's writings and community practice. It's useful because it names the qualitative jumps: from reactive prompting, through persistent context, through reusable automation, to full autonomy. The underlying official features are plan mode, CLAUDE.md, slash commands, skills, hooks, MCP, subagents, and the Ralph loop — each covered in its own section below.

Level 1 — planning before building (plan mode)

Let's quickly make sure it's set up in our environment. We're going to come into a fresh instance of VS Code, open a folder, and open a new terminal. We're going to run the native installer that doesn't require Node.js inside our terminal, and that's going to make sure that Claude Code is installed for us. And we're going to start running it just by running Claude with the basic permissions. Once Claude opens, it's going to look a little bit like this.

Now, level one is actually prompting but with intent. Most people just type and hope at the start. But the difference between somebody who's a beginner with Claude Code and someone who's going to get real results is one single habit, and that is planning before building. Plan mode, which you can turn on by hitting Shift and Tab and cycling between the modes — you can see down here we've got "accept edits," which is the default mode, and plan mode is a read-only mode. It's designed to research your codebase and propose a plan before you touch anything. So, whether you've got an existing codebase and it will go through and read those files, or if we're starting from scratch and giving it some context brief, it's really critical that we start with planning mode.

Boris Cherny, the creator of Claude Code, has strong opinions about using planning mode. If his goal is to write a pull request, he will use plan mode, and he'll go back and forth with Claude until he likes the plan — it does not take any execution actions until a plan is made. From there, he switches into auto-accept edits mode and Claude can usually one-shot it. So it saves going back and forth: you only need to one-shot prompt for the execution once you've got a clear plan.

The reason this mode is so powerful is because it leverages an inbuilt ask-user-questions tool inside Claude Code. Claude will literally come back with a series of questions to understand the underlying detail and the assumptions that we're making in our plan, therefore building out a really comprehensive set of context that it can use in the execution phase. So we're going to look at an example today of a content creation system — and how we can use Claude Code to build us a system that creates all of our content for us. If I put in something vague during plan mode, like "help me build out a social media content creation system," we know that's not going to execute very well or it will veer off course and not meet our assumptions, because we're not actually specifying things. By doing this in plan mode, we can force Claude to come back with the ask-user-questions tool, ask us a bunch of questions, and build out a plan.md file which, when it executes on it, will understand in detail exactly what we want to do.

And we can see down here: which social media platforms do you want to create content for? We click that, it will come up with the next set of questions — about the calendar, the automation, the brand voice — and then we submit and it will create a plan for us and ask for any additional assumptions before we start executing anything. So level one users make sure that they're planning before they execute.

Concept · Plan mode

Plan mode is a read-only mode in Claude Code (toggled with Shift+Tab) that prevents any file edits or command execution. Claude investigates the codebase, asks clarifying questions via the built-in ask-user-questions tool, and proposes a written plan. You approve or refine the plan before switching to execution mode. The upshot: one short planning pass eliminates most of the back-and-forth later.

Applied economist lens

For analytical work, plan mode maps directly onto the scoping stage of the AQuA Book lifecycle. Before you ask Claude to process 25 years of SPE salary data, run it in plan mode first: "I want to extract year-on-year wage growth, flag outliers beyond 2 s.d., and produce a clean panel with no duplicate respondent IDs." Claude will surface ambiguities — what counts as a duplicate? how do you want the 2014 methodology break handled? — before a single cell is touched. That conversation is your AQuA model-design record, essentially for free.

Level 2 — persistent rules with CLAUDE.md

Now that you've got a plan, you need to make sure that Claude Code isn't going to put out generic output within that plan and is going to work in a way that actually saves you time and not cost you more time in the end. That brings us to the CLAUDE.md file and the importance of having one for your team.

Claude, like a co-worker, needs to actually understand your rules and the way you want to work with it. We use CLAUDE.md to show that. It's a file that tells Claude how you work — you think of it like literally onboarding a new team member. You write down your tech stack, your preferences, the mistakes you don't want repeated, and Claude reads the file every time you start a session. And this is how Boris himself uses it. Their team shares a single CLAUDE.md file. They keep it version-controlled on Git, and the whole team contributes lines to that file. They'll even go to the point where they'll tag it on GitHub, and it will make the changes automatically to tell it what mistakes it shouldn't make again.

There are lots of opinions on how you should write your CLAUDE.md file and what should be included, but the golden rule is: keep it short, keep it specific, and only include what Claude can't figure out for itself. We've got a very simple framework that you can follow to set up your CLAUDE.md file for any project. And we're going to talk about how we do that for building a social media content machine that sounds exactly like your brand voice.

Concept · CLAUDE.md

CLAUDE.md is a plain-text file Claude Code automatically reads at the start of every session, making it the primary mechanism for persistent project context. There are two scopes: a global file at ~/.claude/CLAUDE.md that applies everywhere, and a per-project file at ./CLAUDE.md that overrides or extends the global one. It is human-curated (not written by Claude), version-controlled, and kept deliberately short — it is not a dump of everything you know about the project.

Question one — "what is this?" — should be a single line to tell Claude exactly what it's doing. It's not your brand manifesto. "This project generates daily social media content across various platforms in a brand voice, pulling topics from a content calendar in Airtable." That's it. Claude can then understand exactly what its purpose is.

Question two is "how do you run things?" — these should be the exact steps: the content calendar lives in Notion; we pull topics from the "ready to write" view; generated posts go into a new folder; image assets go in a separate folder. Question three is brand voice, your rules, your non-negotiables. What patterns do you follow? You can use emphasis on words like "important" where you need Claude to follow certain instructions precisely. But the better thing here is not to write all of this directly in CLAUDE.md — it's to separate a brand-voice document and then reference it. This brings us to the "don't dump" trick: keep CLAUDE.md super lean, tell Claude where to find the details. So: "For full brand voice guidelines with examples, see /docs/brandvoice.md." Question four is where we add things about mistakes — the counterintuitive edge cases. Like, our Airtable or Notion might have both a "status" and "post status" column — which one should it use? These are the things you'd put for a trainee employee. And then the final part is "how do we work?" — naming conventions, never publishing or scheduling anything directly, and so on.

We're going to keep it simple and give it the project instructions: "This project generates daily social media content across LinkedIn, X, and Instagram in my brand voice pulling topics from a content calendar in Airtable." Then we write a LinkedIn post about a topic and this is the equivalent of not telling a new employee how we want the outputs. As we understand how it performs, we add rules that help clarify the direction. After adding some simple rules — never use em dashes because that signals AI-written text, always replace "Antigravity" with "AG," never use technical jargon — we add those to the project rules inside CLAUDE.md. We need to restart for these to take effect, so we hit /clear. And now it's understood and ingested the rules by reading that CLAUDE.md file. A sense check: can I read this CLAUDE.md in 60 seconds? Am I under 30 instructions? Did I include specific examples, not vague rules? Would a human freelancer taking over find this useful?

If you're already inside a project and want to set up a CLAUDE.md file, you can run /init and it will go through your project repository and create a CLAUDE.md file for you that you can then go back and edit. If you're starting fresh, just copy the template and fill in the blanks as you build. Like Boris, fix mistakes in real time — when something goes wrong mid-session, you can just tell Claude "add this rule to my CLAUDE.md file" and it will do it for you.

Applied economist lens

Your 96-firm consultancy database has exactly the "counterintuitive edge cases" CLAUDE.md is built for. A lean project CLAUDE.md for that research might read: "Companies House filings report in £000s unless stated otherwise — always multiply by 1,000 before writing to the database. Operating profit is taken from note 2, not the face of the P&L. Where a company has multiple SIC codes, use the primary one in the first line of the filing." Three lines that prevent three categories of systematic error across all 96 firms. Add /compact keep the CLAUDE.md rules and the unprocessed company list as a safe word for long sessions.

In practice

Bootstrap a CLAUDE.md for an existing project: /init
Add a rule mid-session: just say "Claude, add this rule to my CLAUDE.md: never write operating profit in £000s without a note."
Verify it survives a restart: /clear then re-ask.

Level 3 — slash commands, skills, and hooks

So at level two, Claude is now following your personalisation and your rules. But Claude may be following your rules, and we're still having to start fresh every single time, typing the same things over and over — like "create me a LinkedIn post" — to get it to perform actions the way we want. Level three is all about slash commands and skills.

You can just think of slash commands as saved prompts that you can reuse with one keystroke — tasks that you're doing again and again that you can manually trigger at specific moments. For example, we might say: "Write me three LinkedIn posts about this topic." All we'd have to do is type /linkedin-post and inside the command the prompt says "Write me three LinkedIn posts about this topic" — and we pass in the topic name dynamically when we invoke it. The way we do this is by creating a new folder inside our explorer called .claude and inside it a commands subfolder. Then inside that, we create a new markdown file — let's call it linkedin-post.md. It's a markdown file inside that commands subfolder, and we've got an example command there: "Write three LinkedIn post variations about {{arguments}}" — where $arguments is where we pass our dynamic values. Then we've got rules specific to this command: pull brand voice from /docs/brandvoice.md, start with a hook then a line break then the body, end each post with a question or CTA, keep each post under 200 words, no hashtags, and save the drafts to an outputs/drafts folder.

Now, if we just clear our memory — which resets the Claude Code instance — and start typing the name of the command, we'll see it appear in the list as soon as we hit slash. We use the command /linkedin-post Antigravity versus Claude Code and it's going to pass that as the argument, write three LinkedIn post variations, follow those rules, and add them to a draft folder. So instead of typing this command every single time I want to create a post, I'm just using the slash command and passing my arguments.

Concept · Slash commands

Custom slash commands are markdown files stored in .claude/commands/. Each file's content becomes the prompt template, with $arguments as a dynamic placeholder. When you type /command-name some text, Claude Code loads the file and substitutes the arguments. They are manually invoked — you decide when to trigger them. Good for tasks you call deliberately (generate content, run a QA check, produce a summary).

We gave an example of writing three LinkedIn posts, but you can turn any task you do on a daily, weekly, or monthly basis into one of these slash commands. The importance here is that you want to manually invoke or trigger it at certain moments. This is where skills come in — and skills are slightly different but still represented as slash commands. You can think of skills as an upgraded version of slash commands. They're effectively background knowledge that's loaded in automatically only when it's relevant. So you don't need to trigger them — although you can tell Claude to use a specific skill. Claude just knows from the project description and the skill description when it needs to use that skill specifically.

And unlike commands, which are fairly basic — a single file like linkedin-post.md — skills can include a whole folder of supporting files. For our content creation example that might be posts, style guides, brand voice, reference docs. So it gives a richer set of context for Claude Code to work from. Skills are effectively repeatable actions with context. And you don't need to create all the skills yourself — you can get them from anywhere. They're again going to sit inside the .claude folder. So if I click on the .claude folder and create a new folder called skills, they sit under there with their skill.md file for specific skills. You might have a brand-voice folder, and inside that a skill.md file with supporting files.

We're going to look for a humanizer skill on Skills MP that takes our writing and creates more human-like and less AI-slop-like written content. We browse by category — content and media — and find a humanizer skill that removes signs of AI-generated writing from text. We open the GitHub repository and clone it into our own skills folder. And if we go into the skill.md file, this is effectively the context for this skill. When Claude Code is understanding how to write a post, it's going to look at this and absorb the context only when relevant. It talks about personality and soul, how to add voice, how to add opinions, and gives clear examples — a very comprehensive file that we didn't need to create ourselves to achieve human-like writing quickly.

Concept · Skills

A skill is a folder under .claude/skills/ containing a skill.md file (with YAML frontmatter: name and description) plus optional supporting files. Claude Code reads only the name and description at startup — the full skill body loads only when the skill is invoked (by name or automatically, when its description matches what Claude is doing). This is progressive disclosure: a large skill library doesn't bloat your base context. Supporting files (examples, reference docs, style guides) load alongside the skill body when needed.

Here's the key thing with a skill: you need a really precise description for Claude to understand exactly when to use it. Here we've got "remove signs of AI-generated writing from text; use when editing or reviewing text to make it sound more natural and human-written." Claude will understand when we ask for a human-like article to use this skill in its process. We can invoke it manually — but it will also trigger automatically. And to reinforce that in the CLAUDE.md file: "Every time you write some content, make sure you use the humanizer skill in /skills/humanizer." So now it will understand both globally and for this specific command that it's going to use that skill when making human-like writing.

You could go and export entire libraries of content and media skills — thousands of them — but I prefer to control exactly what skills I'm getting and pull in just the specific ones I'm looking for, so I know when I'm invoking certain things and not blending too much. You can do things like front-end design, SEO review, email drafter, absolutely anything that people have turned into a skill — and you can go through and edit those skills to your own requirements too. Running our command again — /linkedin-post Antigravity versus Claude Code — we should now have more naturally written content. "I stopped using AG last month. AG builds things fast, you describe what you want, it writes code, you ship. The problem is what happens 6 weeks later when something breaks and nobody on your team understands how it works." So, it's gone through, written the posts, humanised them, and even given us a summary of the things it changed to make them sound more human.

Applied economist lens

For your consultancy research, a companies-house-extractor skill is the natural next step beyond a slash command. The skill body would contain: the exact field names you care about (turnover, operating profit, headcount as a range from notes, auditor name), the rule about £000s units, the schema for your 96-firm database row, and a reference to /docs/ch-filing-quirks.md (a separate file logging exceptions like groups that consolidate at holding level). Then /extract-accounts {company_number} invokes the full skill context automatically — no re-explaining every session.

So we've got slash commands which you want to manually invoke for certain tasks, and we've got skills which are automatically invoked based on their description and can have a huge amount of context in them. That brings us to the final part of level three, which is hooks. Hooks don't require a brain — they don't require any LLM tokens specifically, they're just automatic triggers: anything you can do mechanically that fires off when Claude Code does something. To use them, we create a .claude/settings.json inside our project file. In our example for content creation, we might have a list of banned words that we're going to check every single content post for — and it's going to do that automatically. It doesn't require any thinking at all: just check this text against these words. It's very programmatic, and it's going to do that every single time it's invoked after we've created a LinkedIn post. Therefore, we're not relying on the LLM to actually think through this step — it just automatically does things that we know should be true: check for banned words, check word count and flag if over the limit, auto-format into a certain specific format.

The way we do that is just copy this into our settings.json. We have an outer "hooks" object. And to be honest, I never write these myself — I always get Claude to write the JSON object for me: "run this hook after this happens." It can even add it to your settings.json file. And now, whenever we write a post, this is going to run and make sure no banned words are in our output. From now on, if we've got this in our settings.json and we've restarted our instance, it will automatically run on everything. It will just appear like: "No banned words found in the output draft." So it gives us peace of mind that it's actually running.

Concept · Hooks

Hooks are shell commands configured in .claude/settings.json that Claude Code's harness runs automatically on specific events (PostToolUse, PreToolUse, Stop, etc.). They consume no LLM tokens and cannot be skipped by the model. Use hooks for deterministic, mechanical checks — a linter after every file edit, a banned-word scan after every output, a schema validator before committing to a database. The model never has to "remember" to do them.

The simple way to remember these three things: skills are how Claude thinks — it loads the skill with its context into its working memory while working, depending on the description. Hooks are what happens automatically after Claude Code acts. And commands are stuff that we want to trigger manually with a prompt fed into the context when it runs.

Applied economist lens

A PostToolUse hook on file writes is exactly what your salary-survey QA needs. Every time Claude writes a year's processed data to a CSV, the hook runs a lightweight Python validator: check row count is within ±5% of last year's submission, check no salary value exceeds £1,000,000 (catches a £000s unit error), check no NaN in the gender or region columns. That's your first-pass AQuA check running deterministically — not depending on Claude remembering to run it, not consuming tokens, not skippable.

Level 4 — connecting to everything with MCP servers

So far, we're still doing everything inside Claude Code — we're not actually taking actions in our day-to-day apps like Airtable or Notion. Level four is about connecting to everything using MCP servers. An MCP server is a Model Context Protocol server, which is effectively just a bridge between Claude Code and an external app. So instead of copying data out of Airtable and pasting it into Claude — say we have our content ideas stored in Airtable — we just get Claude to actually connect to our app automatically via the MCP server, read our content calendar, and then create posts based on our ideas in that content calendar.

Let's bring it to life by showing you how to set this up with Airtable. We have this MCP command. And you don't even need to do this manually anymore — you can just ask Claude Code to set up the Airtable MCP for you. But I'm going to show you how to actually do this for a specific MCP server. There's a repository listing literally thousands of services that we can automatically connect to. We just add it to the project root: create a new file, .mcp.json, paste the code in — it's basically saying "pick up these MCP servers" — and then insert our Airtable API key. Or if you didn't want to do that, we could remove this and just use the command /mcp add Airtable, which is built in, and it will start setting up the .mcp.json file for us, taking in our Airtable personal access token.

For Airtable, we get a token from airtable.com/create/token/new, give it a name — "Claude Code" — and give it scopes: reading records, writing records, and access to the Airtable bases. We give it access to a specific base. We copy that token and feed it into the .mcp.json file. Then restart Claude Code to confirm the setup. Once restarted, we test the connection: "What Airtable bases do I have?" — and it should understand that it needs to use the Airtable MCP to actually make that request and return a list of bases. "Would you like me to explore the tables within this base? Yes, show me the tables." So we're already reading from our tables, but we can also take actions with those tables.

Concept · MCP servers

MCP (Model Context Protocol) is an open standard for connecting Claude Code to external tools and data sources. An MCP server exposes a set of typed tools (read records, write a row, search, etc.) that Claude can call. Configuration lives in .mcp.json at the project root (or user-global config). Because the protocol is open, there are community servers for hundreds of services: Companies House, Gmail, Notion, Airtable, GitHub, databases. This is how Claude Code reaches beyond the local filesystem.

If we open up the table, we've got a bunch of content ideas and Claude has pulled out all the different tables and the key fields inside those tables as well. "This looks like a content pipeline for LinkedIn carousels. Ideas flow into draft posts with a brand voice and design templates applied along the way." So it's nailed it from just the MCP understanding of the fields and the descriptions — it understands exactly the content it's looking at. This is where things get really powerful, because if we wanted to combine our content creation system with a content pipeline stored in Airtable, we could say "pull all ideas from after 20 Jan" — just one idea — and it should be able to pull that specific record from the Airtable, which is a transcript of a video. And what we're going to do is push that into our slash command to create post variations: "Take this idea and use /linkedin-post." It's going to take the idea, take the transcript, and turn it into three variations of LinkedIn posts — and we can even demonstrate adding it back into our field inside Airtable.

If we're managing our content pipeline in Airtable, we can still interact and do things with Claude Code, but then put the outputs back into the table where we work every day. And that's the power of connecting to data through an MCP server. "Update the record table with just idea three in the system message column: Everyone's talking about AI coding tools. Most of the demos are underwhelming." And now we're giving it permission to add that directly into our data. We can go and see that it's just added to the system message. It worked exactly where we're working, and it's updated that row perfectly.

At level four then, we're taking actions directly in our apps — reading the data, adding things back to the data, updating. It's gone through a full plan-and-execute loop. We're no longer doing manual updates in our apps directly, and it's using the skills we've given it as well.

You already have a Companies House MCP in your Claude Code setup. The next step is to wire it explicitly into a skill so that /extract-accounts {company_number} invokes the MCP to fetch the latest filing URL, downloads the accounts PDF, extracts the key figures into a structured row, and writes it to your SQLite database — all in one unattended run. Level 5 — the GSD framework for complex projects

But so far, we're still having to contribute the majority of the thinking time and planning effort. But why not leverage a framework like GSD to plan in high detail and move us to a supervisory role instead of somebody who's executing these things on a day-to-day basis? GSD is all about removing us from the thinking overhead. It's a framework that takes skills one step further and is designed to break down huge projects into individual phases, and execute a loop of: plan the phase first, execute the phase second, and verify using user-acceptance-testing methods that the phase has been completed successfully.

This is great for where you haven't got a really defined scope and you have a larger project you want to build out in great detail so it can be run on your behalf. When we start planning the project, it's going to leverage ask-user-questions to come through and get a full understanding of all the project scope, all the dependencies, all the assumptions, and exactly how we want to build it out. It's like planning on steroids. It's going to break down individual to-dos inside individual phases that are marked as completed as it goes through and executes on that brief.

The reason that power users use a framework like this is because a huge problem with Claude during long sessions is that we actually fill up the context window. You can see down here with this bar: we're at 50% of the maximum context window. As soon as we get towards 95%, Claude is going to automatically compress all that context into summaries and we're not going to be able to find it. This is something known as context rot — basically, as we increase the number of input tokens, our reliability on the output gets less, to the point where if we're putting in 10,000 tokens (around 7,500 words), we lose 50% of the context. A framework like GSD helps tackle this because it actually keeps context in individual files. We have overall project documents that are spun up — like roadmap, requirements, and state — but we also have context kept in specific phase-level documents with to-do-level tasks.

Concept · GSD ("Get Shit Done") framework

GSD is a community framework (designed by a creator called Tash — not an official Anthropic product) that imposes a discuss → plan → execute → verify lifecycle. State is persisted in markdown files under a .planning/ folder: a project roadmap, a requirements doc, a state document tracking which phases are complete, and per-phase documents fed into context only when that phase is active. The key anti-context-rot mechanism is scoped context injection: only the documents relevant to the current phase load into the window. The verify step is a human-plus-programmatic user-acceptance check before the next phase begins.

You can see this demonstrated in a previous video where I built out an entire SaaS application using the GSD framework. So it takes planning to the next level. We have a .planning folder with the project roadmap telling us exactly what the phases are: foundation and authentication, data management, the AI generation pipeline, payments. We have the state document — a record of which phases have been executed and which need to be done, so it's a memory of progress against the given plan. And then we've got various other high-level documents which feed in context at the relevant points and solve the context problem. Not to mention each phase is individually broken down into a series of waves and tasks. For example, inside phase five — Stripe Billing — we have a plan of what we're going to do for the first part, a summary of what's been executed when we run the execute command, and a user-acceptance-testing file that runs through and automatically tells us what to test based on the Stripe Billing requirements.

In practice

Install and start a GSD project:

npx sketch-it-done cccc  then  /gsd-new-project → /gsd-plan-phase → execute → verify

Applied economist lens

GSD's lifecycle is structurally the same as the AQuA Book analytical cycle: scope → design → build → assurance. Run your next Substack firm-profile write-up through it: discuss (nail down the 10 questions the article must answer), plan (decide which Companies House filings, which OpenAlex papers, which web sources), execute (subagents draft each section in isolated context), verify (a dedicated reviewer subagent fact-checks every number against the source and flags unsupported claims). The verify gate is exactly the AQuA "independent check" step — and it's built into the framework rather than depending on you remembering to do it.

Level 6 — subagent teams for quality and speed

At level five, we've taken planning to the next level by using the GSD framework, but we're still just supervising one agent while we get on with other tasks in the background. So why not instead run a team of agents that can verify each other's work? That brings us to level six, where instead of one Claude doing everything — researching, writing, reviewing — we split the jobs into separate subagents.

Ultimately, we do this to increase our leverage. It's summarised really well in a Reddit thread. Subagents are specialised personas you offload specific types of work to. Each agent has a fresh context that is then able to be pulled into your main context window that's fed by your CLAUDE.md. So, use subagents to keep your context clean — that is, stop context rot. But also: context isolation means the subagent is not going to bloat your main agent's context. It makes your responses much faster and also cheaper because you've given it less context or isolated context. You do this particularly when you're working on a problem that requires a lot of expertise — break down a bigger project into components and let agents work on it in either sequence or parallel.

For our content creation system, this might be represented like a content research team: a researcher (subagent one), which is in-depth research with access to additional tools that the main Claude Code instance doesn't have; our main instance — the content writer; and a reviewer (a second subagent) that can only suggest edits, not make them. There are two main reasons to run a team. The first is to improve output quality. Concept · The Ralph loop

The Ralph loop is a community pattern (not an official Anthropic feature) that wraps Claude Code in a bash loop which re-submits a fixed prompt (plus the files on disk) at the start of each iteration, then checks a completion condition before deciding whether to loop again or exit. Because it starts from a clean, curated PRD.json plus the current filesystem state — rather than a growing chat history — it sidesteps context rot for repetitive tasks. Its strength and danger are the same: it runs unattended until the condition is met. Tight, well-defined acceptance criteria are non-negotiable.

We're going to set up a new PRD.json file and feed these content ideas into our PRD document — this is a well-scoped task, so it will find it hard to go off the rails and do something different. And all we need to do to run this is hit /ralph-loop. I always recommend running this with a max iterations setting because if you don't, it could potentially run on larger projects indefinitely and cost you a lot of money in tokens. Always try to run it with max iterations as well as a completion promise as a safeguard too.

Ralph is very much an executor. It works brilliantly when all of your tasks are really well defined — clear description, a clear acceptance criteria, a clear completion status. Worth flagging

Simon is explicit that Ralph works for well-scoped, mechanical tasks and breaks down on genuinely complex projects that require ongoing judgment. Always bound Ralph runs with a max-iteration cap and review the output with a fresh eye before it touches anything consequential. Don't confuse "it ran to completion" with "it ran correctly."

Glossary — terms from this episode

Plan mode
A read-only Claude Code mode (Shift+Tab to toggle) that prevents execution and instead researches the codebase, asks clarifying questions, and proposes a plan for user approval before any file is touched.
Ask-user-questions tool
A built-in Claude Code tool that presents structured multiple-choice questions during planning to surface requirements, dependencies, and assumptions before execution begins.
CLAUDE.md
A markdown file auto-loaded into context at session start. Hierarchical: global (~/.claude/CLAUDE.md) and per-project (./CLAUDE.md). The primary mechanism for persistent rules, project context, and onboarding instructions. Keep it short, specific, and human-curated.
Slash commands
Reusable prompt templates stored as markdown files in .claude/commands/, manually invoked with /command-name arguments. Good for tasks you trigger deliberately at a specific moment.
Skills
Folders under .claude/skills/ containing a skill.md file (with a name and description in YAML frontmatter) plus optional supporting reference files. Only the name and description load at startup; the full body and supporting files load on invocation. Enables progressive disclosure of rich context.
Hooks
Shell commands configured in .claude/settings.json that the harness runs automatically on specific events (PostToolUse, Stop, etc.). Consume no LLM tokens, cannot be skipped by the model. Ideal for deterministic mechanical checks: linting, banned-word scans, schema validation.
MCP server
A Model Context Protocol server: an external process exposing typed tools (read, write, search) that Claude Code can call. Configured in .mcp.json. How Claude reaches beyond the filesystem to external apps and databases.
Context rot
Degradation in model reliability as context grows — not deletion, but loss of ability to retrieve the right needle from a large haystack. At ~10,000 tokens (~7,500 words), degradation can reach 50%. The core motivation for all the frameworks in this episode.
GSD framework
A community framework (not official Anthropic tooling) imposing a discuss → plan → execute → verify lifecycle, with state persisted in per-phase markdown documents under .planning/. Only the current phase's documents load into context, keeping the window small and relevant.
Subagents
Separate Claude Code instances spawned from a main agent, each with its own isolated context window and tool set. Only the final summary returns to the main agent. Use for specialised expertise or parallel speed — keeps the main context clean.
Ralph loop
A community pattern wrapping Claude Code in a bash loop that re-submits a fixed PRD.json prompt each iteration until a max count or a "done" completion condition is met. Sidesteps context rot for well-scoped, repetitive tasks. Not an official Anthropic feature.
--dangerously-skip-permissions
A Claude Code startup flag that bypasses all tool-use confirmation prompts. Suitable for sandboxed, well-scoped runs with deterministic guardrails (e.g. a hook-based schema validator). Should never be the default on sessions with write access to production data.
Vibe coding
Accepting AI-generated output with minimal close review. Convenient for exploratory work; dangerous inside an unattended loop where errors compound silently across iterations.
