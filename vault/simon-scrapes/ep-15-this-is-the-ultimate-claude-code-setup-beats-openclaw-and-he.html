Feature 1 — a four-layer persistent memory system

Let me first start with the one that most people get wrong. And that's persistent memory. It's probably the biggest selling point of Hermes and OpenClaw. So unlike temporary chat wrappers, they're supposed to maintain long-term searchable memory of past conversations and projects. And this is what gets us hooked initially because the idea that my agent will actually remember what we worked on yesterday, last week, and therefore contextualise and be better for the results going forward sounds amazing. And if you've worked with Claude Code for more than just a few sessions, you'll know just how important context management actually is for getting high-quality outputs.

But Claude Code actually already has all the ingredients for this. We just need to set them up properly. So I've distilled this into just four memory layers that you can use. The first is CLAUDE.md — or agents.md if you want to actually use this with different models. So think of this just as your agent's operating instructions: who it is, how it behaves, what rules it follows. And it's going to load into every single session. Most of you will have this layer already.

Concept · CLAUDE.md

CLAUDE.md is an instructions file that Claude Code auto-loads at the start of every session. It works hierarchically: a user-global file at ~/.claude/CLAUDE.md applies everywhere; a per-project file at ./CLAUDE.md sits in the repo root and adds project-specific rules. Think of it as the agent's standing brief — the things it should always know, regardless of what the conversation is about. Simon uses "claw.md" and "agents.md" interchangeably, but CLAUDE.md is the official file name.

Now layer two is brand context. So this is where your business context is actually going to live. Think your brand voice, how you speak, ICP, your positioning, all your client details will live in this brand context shared folder, and therefore every skill inside the repository can pull from the same folder. So when Claude writes a LinkedIn post for you, when it's doing client research, when it's building out your landing page, it's going to pull from the exact same business brain. And as you can imagine, it gets much better outputs that are hyper-focused on your business.

Now layer three builds on top of that, and this is what makes agents like OpenClaw and Hermes and those frameworks feel so personal. So this is the agent context folder. So we have things like the soul.md and the user.md which describe how it should act and feel and the actions that you as a user commonly do, so it feels like it understands you. And finally, layer four, we have project memory. So each project you run actually keeps a history of exactly what's happened, as well as a plan that you can refer to. So when you actually come back to a content repurposing project that you did three weeks later, Claude is going to know exactly what you built, what worked, what didn't work, and where we need to pick up on that.

Concept · Memory layers vs. context window

There is an important distinction here. CLAUDE.md and reference files are disk-based memory — they persist between sessions and are only loaded into the context window when referenced. The context window itself is ephemeral — it clears at the end of a session. Simon's four-layer approach keeps the context window lean by deferring the loading of detailed reference files until the moment they're actually needed. This is the practical solution to context rot: don't stuff everything into context upfront.

Now from this four-layer framework, here's the first key learning that I wanted to share with you. It doesn't matter how you set this up. You can put your project briefs anywhere, use whatever planning frameworks you like to write the plans — we use GSD, or the "Get It Done" framework, for more complex projects, for example. But the core ingredients are that you're loading in the right context at the right time, so you're not bloating context and experiencing context rot where the outputs are getting worse the more context you feed in. So your CLAUDE.md shouldn't be 2,000 lines long. It shouldn't even be 1,000 lines long. It should actually just reference the most important process steps that are going to get loaded in at the start of the conversation. So any additional context that needs to be pulled in at a specific point needs to be stored in a separate reference file.

So now I keep my agents.md — if you want to work across models — or your CLAUDE.md, succinct and to the point, and then just point it to reference files. So see here where I've separated my brand context and tell it to load it in only when it needs it. That way, Claude can actually load and offload into separate agents and maintain high-quality outputs because we're not bloating the context. And we use exactly the same approach with every single skill. Short entry skill.md file, detailed references which are only loaded and offloaded when needed. So the first feature then is a memory layer. This is just about organising context in a way that actually suits your business and your needs. So you just need to structure it properly, segment your context, point it to reference files, and not overload those context files.

Concept · Skills

A skill in Claude Code is a SKILL.md file — typically with YAML frontmatter naming it and describing when to use it — plus optional bundled reference files. It packages a repeatable workflow: Claude reads the skill file and follows its step-by-step process document. Skills can be invoked explicitly by the user or auto-selected by the agent when the task description matches the skill's description field. Simon uses the term loosely to include SKILL.md files, slash commands, and community tooling; the official Claude Code primitive is the SKILL.md file.

So if we tackle creating skills first, Claude Code has the skill creator skill, which you might have seen before. It's built by Anthropic themselves, and basically you give it a description of what skill you want, or you point it to a GitHub repo with an existing skill, or describe a process from scratch on how you do it, and it's going to build out the entire thing — the name, the description (and that determines the success rate of whether it's called or not), and then obviously the step-by-step process document of the actual skill.md file itself. And if we reflect back on how we manage context, what we want to do is keep a refined skill.md file less than 200 lines and then strip out everything that's unnecessary. So we've actually adapted the Anthropic skill creator skill to do exactly that — strip out the surplus information and keep that separate and only load it into context exactly when you need it.

Concept · Skill creator skill

The skill creator is itself a skill (bundled with Claude Code) that generates new SKILL.md files. You describe a workflow — either in plain language or by pointing it at an existing process document or GitHub repo — and it produces a complete skill file: name, description, and step-by-step instructions. The description field is critical because it controls when the skill auto-invokes: a vague description means Claude won't call it when it should.

Now on top of creating skills, every skill you build should have a self-learning loop for additional context, and you can go one step further with a learnings.md file — or just a rules segment actually inside your skill.md — which are basically non-negotiable rules that each time you use that skill, or each time Claude is going to use that skill, it will get better at abiding by those specific rules that we've added in. So as part of the skill process, as one of the steps in the skill.md, you get it to ask you for feedback. So it effectively takes the feedback then that you give it, applies any rules inside your learnings.md or inside that skill file, and then always sticks to those rules when you use that skill again and again. So it's effectively a self-learning loop where the skills get better over time.

So inside Claude Code, you can replace that functionality by actually just using the skill creator skill, and then you make them get better by applying rules inside a learnings.md or just rules inside the skill document itself if it's asking you for feedback every time.

Glossary — terms from this episode

CLAUDE.md
The instructions file that Claude Code auto-loads at session start. Hierarchical: user-global (~/.claude/CLAUDE.md) and per-project (./CLAUDE.md). Should be concise and point to reference files rather than containing everything itself.
Memory layers
Simon's four-tier file structure: (1) CLAUDE.md operating instructions, (2) brand/project context folder, (3) agent persona context (soul.md, user.md), (4) per-project history and plan files. Each layer loads on demand, not all at once.
Context rot
Degradation in output quality as the context window fills with irrelevant content. Prevented by keeping CLAUDE.md short and loading detailed reference files only when needed.
Skills
SKILL.md files that package a reusable workflow — name, description, and step-by-step process. The description field controls auto-invocation. Keep under 200 lines; offload detail to reference files.
Skill creator skill
An official Anthropic-bundled skill that generates new SKILL.md files from a description or existing process document. Use it to bootstrap skills, then refine the description and strip surplus lines.
learnings.md
A per-skill file of non-negotiable rules accumulated from user feedback. The self-learning mechanism: each run asks for feedback, appends rules here, and subsequent runs apply them. Not an official Claude Code feature — a convention.
Channels feature
Official Claude Code capability connecting a running session to Telegram, iMessage, or Discord, so you can send tasks and receive outputs from your phone without extra infrastructure.
Scheduled workflows
Jobs that run skills on a cron schedule, chain multiple skills in sequence, and drop outputs into a review folder for human approval. Claude Code's routines feature handles built-in connectors; custom Mac/Windows scheduling is DIY.
Human checkpoint
Simon's 80/20 rule: automate 80% (research, drafting, extraction), but always require a human review step before outputs go live or into a database. Prevents compounding errors from unattended loops.
Business context folder
A shared directory of reference files (brand voice, schema, methodology, personas) that every skill references. Single source of truth — update once, all skills inherit. Simon's term, not an official feature name.
Hermes
A third-party community framework for running scheduled Claude-based agents with Telegram/Discord interfaces. Not affiliated with Anthropic. As of this episode, requires API credits (not available on Pro/Max subscription).
OpenClaw
Another community agentic framework with a "mission control" multi-agent UI. Community branding; not an official Anthropic product. Simon's comparison shows feature parity is achievable natively.
Agentic OS
Creator branding (Simon's term) for a personal system combining CLAUDE.md, a business-context folder, skills, and scheduled workflows. Not an official Anthropic term — a pattern, not a product.
