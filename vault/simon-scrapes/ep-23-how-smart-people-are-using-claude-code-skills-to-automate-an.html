Concept · Skills

A skill in Claude Code is a SKILL.md file (with YAML frontmatter containing name and description) plus optional bundled reference files and scripts. It packages a reusable workflow that Claude invokes by name. The key scaling property is progressive disclosure: only the name and description sit in context until the skill is actually invoked — then the full body loads. This means you can have dozens of skills registered without them eating your context window until they're needed.

Anatomy of a well-built skill

Let's start by talking about building skills in the right way, because once you understand this structure everything else starts to fall into place. A skill is literally just a folder made up of two things. The first is the step-by-step instructions — the process documentation — and that is in the skill.md. The second, the bit that actually makes your process yours, is the deep knowledge that supports it: the references, the assets, and the scripts that support those step-by-step instructions. You can think of those like your brand voice or example outputs that you'd put into a traditional LLM prompt. Ultimately, it's the context in these deep knowledge files — the reference files — that make the process yours.

Here's the thing most people don't realise about skills you download or import from somewhere else, like a skills marketplace: they're designed to be generic on purpose. When you grab a skill from GitHub or a skills marketplace, it's never plug and play. It always needs you to fill in the gaps with your own business knowledge. You can use them without that, but you're going to get poor quality or generic outputs. As a real example: Simon's marketing copywriting skill points to brand-context reference files in the skill.md — "go read the voice, go read the ICP about our customers, also read the samples of good outputs." All of that lives in a shared brand-context folder, and every single skill can access it when it needs it. When the skill runs it's not guessing how Simon sounds or who he's writing for — it already knows, pulling from real files.

Concept · CLAUDE.md & reference files

CLAUDE.md is the primary memory file Claude Code auto-loads into context at the start of every session. It is hierarchical: a user-global ~/.claude/CLAUDE.md applies everywhere; a per-project ./CLAUDE.md overrides or extends it. Skills typically point to additional reference files rather than embedding everything in CLAUDE.md, keeping each file short and allowing progressive loading — only the reference a skill actually needs is read when it needs it.

Layer 2 — the context folder and the learnings loop

Knowing your brand is static — it's a snapshot. The second layer is what makes the system actually feel alive, because the system doesn't just know your brand, it learns how you work too. Most AI setups are frozen in time: you build them and they stay exactly as they are. Building an Agentic OS is different, because you want it to have that second layer alongside the brand context. That's a context folder inspired by how OpenCog actually manages its memory and manages its functions around the user. This is what turns your skill collection into something that genuinely starts to understand you.

Inside the context folder: first, a soul.md file — the agent's identity, how it behaves, how it communicates, what it's prioritising. Then a user.md about you: your preferences, your working style, how you like things structured. When you tell the system you prefer concise bullet points over long paragraphs, it's going to always remember that and save that in your preferences — and every skill reads this too. Then memory.md for long-term business knowledge, plus short-term daily memory files inside a memory folder: a log of what it worked on yesterday, what it produced last week, what's in progress right now. So when you start a new session you're not starting cold — you have continuity, you can pick up where things left off.

The part Simon thinks is most powerful is learnings.md — the system's long-term feedback memory. After a major deliverable, skills will solicit your feedback: was the research useful? Was the newsletter too long? Whatever your feedback is gets logged under that skill's section. At the end of the day those learnings get taken and used to update the skill.md files themselves. The skill is actually learning and maintaining its own instructions. Next time the skill runs, it reads its own learnings first. Finally, no repeating yourself; no having the same conversation twice.

Concept · Memory tool vs CLAUDE.md vs learnings files

Claude Code has a formal Memory tool — a persistent, file-backed store the model writes to autonomously across sessions. This is distinct from the human-curated CLAUDE.md. Simon's learnings.md pattern is a community convention that sits on top: a structured markdown file that skills explicitly read at the start of each run and write to at the end, guided by skill-specific feedback prompts. The three mechanisms complement each other: CLAUDE.md carries stable instructions; the Memory tool carries spontaneous observations; learnings.md carries your deliberate quality feedback.

Layer 3 — self-maintenance: heartbeat and wrap-up

As the system grows, things start to change: you add new skills, remove skills, connect new tools. In most setups that means you're manually updating docs, making sure nothing's broken. The Agentic OS should handle all of that itself. The way Simon has built it is: every time a new session starts, the system runs what he calls a heartbeat — familiar to anyone who has seen OpenClaw's setup. It scans the skills folder and compares what's actually on disk against what's documented in your CLAUDE.md and README file. If you've added a new skill folder since the last session, it adapts: registers the new skill, updates the context matrix, adds a section in your learnings.md. You don't have to touch anything. Every time you start a session it's going to learn exactly what you've added or taken away from the system — and this extends to MCP servers too, not just skills.

The really clever part is what happens when you build a new skill. A common problem with skills is installing too many and having overlapping ones. So before the system creates anything, it should read every installed skill's frontmatter, mapping overlaps and checking for dependencies, so the new skill fits cleanly into the existing ecosystem instead of duplicating what's already there. And at the end of the session the same sync needs to run again — that's done through a wrap-up skill: a checklist that reviews deliverables created, collects feedback, fixes skills, updates learnings, and finally commits all the work. At the end of the session you just say "close session" or "wrap things up" and it triggers that full sync. The system starts to reach a point where you never have to update a registry file again — it maintains itself so you can focus on using it, not babysitting static docs.

The "wrap-up skill" is a complementary end-of-session trigger. Using hooks means the self-maintenance layer runs reliably without relying on you remembering to invoke it.

Concept · Skill chaining

Skill chaining is a usage pattern, not a named Claude Code feature. It means writing a skill (or a slash command) that explicitly invokes other skills in sequence, passing the output of one as the input of the next — typically by writing intermediate results to shared files that downstream skills read. The mechanism is just file I/O: skill A writes research-brief.md; skill B reads research-brief.md as a reference. Claude Code's task (subagent) tool can run skills in parallel with isolated context windows; the results then feed back to the orchestrating skill.

Glossary — terms from this episode

Skill
A SKILL.md file with YAML frontmatter (name, description) plus optional reference files and scripts. Claude Code loads only the name+description into context until the skill is invoked; then the full body loads. The primary reusable-workflow primitive.
Progressive disclosure
The property that only a skill's name and description occupy context until invocation. Means you can register dozens of skills without bloating the context window.
Shared context folder
A community design pattern: a single directory of markdown reference files (voice, ICP, schema, methodology) that every skill reads from by path. Keeps all skills consistent without duplicating content. Not a named Claude Code feature — it's a convention layered on CLAUDE.md and reference files.
Learnings loop / learnings.md
A community pattern where each skill maintains a learnings.md file it reads at start and writes to at end, logging user feedback. The skill's behaviour improves across sessions without the user repeating feedback manually.
Heartbeat
A session-start routine (often wired to a Claude Code SessionStart hook) that scans the skills folder, compares it against the CLAUDE.md registry, and self-updates any differences. Keeps skill documentation in sync automatically.
Wrap-up skill
A session-end skill that reviews deliverables, collects feedback, fixes skill files, updates learnings, and commits work. The end-of-session mirror to the heartbeat.
Skill chaining
A usage pattern where one skill writes an output file that a downstream skill reads as its input — effectively a pipeline. Not a named Claude Code feature; implemented via file I/O between skills.
Agentic OS / Agentico OS
Simon's community branding for an opinionated assembly of Claude Code primitives (CLAUDE.md + skills + hooks + memory) into a structured personal system. Useful framing; not an Anthropic product.
ICP (Ideal Customer Profile)
A reference document describing the target audience. In Simon's setup, a foundation skill interviews you and writes this as a structured markdown file that all content skills read. Equivalent in research context: your "intended reader" profile for Substack posts or board papers.
Hooks
Shell commands Claude Code runs automatically on session events (SessionStart, Stop, PreToolUse, PostToolUse), configured in settings.json. Deterministic — the model cannot skip them. The right mechanism for the heartbeat and wrap-up patterns.
Memory tool
Claude Code's built-in persistent, file-backed store the model writes to autonomously across sessions. Distinct from the human-curated CLAUDE.md and from Simon's community learnings.md pattern — all three complement each other.
OpenClaw
A community framework/setup referenced by Simon as inspiration for the heartbeat pattern. Not an Anthropic product; a third-party opinionated Claude Code configuration. Treat as community tooling whose specifics change.
