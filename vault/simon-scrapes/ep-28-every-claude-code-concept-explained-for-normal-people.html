Concepts 1–3 — what Claude Code is, the terminal, and prompts

Let's start with concept one: what is Claude Code? Most people already know chatbots like ChatGPT, Claude, Gemini. You type a message, you get a reply. Those tools can only talk to you — they can't actually do anything on your computer, and they've only recently been given access to do anything more than that, like web search. Claude Code is entirely different. It can create files, build websites, set up databases, install packages on your machine, and do your repetitive tasks, all from a simple conversation in plain English. You describe what you want, and it builds it. Chatbots give you advice; Claude Code takes actions.

Concept · Claude Code

Claude Code is Anthropic's agentic CLI (command-line interface) — also available as a desktop app, web interface, and IDE plugin. Unlike a chatbot, it runs a continuous loop: read files, call tools, execute bash commands, write or edit code, and report back. It can take real actions on your machine and, via MCP servers, on external services too. The model underneath (Haiku, Sonnet, or Opus) can be switched mid-session.

Concept two is the terminal. Claude Code doesn't run in a browser like ChatGPT — it runs in the terminal, that black screen with white text that looks so cryptic whenever you've seen developers use it. The terminal is actually really simple: it's just another way to interact with your computer. Instead of clicking buttons, you type commands. But here's the thing — when you're using Claude Code, you don't need to know many terminal commands at all. You type in plain English and Claude handles everything else underneath. You really only need to know a few: claude to open it, Control-C twice to close it, and /clear to clear its memory. And then, of course, how to talk to it well — which brings us onto prompts.

Concept three: a prompt is simply what you type to tell Claude Code what to do, the same as when you're typing something into ChatGPT. You write prompts in plain English — build me a landing page, create me a dashboard, fix this error. There's no special syntax, no code, just describe what you want. The more specific you are, the better the result. So instead of "build me a website", say something like: "Build me a one-page landing page for a consulting business with a green colour scheme. I want it to have a contact form and three cards explaining our service." Better prompts, better results — and that is really the whole game when it comes to Claude Code. But there's a concept later that will make prompting much easier.

Applied economist lens

The "specificity = quality" rule for prompts maps directly onto analytical instruction-writing. When you kick off a Companies House extraction, the difference between "pull the accounts" and "extract turnover, operating profit, and headcount from the latest filed accounts PDF for company number 12345678, returning a single JSON object with those three fields" is the difference between a usable row in your 96-firm database and a vague narrative summary. Invest ten seconds in specificity at the prompt — it buys minutes of correction later.

Concepts 4–5 — permissions and tool use

Concept four: permissions. This is an incredibly important concept because Claude Code can make actual changes to your computer — it can create files, run commands, edit code, even delete files. By default, it asks for your permission before it does anything significant. But if you're clicking approve every 10 seconds on a project that's going to take a day, it really kills your flow, and there is a better way.

I recommend starting with a default where Claude asks approval on most actions, so you understand exactly what it's asking for. But as you get comfortable, you can pre-approve specific actions so Claude works faster without stopping you constantly. You can go nuclear and give it permissions to do absolutely anything using the --dangerously-skip-permissions flag — but even Boris Cherney, the creator of Claude Code, says he never does this. Instead, use a permissions file in settings.json. You can edit this file directly or type /permissions and add commands to an allow list. These get saved to your settings.json file, which means you can check it into your repo and share it with your team, so everyone has exactly the same permission setup.

Here's my thinking for most projects. Things safe to pre-approve: reading files and folders, running the dev server, running tests so Claude can check its own work, and Git operations — things like committing changes and keeping version control in place. Things to keep gated and requiring your approval: installing new packages, deleting files or folders, and anything touching the internet — API calls, sending data. Keep your eyes on those. And the nice thing is you don't need to write that JSON by hand — just tell Claude: "Add the common safe permissions to my settings — reading files, running tests, Git operations, starting the dev server. Show me what you're adding before you save it." Claude will go and do the hard work for you.

Concept · Permissions / settings.json

Permissions in Claude Code are controlled by settings.json (per-project) and a user-level settings file. Each entry is either an allow rule (runs without confirmation) or a deny rule (blocked outright). The format uses tool names and optional argument patterns — for example, allowing Bash(ls *) lets Claude list files without prompting. You can invoke /permissions in the terminal to view and edit these interactively. The --dangerously-skip-permissions flag bypasses all checks and is strongly discouraged for anything beyond throwaway experiments.

Applied economist lens

A shared settings.json checked into your consultancy-research repo is the right way to standardise what Claude can do unattended. Allow reads, test-runs, and Git commits; gate API calls and file deletion. That way, if you hand the project to a colleague or run it on a different machine, the safety envelope travels with the repo — the same guarantee you'd want on any shared analytical pipeline. Add a deny rule for your .env and any credentials folder so API keys can never be accidentally read into context.

Concept five is all about tool use. What makes Claude Code more powerful compared to regular Claude or ChatGPT is that it has access to tools — built-in capabilities that let it interact with your computer. The main ones are really simple: Read (Claude can look at your files and data), Write (Claude can create new files), and Bash (Claude can run commands like installing packages, running tests, or moving files around). You don't need to tell Claude which tool to use — you just describe what you want and Claude picks the right tools automatically. You just need to understand the concept of tools; you describe the goal, Claude picks the tools.

Concept · Tool use

Claude Code's built-in tools are: Read (read file contents), Write (create or overwrite files), Edit (targeted in-file edits), Bash (run shell commands), and several specialised tools like web search and the ask-questions tool used during planning. Each tool invocation is visible in the terminal and subject to your permissions settings. You never need to invoke tools yourself — Claude selects them based on what your prompt requires.

Concepts 6–8 — context window, conversation history, and token cost

Concept six: the context window. The context window is Claude's short-term memory — it's everything Claude can see and think about at any given moment during a conversation. Every message you send, every file Claude has read, every response it has generated: all of that lives inside the context window. And it has a limit. When a conversation gets too long or too much information piles up, Claude is going to start losing track of the earlier stuff. This is a phenomenon called context rot — that's when outputs get worse and Claude seems to forget what you've told it, similar to hallucinations on ChatGPT.

The fix is simple: keep your conversations focused, start fresh sessions when you need to switch tasks, or use the /clear or /compact commands to help manage it — we'll talk more about those shortly. You can track context limits by the green bar at the bottom of the terminal. Think of the context window as Claude's short-term memory: try to keep it clean as you go through a session.

Concept · Context window & context rot

The context window is the total amount of text (measured in tokens) the model can "see" at once — your prompt, the files it has read, tool outputs, and the conversation history. Everything the model knows in a session lives here; when it fills, earlier content gets pushed out or summarised. "Context rot" is the empirical finding that as the window gets fuller, the model becomes less reliable at retrieving the most relevant information — not because it forgets, but because the signal-to-noise ratio falls. This is why a 30-minute build session can drift and why keeping context lean matters.

Applied economist lens

Context rot is the hidden reason your longer analytical sessions degrade. When you load 25 years of salary-survey tabs or paste a methodology section before asking for QA analysis, the AQuA rules you stated at the top are precisely the instructions most likely to get buried by the time Claude is processing year 2014. The structural fix is CLAUDE.md (concept 9 below) — keep your binding rules in a short file Claude re-reads at the start of every session, and feed data in focused chunks rather than one giant dump.

Concept seven: conversation history. Claude Code automatically saves your previous conversations — every single session is stored, and you can get back to any of them. If you close your instance, just type claude --resume and it will drop you straight back into your most recent conversation: same project, same context, picking up right where you left off. This shows you a list of all your previous sessions; you can scroll through, see what each one was about, and pick the one you want to jump back into. This is essential in practice, because you're rarely finishing a big project in one conversation instance — you build something, close your laptop, come back the next day, and instead of starting over, you just resume.

Concept eight: token usage. When you use Claude Code, it costs money, and the way that cost is calculated is through token usage. A token is roughly three-quarters of a word, so 1,000 tokens is roughly 750 words of input or output. Every prompt you send, every response you receive, every file Claude reads — it all uses tokens. The price depends on which model you're using: Sonnet is cheaper, Opus is more powerful but costs more. Claude Code tracks your usage so you can see exactly what you're spending — use /cost and /stats to bring up the amount of tokens spent in a given session.

Concept · Tokens and model pricing

Tokens are the billing unit for Claude API usage. One token ≈ 0.75 words in English. Input tokens (your prompt, files read, tool outputs fed back) and output tokens (Claude's response) are both charged. Claude Code runs on Anthropic's API, so costs accrue per token unless you're on a Claude Max subscription ($100 or $200/month flat-rate). The /cost command shows cumulative spend for the current session; /stats shows token counts broken down by input/output.

Applied economist lens

For research-intensity work — pulling 96 Companies House filings, each requiring several tool calls and file reads — the per-token cost can stack up surprisingly quickly on pay-as-you-go. If you're doing regular batch extraction runs, the flat-rate Claude Max subscription almost certainly beats API billing once you're past a few thousand tokens a day. Treat it the same way you'd model a data-subscription cost: estimate your monthly token volume at current rates, compare to the subscription, and switch when the crossover is clear.

Concepts 9–10 — CLAUDE.md and persistent memory

Okay, so now you've got the foundations — you know what Claude Code is, how it works, and how to talk to it. But until now it's still a blank slate. This is where we make Claude Code personal to you, which brings us to CLAUDE.md — possibly the most important concept in this whole video.

Every time Claude Code starts working on your project, it needs to know how you want to get things done. Without guidance, it's going to completely guess — and guessing leads to poor results. CLAUDE.md is just a markdown file in your file structure where you write your preferences, your rules, and your project structure. Claude reads it first, every single line, every time you open a new instance. It'll see things like "We use TypeScript. Always write using our brand voice. Don't ever use em dashes." You write it once, and Claude Code follows it in every single conversation. Think of CLAUDE.md as your instruction manual for Claude — you iterate on it over time, and benefit from that forever.

Concept · CLAUDE.md

CLAUDE.md is a plain markdown file that Claude Code automatically loads into context at the start of every session. It is hierarchical: a user-global version lives at ~/.claude/CLAUDE.md (applies to all projects) and a per-project version lives at ./CLAUDE.md (or .claude/CLAUDE.md) in your repo. The per-project file takes precedence for project-specific rules. It is the primary mechanism for persistent context — not AI-written, but human-curated. Put your coding standards, QA rules, schema definitions, and project conventions here. Simon's /init command helps you create a good starting version.

Applied economist lens

CLAUDE.md is your AQuA rulebook and schema registry made permanent. In your consultancy-research project, the CLAUDE.md should carry: the 96-firm database field definitions (company_name, registration_number, turnover, operating_profit, headcount, filing_year), your data-quality rules (source must be filed accounts not estimates; flag if turnover differs by >30% year-on-year as a potential outlier), and the Substack house-style guide (no em dashes, active voice, economist not journalist). That way every new session — whether you're extracting a new firm or drafting a profile — starts from the same authoritative baseline without you restating it.

In practice

Bootstrap a project CLAUDE.md with: /init — Claude Code will inspect your repo structure and draft a starter file. Then edit it to add your domain rules.

Concept ten: memory. Everything we've covered so far lives within a single conversation — but what if you want Claude to remember things across multiple conversations? This isn't about bringing up previous conversations like the history we just showed; it's about recognising patterns you use repeatedly and storing those so Claude can be even more personalised. CLAUDE.md was our manual memory file, but Claude Code also has an auto-memory file that lets it store your preferences and facts across different sessions. If you always work in JavaScript or prefer a certain coding style or have specific project conventions, those get automatically saved to memory and Claude will know them every time you start something new. You can ask Claude to add, remove, and review what it remembers about you and the way you work. It's all about learning your preferences over time, not just within a single chat. You won't see the memory files in your project directory, but they are being built as you work.

Concept · Memory tool

The Memory tool is a Claude Code feature that persists facts across sessions by writing to a file in Claude's data directory (separate from CLAUDE.md). Claude can be instructed to save preferences, update what it remembers, or clear stale entries. The key distinction: CLAUDE.md is human-curated — you write and maintain it deliberately. The memory tool is AI-maintained — Claude writes to it automatically as it learns patterns. Both complement each other; use CLAUDE.md for stable, structural rules and memory for evolving personal preferences.

Concepts 11–13 — compact context, models, and file deny-listing

Concept eleven: compact context. We talked earlier about context length — when conversations get really long, the context window fills up. If you don't manage that, Claude starts losing track of the important info. Here's how we actually manage it. Claude can do it automatically: when it detects the context is getting close to full, it automatically takes the key information, summarises it, and clears out the noise, passing the summary to the next instance of the conversation. This happens in the background without you doing anything at around 85% of the context window. But if you notice Claude getting confused mid-session, you can manually trigger this using the /compact command and append arguments like "keep info about API calls" to give more specific instructions on what to keep. Memory compacting keeps Claude sharper during longer sessions — you can leave it to itself or help it by doing it at key stages.

Concept · Compaction (/compact)

Compaction replaces the bulky conversation history with a model-written summary when the window nears full, so the session can continue. It triggers automatically at roughly 85% capacity, but you can invoke /compact yourself at a clean moment — ideally before Claude gets confused — and append retention guidance. The risk: a poor auto-summary silently drops something load-bearing (a schema definition, a QA rule, an outstanding to-do list). Compact deliberately at natural checkpoints rather than waiting for the automatic trigger.

In practice

Steer compaction to preserve what matters: /compact keep the 96-firm schema, the AQuA outlier rules, and the list of companies still needing accounts pulled. That pins your method and to-do list across the summary boundary.

Applied economist lens

Treat /compact like saving a model checkpoint before a risky edit. In a long salary-survey QA session — say, checking all 25 years for the annual-hours outlier pattern — compact at the end of each decade block: "keep the outlier flag rules, the years completed, and any firms already flagged". That way if the auto-summary fires unexpectedly, your methodology and progress notes survive rather than the chatty tool output from 2001–2010.

Concept twelve: models. When people say "Claude Code", they're actually referring to a family of AI models underneath. Each one has different strengths, speeds, and costs. Haiku is the fastest and cheapest — great for simple, quick tasks. Sonnet is the middleman: an all-rounder, good at most things, reasonably priced and reliable. Opus is the most intelligent — it handles complex problems that require a lot of thinking, but costs the most. You can switch between them mid-conversation using /model in the terminal. The key: match the model to the task. Use Sonnet for most things, Opus when it genuinely matters, and Haiku for very simple tasks.

Concept · Model selection

Claude Code currently runs on Claude Haiku, Sonnet, and Opus (with numbered sub-versions, e.g. Claude Sonnet 4.6). Haiku is best for quick, low-stakes tasks — formatting, simple lookups. Sonnet is the everyday workhorse — it handles code generation, multi-step extraction, and most analytical tasks well at reasonable cost. Opus is the right choice when you need genuine multi-step reasoning: complex QA, ambiguous interpretation, or designing an evaluation framework. Switch with /model claude-sonnet-4-6 (or whichever version is current).

Concept thirteen: denying access to files. Not every file in your project should be passed to Claude Code. Some are massive and would waste your context window. Some contain sensitive data — API keys and passwords. Some are just totally irrelevant. You need a way to tell Claude: do not touch these under any circumstance. You do this in the settings.json file — you can add a deny list that tells Claude which files and folders are completely off-limits. Claude won't even discover these files, won't include them in search results, and can't read them even if you ask directly. Deny rules in settings.json are the proper way to do this inside Claude Code.

Applied economist lens

Any project that touches real data needs a deny list. In your consultancy-research repo: deny .env, secrets/, and any folder with raw filed accounts PDFs that haven't been cleaned yet (to avoid accidentally feeding PII or confidential data into the model's context). In your salary-survey project, deny the raw submission CSVs before your PII-scrubbing step has run — this is a cheap guard against accidentally processing un-anonymised respondent data mid-session.

Concept 14 — launch flags

Concept fourteen: flags. Flags are options you pass when you start and launch Claude Code — they let you customise how Claude behaves for that specific session before it even begins. You can use --model to choose the model, --allowed-tools to control which tools it has access to (like reading or writing), --verbose to get a more detailed output every time Claude responds so you know exactly what it's doing, or --dangerously-skip-permissions (the nuclear option we mentioned earlier). Flags are just launch settings — entirely optional, but a useful shortcut for non-standard sessions.

Concept · CLI flags

Claude Code flags are passed on the command line when you start a session: claude --model claude-opus-4-6 --verbose. Key flags include: -p "<prompt>" for headless/CLI mode (runs once with no conversation loop); --allowed-tools to restrict the tool set; -w <name> / --worktree <name> to start in an isolated work tree; and --verbose for detailed output. They are session-scoped — they do not persist to settings.json.

Concepts 15–18 — extended thinking, slash commands, skills, and hooks

Now the foundations and configuration are solid. This next section is where things get genuinely game-changing — features that take Claude Code from useful to transformative.

Concept fifteen: extended thinking. Some tasks are super simple — rename this file, for example. But some tasks need Claude to genuinely reason through the problem before it acts. Extended thinking gives Claude a dedicated reasoning budget in tokens to think through the problem step-by-step before taking any action. You used to have to write words like "think", "think hard", or "ultra think" in your responses to activate this, but now by default Claude Code has the ability to use that thinking budget with a cap of max budget for thinking tokens. So all you need to know is: extended thinking is Claude reasoning before it acts, it's on by default, and it has a max token budget allocated with it.

Concept · Extended thinking

Extended thinking is a Claude capability where the model uses a separate reasoning "scratchpad" (a block of thinking tokens) before producing its visible response. This improves performance on tasks requiring multi-step planning, logical decomposition, or ambiguity resolution. In Claude Code, a thinking budget is allocated automatically for complex requests — you do not need to enable it manually. The thinking block is not always visible in the terminal output, but it is happening. Expect it to be most valuable during plan-mode, agent-team orchestration, and complex debugging sessions.

Applied economist lens

Extended thinking is most valuable at the design stage of a complex analytical task. When you're asking Claude to plan a 3-arm RCT power calculation or to design a financial-ratio extraction pipeline for 96 firms with varying account formats, it pays to let it think through edge cases before writing a line of code or making a tool call. Use Opus with extended thinking for those design phases; switch to Sonnet for the execution once the plan is locked.

Concept sixteen: slash commands. Throughout this video we've been using slash commands — typing a forward slash followed by a command name to trigger a specific action. /init sets up a new project and helps you create your CLAUDE.md file. /compact shrinks your conversation to save context space. /clear wipes the conversation for a fresh start. If you forget any, just type /help and it will show you a list of all available slash commands. And importantly, you can turn any task you do on a day-to-day basis into a slash command by putting it in the slash commands folder. Think of slash commands as shortcuts — and /help shows you all of them.

Concept · Slash commands

Slash commands are built-in shortcuts invoked by typing /<name> in the Claude Code terminal. Built-in examples: /init, /compact, /clear, /model, /cost, /stats, /rewind, /permissions, /help. You can also create custom slash commands by placing markdown files in .claude/commands/ — each file becomes a /filename command that submits that markdown as a prompt. This is distinct from Skills (which are richer, with YAML frontmatter and supporting files) but useful for quick repeatable prompts.

In practice

Create a custom slash command for your most repeated research task: save a prompt template as .claude/commands/pull-accounts.md and invoke it with /pull-accounts. Useful for a standard "given company number X, extract turnover, operating profit, and headcount from the latest filed accounts and return JSON" instruction.

Concept seventeen: skills. Claude Code is a generalist — it can do a lot of things to an okay degree. But for specific task types, like creating a social media strategy or building a front end, it can do even better if you give it specialised instructions. Skills are pre-written instructions that teach Claude how to do specific tasks really well — whether that's building presentations, writing good copy, or designing front ends. When you ask Claude to do something that matches a skill's description, it reads that skill file, which can also have supporting reference files to understand best practices and optimal approaches, and then builds using all of that knowledge. Skills use more context because those files are loaded in, but they improve the quality of your outputs. You can find a skill and install it, or create your own. Skills are expert playbooks, and Claude reads them before building.

Concept · Skills

A Skill is a SKILL.md file with YAML frontmatter (name, description) plus optional bundled reference files. Claude Code detects which skill to load based on matching the description to your request, then pre-loads the skill's instructions and supporting files into context before responding. Skills package reusable domain knowledge — a copywriting style guide, a financial-ratio extraction methodology, a chart-formatting standard. They are shareable, installable, and composable. Simon's community maintains a curated library; you can also write your own from scratch.

Applied economist lens

Skills are the right abstraction for your repeatable analytical workflows. A companies-house-extractor skill could carry: the exact fields to extract (turnover, operating profit, headcount, filing date, SIC code), the logic for handling different account formats (abbreviated, full, micro-entity), and a QA checklist (cross-check turnover against prior year, flag if operating margin is outside −20% to +60% band). Once written, every extraction session in your 96-firm project starts from that methodology without any re-prompting. Similarly, a spe-salary-survey-qa skill could encode your AQuA outlier-detection rules and year-on-year consistency checks.

Concept eighteen: hooks. Sometimes you want specific rules to run automatically at certain moments — after an event has taken place — but you don't need AI to do that every single time and spend AI tokens on it. Hooks are custom scripts that trigger at those specific points. Whenever Claude saves a file, we're going to auto-format that file using our rules. Or we're going to log every single command that Claude runs. Setting them up requires a bit more technical know-how — though you can just ask Claude Code to do that for you. Think of hooks as custom rules that trigger automatically every single time, like guardrails that run in the background without using any AI tokens.

Concept · Hooks

Hooks are shell scripts configured in settings.json that fire at defined lifecycle events in the Claude Code loop — for example: PostToolUse (after any tool call), PreToolUse (before), or Stop (when the session ends). They run as regular processes, not via the model, so they consume no AI tokens. Common uses: auto-lint after every file write, log every bash command for audit purposes, or send a Slack notification when a long-running task finishes. Because hooks are deterministic scripts, they are also a safer guardrail than trying to instruct Claude to "always do X" in a prompt.

Applied economist lens

A PostToolUse hook is a natural fit for your data-integrity pipeline. After every Write tool call in your consultancy-database project, fire a hook that runs a short Python validator — checking the JSON Claude just wrote conforms to the schema (required fields present, types correct, no negative turnover). That validation runs outside the model, costs nothing, and catches schema drift before it silently corrupts 50 rows. Similarly, a hook that appends every bash command to an audit log gives you the analytical equivalent of a version-controlled methodology trail.

Concepts 19–20 — MCP servers and subagents

Concept nineteen: MCP servers. By default, Claude Code works with files on your local computer — but your business doesn't just live on your machine. It lives across Airtable, Notion, Asana, and dozens of other tools. MCP stands for Model Context Protocol, and in simple terms, MCP servers connect Claude Code to those external tools. Once connected, Claude can pull data from those tools, push updates to them, and interact with your full tech stack from that single conversation window. This is what makes it so powerful — Claude can take actions not only on your desktop, but in the apps you use day-to-day.

Concept · MCP servers

MCP (Model Context Protocol) is an open standard for connecting AI agents to external data sources and services. An MCP server is a lightweight process that exposes tools (callable functions) and resources (readable data) to Claude Code over a local socket or network connection. Once configured in settings.json, Claude can invoke those tools just like it uses its built-in Read/Write/Bash tools. Examples: a Companies House MCP server that accepts a company number and returns filing metadata; a Gmail MCP server that reads and labels threads; a Notion MCP server that queries and creates pages. MCP servers are independent of Claude Code — they can be written in any language and shared as open-source packages.

Applied economist lens

You already run MCP servers for Companies House, Gmail, and Notion — which means you can build Claude Code workflows that span all three in a single conversation. A canonical example: ask Claude to find all Companies House filings for the 15 firms in your PE-owned cohort that were filed in the last 12 months, pull the accounts summary for each, draft a one-paragraph Substack update for each firm, and log the results to a Notion database — all in one unattended loop. That's the MCP value proposition: your whole research stack, orchestrated from a single prompt.

Concept twenty: subagents. So far, everything we've covered runs in a single conversation — one terminal, one context window — and that works for 80% of your daily tasks. But the longer a session runs, the more context piles up and the more Claude starts forgetting things. We talked about compact as one fix. But there's another way to stop that: delegating to subagents. Subagents are specialists that run in their own separate context window with their own system prompt and their own tool access, but still follow the main CLAUDE.md file. Your main instance hands off a task, the subagent does the work independently, and then sends back the result to the main agent.

There are two reasons to use subagents. First, output quality: a focused specialist with a clean context window beats a generalist with too much information. Second, speed: you can run multiple subagents in parallel on unrelated tasks. Similar to skills and commands, you can create your own agents in the agents folder or leverage other people's agents too. The general rule: use a skill when you want Claude to be better at something in your current conversation; use a subagent when the task is entirely self-contained — "go away, analyse this report, come back with your findings" — something it can do unsupervised. Note that Claude Code will also spin out subagents on its own whenever it deems that architecture necessary.

Concept · Subagents

A subagent is a Claude Code instance spawned by an orchestrator (the main agent) with its own isolated context window, its own system prompt, and its own tool permissions. It receives only the information it needs for its specific task, executes independently, and returns a summary result to the orchestrator. Subagents cannot communicate with each other directly — everything flows through the main agent. This isolation means each subagent starts fresh, avoiding context rot, but it also means the orchestrator must carefully specify what each subagent needs to know. Subagents respect the project's CLAUDE.md even though they have isolated context.

Concept 21 — agent teams

Concept twenty-one: agent teams. Subagents have a limitation: they can only report back to the main agent. So if you have a researcher, a writer, and a reviewer as subagents, the reviewer can't see the research directly, and the researcher can't see the reviewer's feedback. Everything flows through the main agent, and that becomes a bottleneck when you have five or more subagents reporting in.

Agent teams — a newer feature of Claude Code — are designed to solve this. Instead of that hub-and-spoke model where everything goes through one central point, teammates can communicate directly with each other and share a task list. The main agent spins up teammates only when you specifically ask it to, and each runs in their own Claude Code instance — they get a sample of the information, but not the full context of the main conversation history. If you're doing something complex like building a SaaS app, you might have an API developer, a front-end developer, and a tester all working in parallel as separate agents who need to share context on what each other is working on and feed it back to a shared task list.

Here's the simple rule for when to use what: a single agent for one feature or one file that needs changing; subagents when you need specialists and your context is getting bloated; agent teams when you have a genuinely complex build that requires collaboration between the team.

Concept · Checkpoints and /rewind

Claude Code creates an automatic snapshot of the codebase state before every file write or edit. These checkpoints are session-level (not a full Git commit) and are stored internally. Concept twenty-seven — and this is a big one — work trees. When you're working on a bigger project, you often need to work on multiple things at once: a new feature, a bug fix, an experiment. Work trees give you multiple Claude Code instances working on different tasks simultaneously, each in a separate working directory with its own files and its own branch, but all sharing the same repository history. When finished, merge the branch back into your main project. When you exit a work tree session, Claude handles cleanup based on whether changes were made. Concept · Work trees

Work trees are a Claude Code feature built on top of Git's native worktree mechanism. Each work tree is a separate checkout of the same repository into a new directory, on a new branch — so you get full file isolation without duplicating your entire repo. Claude Code manages the branch and directory creation for you via the -w flag. Work trees are particularly powerful in combination with subagents: an orchestrator can spawn each subagent into its own work tree, so parallel tasks never touch the same files. Merging is a standard Git operation once work is complete.

Glossary — terms from this episode

Claude Code
Anthropic's agentic CLI tool. Takes real actions on your computer and external services via a continuous loop of reading, reasoning, tool use, and writing — unlike a chatbot, which only talks.
Terminal
A text-based interface to your computer where you type commands instead of clicking. Claude Code runs here; you only need a handful of commands to operate it.
Prompt
The plain-English instruction you type to Claude Code. The more specific, the better — include format, fields, constraints, and examples where possible.
Permissions / settings.json
The allow/deny rules governing what Claude Code can do without asking. Stored in settings.json; can be checked into your repo for team sharing.
Tool use
Claude Code's built-in capabilities: Read, Write, Edit, Bash, and specialised tools. Claude selects them automatically; you just describe the goal.
Context window
The total tokens the model can "see" at once — your prompt, files read, tool outputs, and conversation history. Finite; when full, older content is pushed out.
Context rot
The degradation in model reliability as the context window fills — not forgetting, but losing the ability to find the most relevant information in a noisier haystack.
CLAUDE.md
A human-curated markdown file auto-loaded into every session. The primary mechanism for persistent project context: schema definitions, QA rules, house style, coding standards. Hierarchical: user-global and per-project versions both apply.
Memory tool
An AI-maintained persistent-memory file that Claude Code updates automatically across sessions to remember your preferences and patterns. Distinct from CLAUDE.md, which you write.
Compaction (/compact)
Replaces bulky conversation history with a model-written summary when the context window nears full. Automatic at ~85% capacity; invoke manually with /compact [instructions] to control what survives.
Slash commands
Built-in shortcuts invoked with /<name> in the terminal (/init, /compact, /clear, /rewind, etc.). Custom slash commands can be added via .claude/commands/.
Skills
Packaged domain-knowledge bundles: a SKILL.md with YAML frontmatter plus optional supporting files. Claude loads the relevant skill when your request matches its description, improving specialist output quality.
Hooks
Shell scripts that fire at lifecycle events (PostToolUse, PreToolUse, Stop) without consuming AI tokens. Used for auto-linting, audit logging, notifications, and deterministic guardrails.
MCP servers
External processes exposing tools and data to Claude Code via the Model Context Protocol. Extend Claude's reach beyond local files to APIs, SaaS tools, and databases.
Subagents
Isolated Claude Code instances spawned by an orchestrator, each with their own clean context window. Best for self-contained tasks; cannot communicate with each other directly.
Agent teams
A Claude Code architecture where multiple agents communicate peer-to-peer and share a task list, removing the single-orchestrator bottleneck. Suited for large, multi-component collaborative builds.
Extended thinking
A reasoning budget of tokens Claude uses internally before producing a response — improving performance on multi-step planning and ambiguous tasks. On by default in Claude Code.
CLI mode / headless mode
Non-interactive mode invoked with the -p flag. Claude runs the full agentic loop, outputs the result, and exits — no conversation. Makes Claude Code scriptable and pipeable.
Ralph loop
A community pattern that wraps Claude Code in a while loop, re-submitting a fixed prompt.md with the current file state each iteration until the task is complete. Enables unattended, iterative project builds. Not an official Anthropic feature.
Work trees
Isolated repository checkouts on separate branches, each with their own Claude Code session. Enables parallel, non-interfering work streams and is particularly powerful when combined with subagents.
Checkpoints / /rewind
Automatic session-level snapshots taken before every file edit. /rewind lets you restore code and/or conversation to any prior state within the session.
