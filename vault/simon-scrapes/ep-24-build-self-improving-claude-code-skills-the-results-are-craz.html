Concept · Auto-research loop

Auto-research (Karpathy's term) is a self-optimisation pattern where an AI agent autonomously edits a file, runs an evaluation, reads the score, and branches: if the score improved it commits the change; if it dropped it resets to the previous state. The loop continues without human input until a target score is reached or the operator interrupts it. It is a community pattern, not an official Claude Code feature, but it runs cleanly inside Claude Code because of Claude's built-in git tooling.

We've got: tune train.py with an experimental idea by directly hacking the code (i.e. make a change). Run the experiment. Read out the results. If the value has improved, advance the branch and keep the commit. If the value is worse, reset to where we started. And I love this line too: "Never stop. Once the experiment loop has begun, do not pause to ask the human if you should continue. Do not ask if I should keep going or is this a good stopping point. The human might be asleep or gone from the computer and expects you to continue working indefinitely until you are manually stopped. You are autonomous." In other words: keep working until you've either improved the results to the point where there are no additional gains to be made, or we interrupt you.

This means the file on disk is always the best version seen so far.

Concept · Eval.json and binary assertions

An eval.json file lives inside a skill's eval/ subfolder and defines a set of test cases. Each test case has a prompt, an expected-output description, and a list of binary (true/false) assertions that can be mechanically checked. Examples: "the final line is not a question", "word count is under 300", "output does not contain em-dashes". The agent runs the skill, checks each assertion, tallies a pass rate, then decides whether the current SKILL.md version should be kept or reverted. Binary assertions are critical: anything subjective ("is this compelling?") cannot be automated.

Why binary assertions are everything

Let's talk about what binary assertions are and why they're important. The word binary is everything here — this is where most people are getting it wrong when they're executing tests on their skills. On the right-hand side we have something binary: it does not contain em-dashes; it is under 300 words; the final line is a question. All true-or-false statements. Versus something very subjective like "does it have a compelling subject line?" — obviously not binary, because two people can disagree on what compelling actually means, which means we can't automate it. Of course you can get assistance from Claude Code to say "based on certain frameworks, this is considered compelling", but it's not the binary true/false approach. Those still need a bit of human judgement.

Concept · "One change at a time" discipline

Making only one change per loop iteration is a deliberate scientific constraint. If you allow the agent to make multiple simultaneous edits and the score improves, you cannot tell which change caused the improvement — or which caused a regression. One-change-per-commit preserves a clean git history where each commit represents a single testable hypothesis about the SKILL.md, making it possible to bisect if the skill later degrades.

Glossary — terms from this episode

Auto-research loop
Karpathy's pattern: an AI agent autonomously edits a file, runs an evaluation, and commits or reverts based on whether the score improves. Applied here to SKILL.md files rather than ML training scripts.
SKILL.md
The core instruction file inside a Claude Code skill folder. Contains YAML front-matter (with the activation description) and markdown instructions that shape Claude's behaviour when the skill is active.
Binary assertion
A test that returns strictly true or false and can be checked mechanically — e.g. "word count is under 300". Contrasted with subjective quality judgements that require human review. Binary assertions are the only type that can be used in an autonomous improvement loop.
Eval.json
A file inside a skill's eval/ folder defining test cases: each has a prompt, expected-output description, and a list of binary assertions. The Karpathy loop runs the skill against each test case and tallies a pass rate.
Skill activation
Whether Claude Code chooses to invoke a skill when a user types a request. Determined primarily by the YAML description: field. Low activation (as low as 20% with vague descriptions) makes a skill unreliable in production.
Skill creator
An official Anthropic-provided skill (/skill-creator) that can scaffold, evaluate, and improve other skills. Includes a built-in activation-improvement loop and a qualitative output-evaluation dashboard.
Reference files
Supplementary markdown or text files inside a skill folder — tone-of-voice guides, brand rules, example outputs — that Claude reads as context when the skill is active. Whether the skill correctly uses them is a qualitative check, not a binary one.
Pass rate
The fraction of binary assertions across all test cases that evaluate to true. The Karpathy loop commits a SKILL.md change if and only if the pass rate improves; otherwise it reverts via git reset.
Agentic operating system
Simon's commercial branding for a pre-packaged Claude Code bundle (18 skills, brand memory, self-improvement loop, Telegram interface). Community product built on standard Claude Code primitives — not an Anthropic product.
val_bpb
The validation metric Karpathy's original ML loop optimises (bits-per-byte on a validation set). The direct analogue in the skills loop is pass rate across binary assertions.
