Install VS Code

First up, we need to install VS Code. This is where we’ll operate Claude Code.

Run through the installation steps to install. Don’t open up VS Code yet, let’s check Git first so that when VS Code opens it knows it exists.

Install Git

Mac usually comes with Git pre-installed.

To check, open the terminal inside VS Code by pressing Cmd + ` or click Terminal > New Terminal in the menu.

git –version

If you see a version number, you’re good. If Git isn’t installed, your Mac will prompt you to install the Xcode Command Line Tools. Just follow the prompt and let it install, then you’re done.

What is Git?

Git is basically a “save system” for code. Developers use it to track changes to files over time. It’s free and widely used. You’ll need it because many developer tools and packages rely on it in the background.

Install Claude Code

Open VS Code and then open the terminal by pressing Cmd + ` or click Terminal > New Terminal in the menu.

Run curl -fsSL https://claude.ai/install.sh | bash in the Terminal to install Claude Code.

Check to see if it installed correctly

Close and reopen VS Code, open a new Terminal then run claude in the Terminal. This should start up Claude Code.

Can’t start Claude Code even after installing it?

Try closing and reopening your Terminal completely.

If it still doesn’t work, you may need to add Claude to your PATH. Run this in your Terminal:

export PATH=“HOME/.local/bin:PATH”

If Claude works after that by running the claude command, you’re done!

Install Node.js

Node.js is a tool that lets your computer run JavaScript code outside of a web browser. While Claude Code’s native installer doesn’t technically require it, you’ll need Node.js as soon as you want to install other tools and packages (like MCP servers) using npm. Since you’ll almost certainly need those later, it’s best to install Node.js now and get it out of the way.

How to install it (Mac):

  1. Go to nodejs.org/en/download

  2. Select macOS as your operating system

  3. Download the macOS Installer (.pkg file)

  4. Run the installer and accept the default settings

  5. Close and reopen VS Code so it picks up the new installation

Two things come bundled with Node.js:

npm (Node Package Manager) is like an app store for developer tools. When you need to install something (an MCP server, a library, a plugin), npm is the command that fetches and installs it for you.

nvm (Node Version Manager) lets you switch between different versions of Node.js. You probably won’t need it right away, but it’s handy if a tool ever requires a specific version.

To check everything installed correctly, restart VS Code, open a new Terminal in VS Code and type:

  1. node –version (you should see v18 or higher)

  2. npm –version (you should see a version number)

If both return a version number, you’re all set. If you’re having trouble send your error screenshots in a Q&A post and we’ll help you resolve it asap.

Now re-open Claude Code using the claude command!

Markdown or ‘.md’ files

Before we get started using Claude Code, it’s useful to understand Markdown.

Markdown is a lightweight way to format plain text - that AI understands well. It’s used everywhere - GitHub READMEs, documentation, blogs, and tools like Notion and Obsidian all rely on it.

The key idea is that you write in plain text with simple symbols like #, *, and -, and these get converted into formatted headings, bold text, lists, and so on when rendered.

The file extension is .md, and most code editors (including VS Code) can preview the formatted output side by side with the raw text.

It’s worth learning because almost every developer tool uses it.

For example, the CLAUDE.md file which we’ll talk about next is just a markdown file - so understanding the basics will help you structure those files clearly.

In simple, it’s a way to signify different heading levels, and formatting on text.

Checklist

Before you finalize your CLAUDE.md, run through this:

  • Could I read this in 60 seconds?

  • Does every line tell Claude something it can’t guess?

  • Am I under 30 instructions? (20 is ideal)

  • Did I include specific examples, not vague rules?

  • Would a freelancer taking over my content find this useful?

Tips

  • If you already have a project, run /init inside Claude Code. It reads your files and drafts a CLAUDE.md for you. Then cut ruthlessly - deleting is easier than writing from scratch.

  • If you’re starting fresh, just copy the template from this guide and fill in the blanks as you build.

  • Fix mistakes in real time - When Claude gets something wrong mid-session, just tell it: “Add to my CLAUDE.md that we never use hashtags on X posts.” Claude will update the file for you. Next session, it already knows.

  • Add “IMPORTANT:” to critical rules - Anthropic’s own team confirms this genuinely improves how well Claude follows instructions. Use it sparingly on the rules that matter most.

  • Drop your best posts into a reference folder - Put 5-10 top-performing posts into /content/best-performers/ and tell Claude to reference them. It’ll match your voice far better from real examples than from any written description.

  • Build it over weeks, not in one sitting - Every time Claude writes something off, add a line. Every time a rule never matters, remove it. The best CLAUDE.md files grow from real mistakes.

  • Separate personal from shared - Use CLAUDE.md (checked into your repo) for team rules, and CLAUDE.local.md (gitignored) for your personal preferences.

7. Permissions

Video · 4:27

Every time Claude Code wants to run a command, edit a file, or access something on your system, it asks for permission first. This is a safety feature, and it’s a good one. But if you’re clicking “Yes” forty times during a single session, it gets old fast.

So here are multiple ways around it:

The Nuclear Option: –dangerously-skip-permissions

You can start Claude Code like this:

claude –dangerously-skip-permissions

As it sounds, this skips all permission requests and gives Claude Code full access to do everything it wants. So long as your commands aren’t dangerous (inferring to delete your own files etc.) then it usually works out ok. Is it the safest? no! Should you use it in production environments? Absolutely not! But for isolated small projects - it’ll mostly be ok.

Boris Cherney himself (the creator of Claude Code) doesn’t use this for normal work. He’ll only use it inside a sandbox for autonomous tasks that need to run for hours without human input. The key word there is sandbox. If you run this flag on your main machine with no isolation, Claude has full access to everything - your files, your credentials, your system commands. For most of what we do, that’s unnecessary risk.

Pre-Approve What You Trust

This is how Boris Cherny does it. Instead of skipping permissions entirely, he uses /permissions to pre-allow the specific commands he knows are safe in his environment. Things like running tests, formatting code, git operations, reading files. These get saved to .claude/settings.json, which means you can check them into your repo and share them with your team.

Type /permissions in Claude Code and add commands to the allow list using wildcard syntax. Here’s an example of what that looks like in your settings file:

{

“permissions”: {

“allow”: [

“Bash(npm run *)“,

“Bash(git add:*)“,

“Bash(git commit:*)“,

“Bash(git status:*)“,

“Bash(git diff:*)“,

“Bash(cat:*)“,

“Bash(ls:*)“,

“Edit(/src/**)”

]

}

}

The result: fewer interruptions, but you still have guardrails around anything unexpected. You’re telling Claude “these specific things are fine, everything else still needs my approval.”

It’s the sweet spot between safety and speed.

But what does that all actuall mean above? and what should you actually allow?

Here’s my 101 thinking…

For most projects, these are the safe ones to pre-approve:

  • Reading files and folders - Claude needs to look at your project to understand it. Letting it read freely saves you clicking “Yes” every time it wants to check a file.

  • Running your dev server - if you’re building a web app, Claude will need to start it up to test things. Commands like npm run dev or npm start are safe to allow.

  • Running tests - you want Claude running tests constantly. That’s how it checks its own work. Allow your test commands like npm test or pytest.

  • Git operations - things like checking what’s changed (git status, git diff), staging files (git add), and committing work (git commit). These are standard development operations.

  • Formatting and linting - commands that tidy up code style. These can’t break anything, they just make things neat.

Which with comments, equates to this below. Note: this is broken JSON formatting just to show comments, copy the permissions file above.

Can Claude set this up for you?

Yes. You don’t need to write JSON by hand - although we have included the full JSON below if you want it!

You can instead just tell Claude Code:

“Add the common safe permissions to my settings - things like reading files,

running tests, git operations, and starting the dev server.

Show me what you’re adding before you save it.”

Claude will write the permissions block and add it to your .claude/settings.json for you.

Once it’s saved, those permissions persist across sessions. You set it up once and you’re done.

Protecting Sensitive Files

Projects like GSD use codebase mapping and analysis commands to read files to understand your project.

Protect files containing secrets by adding them to Claude Code’s deny list:

  1. Open Claude Code settings (.claude/settings.json or global)

  2. Add sensitive file patterns to the deny list:

{

“permissions”: {

“deny”: [

“Read(.env)”,

“Read(.env.*)“,

“Read(**/secrets/*)“,

“Read(**/credential)“,

“Read(**/*.pem)“,

“Read(**/*.key)”

]

}

}

This prevents Claude from reading these files entirely, regardless of what commands you run.

GSD includes built-in protections against committing secrets, but defense-in-depth is best practice. Deny read access to sensitive files as a first line of defense.

Global vs Local Installs

Video · 8:10

We’re about to talk about different packages that you can install from others (called slash commands, skills, hooks and more).

When you install a slash command, skill or hook in Claude Code, you’ve got two options:

local or global.

Local installs to your current project folder only. It lives inside ./.claude/ in that specific project directory. You can see it right there in your file structure, open each file, and understand exactly what it does.

Global installs to your home directory. It lives inside ~/.claude/ and is available across every project you work on. You won’t see these files in your project folder unless you go looking for them.

Most install instructions you’ll find online default to global.

That’s fine once you know what you’re doing, but when you’re starting out, I’d recommend installing locally first so you can see exactly what’s been added, open the files, and understand what you’re working with. If you decide to keep it, you can always install globally later.

The difference comes down to one character in the path when writing the installation line:

~/.claude/ ← global (~ = your home directory, shared across everything)

./.claude/ ← local (. = this project folder only)

Here’s where they actually land:

GLOBAL → ~/.claude/

~/.claude/

├── commands/

1. Slash Commands

Video · 10:52

Think of slash commands as shortcuts for your most common tasks. Instead of typing out the same instructions every time, you create a command once and trigger it whenever you need it.

You invoke them in the terminal by typing a forward slash followed by the command name, like /generate-proposal or /write-blog-outline.

So say, for example, you write client proposals regularly. Every time, you’re giving Claude the same brief - tone of voice, structure, what to include, what to leave out. With a slash command, you write those instructions once, save them as a markdown file, and from that point on you just type /generate-proposal and it follows your exact process every time.

They’re reusable prompts that streamline your workflow. You’re not re-explaining yourself on every conversation.

And here’s where it gets really useful - you can pass in $ARGUMENTS to make your commands take dynamic data. So instead of a static prompt, you can type something like /generate-proposal Amazon.com website redesign and Claude will slot “Amazon.com website redesign” into the right place in your instructions. One repeatable command, infinite use cases.

3. Skills

Video · 19:24

Think of skills as the autopilot version of slash commands. Instead of you typing a command to trigger them, Claude detects when a skill is relevant and uses it automatically.

So say you’ve got a skill that defines how you write blog content - your tone of voice, structure, formatting rules. You don’t need to type /write-blog every time. You just ask Claude to write a blog post and it pulls in those instructions on its own because the skill’s description matches what you’re doing.

Note: the description for your skills is really important, it is what Claude Code uses to determine if the skill is relevant to use at all.

The key difference: slash commands are you-invoked (you type it, it runs). Skills are Claude-invoked (Claude recognises the task and applies the right skill automatically).

You can still invoke a skill manually with a slash command too - it works both ways.

Skills also support a full folder structure. So if your skill needs reference files, templates, or examples, you can bundle them all together in one place.

Setting Up Your Skills Folder

Same idea as commands, slightly different location.

Step 1: Create the folder structure

Inside your .claude folder, create a skills directory. Each skill gets its own subfolder with a SKILL.md file inside:

your-project/

└── .claude/

├── commands/

└── skills/

└── blog-writer/

└── SKILL.md

The folder name becomes the skill name. The SKILL.md file contains the instructions.

Step 2: Add the frontmatter

This is the bit that makes skills different from slash commands. At the top of your SKILL.md, you add a small block called frontmatter that tells Claude what the skill is for and when to use it:

name: blog-writer

description: Write blog posts using our brand voice and structure.

Use when asked to write blog content, articles, or long-form posts.

Your instructions go here…

The name becomes the /blog-writer command.

The description is what Claude reads to decide whether this skill is relevant to what you’re asking. Like we mentioned, the better your description, the more reliably Claude Code will pick it up.

These are the only two required fields. You can also add extra fields to your frontmatter to control how a skill behaves:

name: blog-writer

description: Write 300-word blog posts using our brand voice and structure.

Use when asked to write blog content, articles, or long-form posts.

allowed-tools: Read, Grep, Glob

argument-hint: [topic or brief for the blog post]

Your instructions go here…

allowed-tools limits what Claude can actually do when using this skill. In this example:

  • Read - Claude can read files (like your tone-of-voice.md and example-post.md)

  • Grep - Claude can search inside files for specific text or patterns

  • Glob - Claude can scan folder structures to find files by name or type

That’s it. With just those three, Claude can read your reference files and find what it needs, but it can’t edit anything, create new files, or run terminal commands. It’s read-only.

Some other common options you might use:

  • Write - Claude can create and edit files

  • Bash - Claude can run terminal commands

  • Task - Claude can spawn sub-agents

By default, a skill has access to everything. You only need allowed-tools when you want to restrict that. For a blog writing skill, read-only makes sense. For a skill that builds and deploys something, you’d need Write and Bash too.

argument-hint is the placeholder text that shows up when someone types /blog-writer. So in this case, typing /blog-writer would show [topic or brief for the blog post] as a hint. Purely cosmetic but helpful when you’ve got a few commands and can’t remember what each one expects.

Step 3: Add supporting files (optional)

This is where skills really shine as they can be far more comprehensive than slash commands (think of them as an evolution on slash commands).

Need Claude to reference your tone of voice guide when writing? Drop it in the same skills folder:

skills/

└── blog-writer/

├── SKILL.md

├── tone-of-voice.md

└── example-post.md

Then reference those files in your SKILL.md instructions. Claude can read anything in the skill folder.

Anthropic’s guide on building skills

You’ve got two options with skills: build your own or install ones other people have already made.

Building your own is best for anything specific to your business - your brand voice, your proposal format, your client onboarding process. Nobody else can write those for you because nobody else knows your business like you do.

But for general stuff like copywriting frameworks, SEO content structures, or landing page optimisation? Don’t reinvent the wheel. There are hundreds of free, well-built skills out there ready to install. Clone the repo, copy the skill folder into your .claude/skills/ directory, and you’re done.

The next lesson is a curated list of the skills we think are going to be most useful for you.

Global vs Project - Where Should Skills Live?

Two options:

  • Global (~/.claude/skills/) - available in every project you open

  • Project (your-project-folder/.claude/skills/) - only available in that specific project

We just demonstrated installing them locally in a project, however the general standard is global for most things. Your brand voice, your writing style, your proposal format - you want those everywhere, not just in one project folder.

Use project-level skills when something is genuinely specific to one build. For example, a skill that understands the data structure of one particular client’s dashboard. You wouldn’t want that bleeding into every other project.

When in doubt, go global. You can always move a skill to project-level later if it starts getting in the way. Going the other direction (project to global) means copying files around, which is more hassle than it needs to be.

One thing to know: global skills live in a hidden system folder on your machine. You won’t see them in your project’s file tree, but they’re still there and Claude still reads them. If you ever wonder why Claude is applying instructions you can’t see in your project, it’s probably a global skill doing its thing. You can check what’s loaded by running /skills in the terminal.

5. Hooks

Video · 2:38

Hooks are shell scripts that run automatically at specific points during Claude Code’s lifecycle. Think of them like triggers: before a tool runs, after a file is edited, when a session starts, when Claude finishes a task - they activate automatically based on a particular event.

The difference between hooks and skills is important. Skills are suggestions - Claude can choose to follow them or not. Hooks are deterministic and they don’t need AI tokens to run!

If you set up a hook that blocks dangerous commands, it blocks them every time. No exceptions, no “Claude forgot.”

You might not even need any hooks, but it’s worth knowing what they are.

How to install: Hooks live in your .claude/settings.json file (project-level) or ~/.claude/settings.json (global). Like commands and skills, you can import other’s pre-built hook scripts, by cloning the repo and copy the script files into your .claude/hooksdirectory, then reference them in your settings.

8. Agents, sub-agents & teams

Video · 16:59

There’s a progression here from the previous concepts of commands, skills, hooks & MCPs - understanding it will save you a lot of wasted tokens and frustration.

Single agent

This is what you’ve been using so far. One terminal window, one conversation. That single agent can use everything we’ve covered: commands, skills, hooks, your CLAUDE.md file. You’re building a landing page, writing a script, fixing a bug. This is where you’ll spend most of your time and it works great until the context window starts filling up. The longer it runs, the worse it gets. It forgets context, introduces bugs, and you end up repeating yourself. See Context Rot.

Sub-agents

When a single context window gets too much context it gets overwhelmed, so it’s worth delegating. Sub-agents are specialists that run in their own context window with their own system prompt and tool access. The main agent hands off a task, the sub-agent does the work, and sends back the result. Think hiring a specialist versus doing everything yourself.

Two reasons to use sub-agents:

1. Improve output quality (specialist vs generalist)

One Claude delegating to focused specialists within the same session. This is because each agent masters their domain with deep, current knowledge and isn’t bloated with context.

2. Get things done faster (parallel non-dependent tasks)

Multiple terminals doing unrelated work at the same time in parallel. Agents can coordinate, sharing context and handing off tasks (in sub-agents they do this through the main orchestrator agent i.e. your first open terminal window).

Agent teams

When sub-agents aren’t enough because the specialists actually need to talk to each other. This is the newest addition and it changes things significantly.

The key difference: hub-and-spoke vs real collaboration

Sub-agents can only report back to the parent. So if you have a researcher, a writer, and a reviewer as sub-agents, the reviewer can’t see the research and the researcher can’t see the reviewer’s feedback. The writer in the middle handles everything, and that main agent becomes the bottleneck.

Agent teams solve this with a shared task list. The main agent spins up teammates (literally in their own claude code instance so only a small bit of the main context is passed), but instead of everything flowing through one central point, teammates can communicate directly with each other. The blog writer messages the newsletter writer with their hook angle so they don’t duplicate. They coordinate without you being the middleman.

When to use what

A single agent handles one feature, one file, one conversation. That’s 80% of your daily work.

Sub-agents are for when you need specialists and your context is getting too large. A researcher gathering information, an SEO auditor crawling a site, a code reviewer checking your work. Self-contained tasks where you want the result, not the process. The sub-agent works in its own context window, does the job, and hands back a summary. Your main session stays clean.

Agent teams are for complex builds where parallel tasks need cross-collaboration. Building a SaaS app where the API developer, the frontend developer, and the tester all need to be aware of each other’s work. They’re working in parallel, but they need to coordinate - the tester needs to know what the API developer just shipped, and the frontend developer needs to know the endpoints are ready.

Start simple. Graduate to sub-agents as the project gets more complex. Reach for agent teams when the task genuinely needs that cross-collaboration.

So many concepts…. so when should I use a sub-agent vs just using a skill?

This trips people up, because there are sub-agent repos out there with “email copywriter” and “social media strategist” agents. You probably don’t need them.

Use a skill when you want Claude to be better at something in your current conversation. You’re iterating, tweaking, asking for changes. Email copy, landing pages, social content - that’s skill territory. You want the conversation.

Use a sub-agent when the task is self-contained. “Go away, analyse this, come back with a report.” Research, audits, code reviews, documentation. You want the result, not the process.

Agents can also have their own set of skills.

Same rule as skills: don’t install hundreds. A handful of focused sub-agents beats a bloated collection every time.

10. The most important thing

Video · 10:54

Planning before you build is the single biggest factor in whether your Claude Code project succeeds or fails.

Here’s why it matters so much with AI specifically. Claude Code has a context window - think of it as short-term memory. Every message you send, every file it reads, every mistake it makes and you correct - all of that fills up the memory. And as that memory fills up, quality degrades. You’ve probably seen it yourself: Claude starts cutting corners, forgetting what you said earlier. That’s context rot, and it’s why the output quality varies so much.

Planning solves this, but only when it’s documented somewhere for later reference.

When you have a clear plan written down in files that Claude can reference, it doesn’t need to hold everything in memory. It reads the plan, does the work, and moves on.

So how do you actually plan? There are three levels I’d recommend, and which one you use depends on how big your project is.

Level 1: Plan Mode (Built-in, No Setup)

For quick builds - a landing page, a simple tool, a single automation - you don’t need any framework at all. Claude Code has Plan Mode built right in.

Press Shift+Tab twice to enter Plan Mode. In this mode, Claude becomes an architect. It can read your files, analyse your codebase, and think through the approach, but it can’t actually change anything. It’s thinking, not doing.

Tell it what you want to build and let it interview you. Let it ask questions. Let it poke holes in your thinking. Then have it write the plan to a plan.md file before you give it the green light to build.

Why a file and not just a conversation? Because when Claude’s context window fills up and it compacts, that plan disappears from memory. A written plan.md survives compaction. It becomes working memory that persists across sessions.

“I want to build [brief description]. Interview me about the requirements

using detailed questions. Ask about technical implementation, edge cases,

and tradeoffs. Don’t start coding until we’ve agreed on a plan.

Write the final plan to spec.md.”

This is your default. Use it for anything that takes less than an hour to build.

No setup required, no framework to install, just Claude and a clear plan.

Level 2: PRD Generator (The Bridge Between Idea and Build)

For bigger projects - a client deliverable, a multi-page application - you want more structure than a quick plan. This is where a Product Requirements Document comes in.

A PRD is exactly what it sounds like: a document that defines what you’re building, who it’s for, what it needs to do, and how success is measured. In the world of software, product managers write PRDs before developers write code. In the world of Claude Code, you generate one before Claude writes code.

Think of it as the bridge between “I have an idea” and “Claude, build this.”

The PRD typically includes a project overview, tech stack recommendations, architecture decisions, feature tasks broken down with acceptance criteria, and an order of operations so Claude knows what to build first. It’s everything Claude needs to build your project without constantly asking you questions or making assumptions - it’s worth the upfront effort!

Quick Setup with prd-taskmaster

This is a Claude Code skill that generates comprehensive PRDs through an interactive interview process. Install it with one command in your terminal:

curl -fsSL https://raw.githubusercontent.com/anombyte93/prd-taskmaster/master/install.sh | bash

This installs the skill to the global folder ~/.claude/skills/prd-taskmaster/

Once installed, just tell Claude what you want to build and the skill activates automatically. It asks you targeted questions, validates the output against 13 quality checks, and produces a complete PRD ready for implementation.

“Think of it as your AI product manager that asks the right questions, writes comprehensive specs, and sets you up for successful implementation.”

Alternative: Slash Command Approach

If you prefer something lighter, you can install PRD Generator Gist as a slash command instead of a skill. Download the generate-agent-prd.md file, place it in your project’s .claude/commands/ folder, restart Claude Code, and run:

/generate-agent-prd A SaaS app for tracking fitness goals

Replace the description with your project idea. Claude generates a complete PRD covering project overview, architecture tasks, testing infrastructure, feature tasks with acceptance criteria, and a state tracking section. The key here is that architecture and testing get set up first, before any features get built. That ordering alone prevents a huge number of problems.

Level 3: GSD - Get Shit Done (Full Framework for Bigger Builds)

For anything that takes more than a few days - a full application, a multi-feature platform, a client project with real complexity - you want a proper framework.

The popular options in the developer world are things like BMAD (simulates an entire agile team with Analyst, PM, Architect, and Developer agents), GitHub’s Spec Kit (structured slash commands for specification-driven development), and Taskmaster (AI-powered task management). They all work. But they’re all designed for enterprise grade applications - not spinning up an application for a small team or to serve to clients. Sprint ceremonies, story points, stakeholder syncs - it’s not what small teams and solo builders need to get things done quickly.

This is where I’d use GSD. Behind the scenes, GSD handles context engineering, prompt formatting, sub-agent orchestration, and state management.

We explain GSD in the video above, but for a bit more detail, here’s me explaining how to get a project started and installed (this will be covered in the Guided Builds section too).

Why GSD works for us: It leverages the same interview-style questioning we see in Plan Mode, but the key difference is the level of detail in the breakdown. GSD splits your project into milestones, phases, and individual tasks - each small enough to fit cleanly in a fresh context window. This is what solves context rot. Instead of one massive session where quality degrades over time, each task runs in its own clean 200k token context with zero accumulated garbage.

The core workflow uses three commands in sequence for each project phase:

/gsd:plan phase X

/gsd:execute phase X

/gsd:verify phase X

Plan creates the detailed breakdown. Execute spawns fresh sub-agents to build each task. Verify checks the work and creates fix plans if something breaks. Then you move to the next phase.

Context is pulled from project-level documents that GSD maintains:

  • ROADMAP.md - the big picture of milestones and phases

  • REQUIREMENTS.md - what needs to be true when you’re done

  • STATE.md - where you are right now

As well as phase-specific documents that give Claude exactly the context it needs for each piece of work, nothing more. Every task gets its own atomic git commit, so if something goes sideways, you can roll back to exactly the point before it broke.

Install it with one command:

npx get-shit-done-cc

Picking the Right Level

Here’s the simple way to think about it:

Plan Mode (no setup) - Quick builds under a day. A landing page, a simple script, a one-off tool. Just interview with Claude and write a spec file.

PRD Generator (<5 minutes setup) - Multi-hour projects with clear scope. A client deliverable, a complete automation system, a tool with multiple features. Generate a proper requirements document first.

GSD (<5 minutes setup) - Multi-day builds with real complexity. A full SaaS application, a platform with many moving parts, anything where context rot would normally kill your momentum.

The mistake most people make is jumping straight to building without any planning at all. The second mistake is using an enterprise framework when a simple plan would do. Match the tool to the job. A quick spec file for small things. A PRD for medium things. GSD for big things. But always, always plan first.

Two minutes of rewriting your prompt beats twenty minutes of going back and forth in a conversation that’s already off the rails.

1. Why Skills building is your top priority

I know we’ve already spoken about Skills - but I decided to make an entire module on it because I believe it single handedly is the most important topic you can learn inside Claude Code (by the way, the concept exists for other software too)

Skills are how you teach AI to work like you once, and benefit every time. I say you, assuming you’re an expert in some subject area, but it can be applied more broadly “Skills are how you teach AI to work like an expert marketer/VA/financial consultant/[insert role here] once, and benefit every time”

They’re the difference between starting every conversation from scratch and having an AI that already knows the instructions on what path it needs to follow to deliver a given outcome and has access to the KNOWLEGDE to do it in your way (your brand voice, your processes, your frameworks, and your standards).

What is a skill?

A skill is a folder containing instructions, reference material, and supporting files that teach Claude how to handle specific tasks or workflows. Instead of re-explaining your preferences, processes, and domain knowledge every conversation, you teach Claude once.

Think of it this way. Right now, most people use AI like this:

“Write me a LinkedIn post. Here’s my tone of voice. Here’s what I never say. Here’s my audience. Here’s my brand colours. Here’s examples of my best posts. Now write something about automation.”

Every. Single. Time.

With a skill, you front-load all of that context into a structured folder (called /references). Next time you say “write me a LinkedIn post about automation,” Claude already knows your voice, your audience, your banned words, and your best-performing examples. It just writes.

That’s the surface-level benefit, that compares it to a good system prompt. But really, the real power goes much deeper.

Why skills matter RIGHT NOW

  1. They solve the biggest problem with AI: inconsistency

Without skills, every conversation is a blank slate. Claude doesn’t remember what you told it yesterday. Your brand voice doc? Gone. Your copywriting frameworks? Gone. The specific way you like proposals structured? Gone.

Skills fix this permanently. Your brand voice, your ICP, your frameworks, your process steps all persist across every conversation. Claude doesn’t just remember; it applies them automatically without you asking.

  1. The way they’re built protects your context window, and improves output quality

One of the key principles we’ve taught already is “Point, Don’t Dump”: keep your CLAUDE.md lean and reference files instead of pasting everything inline. Skills take this further with what’s called progressive disclosure.

  1. The name and description of each skill are loaded in first (always part of Claude’s system prompt)

  2. If the name and description match the request, the instruction set is loaded in (SKILL.md)

  3. If the instructions link to important reference info elsewhere (e.g. brand voice), then Claude will pull in that separate file only when it is necessary (and not use it in every request)

Claude pulls in exactly what it needs, when it needs it. Nothing wasted. This is why skills are more token-efficient than dumping everything into CLAUDE.md. They’re designed for how LLMs actually process information - up to ~200 lines at a time in any file.

  1. They’re portable and stackable

A skill works across Claude.ai, Claude Code, Cowork, and the API. Build once, use everywhere. And because Claude can load multiple skills simultaneously, they compose together. Your brand-voice skill works alongside your copywriting-coach skill works alongside your platform-rules skill. Each one is modular, self-contained, and focused.

The shift from tools to knowledge

MCP gave AI access to tools: the ability to read Notion databases, create Linear tickets, send emails. But tool access without workflow knowledge is like giving someone a professional kitchen without recipes. Skills are the recipes.

Anthropic’s official guide puts it clearly: MCP is what Claude can do. Skills are how Claude should do it. Together, they turn raw capability into reliable, repeatable workflows.

As MCP adoption grows (and it’s growing fast: Notion, Asana, Linear, Slack, GitHub all have MCP servers now), the demand for skills to guide those tools grows with it. Every MCP connector needs workflow guidance. Every business process needs to be encoded somewhere.

The compounding effect

Every skill you build makes the next one easier. Your brand-voice skill informs your copywriting skill. Your ICP reference file gets reused across your outreach skill, your content skill, your proposal skill. Your frameworks get refined with every client engagement and fed back into better skills.

This is the “never starting from scratch” principle in action. Each skill is a layer of accumulated expertise that future work builds on. Over time, you’re not just automating tasks. You’re building an ever-growing library of institutional knowledge that makes Claude more effective with every addition. Each skill can be customized per client, with their own domain knowledge.

2. Skills build process - the 101

Useful for tips -> Anthropic official skill building guide

A brilliant lesson on doing the wrong thing, and correcting it later for better results (Skills related) - lessons learned from Skill building

What makes up a skill?

Minimum, one file in a .claude/skills/[skill-name] folder. But if you want it to be great, it’ll have more files.

• SKILL.md (required): Instructions in Markdown with YAML frontmatter

• scripts/ (optional): Executable code (Python, Bash, etc.)

• references/ (optional): Documentation loaded as needed

• assets/ (optional): Templates, fonts, icons used in output

Core Design Principles - Progressive Disclosure

I.e. disclosing the right things at the right time - and not bloating the SKILL.md context unnecessarily!

Skills use a three-level system:

• First level (YAML frontmatter) (~100 words):

Always loaded in Claude’s system prompt. Provides just enough info for

Claude to know when each skill should be used without loading all of it

into context. Just enough for Claude to decide if the skill is relevant

• Second level (SKILL.md body) (~200 lines max):

Loaded when Claude thinks the skill is relevant to the current task.

Contains the full instructions and guidance. Overview, quick start,

navigation map. Points to references but doesn’t include their content.

• Third level (Linked files) (~200-300 lines each):

Additional files bundled within the skill directory that Claude can

choose to navigate and discover only as needed. Detailed documentation

Claude reads only when needed. Modular and focused on single topics.

The 200-line limit in the Skill.md isn’t arbitrary. It’s based on how much context an LLM can efficiently scan to decide what to load next. Keep the entry point under ~200 lines, and Claude can quickly:

  • Understand what the skill offers

  • Decide which reference file to read

  • Load just that file (another ~200-300 lines)

References (Linked files)

Contextual information that helps bring KNOWLEDGE to the skill instruction set (SKILL.md)

These should be instructions that are relevant for the specific context or knowledge of the skill.

For your own business, it is highly recommended to build out the following reference material that’ll sit across multiple skills:

  • Personal Profile / Background (like your own CV)

  • Product Offering

  • Ideal Customer Profile (ICP)

  • Brand Voice

  • Content Strategy Document

The Six Steps To Building a Skill

Every skill follows the same build sequence. Work through these in order. This applies to editing an existing Skill too.

Step 1: Name

The folder name and the name field in your YAML frontmatter.

Rules:

  • kebab-case only: copywriting-coach not Copywriting Coach or copywriting_coach

  • Must match the folder name exactly

  • Cannot contain “claude” or “anthropic” (reserved by Anthropic)

  • No spaces, no capitals, no underscores

Example:

copywriting-coach/

├── SKILL.md

└── references/

name: copywriting-coach

That’s it. Simple, but get it wrong and the skill won’t upload.

Step 2: Trigger

The description field in your YAML frontmatter - alongside the name in the — section. This is the single most important line in your entire skill. It’s what Claude reads to decide whether to load your skill or ignore it. Activation rates in a poorly described skill are regularly 20% - that’s 1 in 5 times that you’d want it to activate that it actually would - poor!

The format (in this order):

  1. The event or situation in which this skill gets used

  2. The terms and phrases that should trigger it - almost like keyword tagging

  3. What the skill actually does

  4. What should NOT trigger it

  5. File types it handles (if relevant)

Good example:

name: copywriting-coach

description: >

Use when a user wants marketing copy reviewed, scored, or

rewritten. Triggers on “copywriting”, “sales copy”, “headline”,

“CTA”, “landing page copy”, “email sequence”, “ad copy”,

“conversion copy”. Analyses copy against proven frameworks

(AIDA, PAS, BAB), scores weaknesses, and rewrites to match

brand voice and ICP. Does NOT trigger for blog posts, SEO

content, or general creative writing.

Bad example:

name: copywriting-coach

description: >

Analyses and improves marketing copy using proven copywriting

frameworks.

That bad example leads with what it does (backwards), has no trigger phrases, no negative triggers, and is too vague for Claude to know when to load it.

Test it: Ask yourself, if someone described this task five different ways, would your description catch all of them? If not, add more trigger terms.

Keep it concise: ~100 words max. Dense and specific beats long and vague.

Step 3: Outcome

Before writing a single instruction, define what “done” looks like. This is your success criteria: the deliverable the user walks away with.

Define:

  • The specific output format (file type, structure, style)

  • What makes a good output vs. a bad one

  • What the user should NOT have to do manually after the skill runs

Example for the copywriting skill:

The user receives a scored breakdown of their existing copy against the relevant framework, plus a rewritten version that matches their brand voice and speaks directly to their ICP’s pain points. Each change is explained with the framework principle behind it. The user can use the rewrite immediately without further editing.

You can add this into the SKILL.md as a formal section using # Outcome. It’s the mental clarity you need before writing the process steps. If you can’t describe the outcome in two sentences, you don’t understand the skill well enough yet.

Step 4: Dependencies

Everything the skill needs access to in order to deliver that outcome.

Four categories:

Tools and connectors # Tools

  • Built-in tools needed (code execution, file creation, web search)

  • MCP connectors required (Notion, Google Drive, Nano Banana, etc.)

  • Fallback behaviour if a connector isn’t available

Reference files (the references/ folder):

  • Contextual knowledge that brings depth to the instructions

  • One topic per file, self-contained, ~200-300 lines each

  • Claude reads these only when a specific step points to them

Scripts (the scripts/ folder):

  • Executable code for deterministic tasks (validation, formatting, data processing)

  • Use scripts when code is more reliable than language instructions

  • Claude can execute these without loading them into context

Assets (the assets/ folder):

  • Templates, fonts, icons, brand files used in the output

  • Things Claude uses to produce the deliverable, not things Claude reads for knowledge

Example for a copywriting skill:

copywriting-coach/

├── SKILL.md

└── references/

├── copywriting-frameworks.md Frameworks library (AIDA, PAS, BAB, etc.)

├── brand-voice.md Tone, language rules, examples

├── ideal-customer-profile.md Who the audience is, their pain points

└── product-offering.md What’s being sold, key differentiators

No scripts or assets needed here. This is a pure knowledge + workflow skill.

For your own business, build out these reference files as a foundation:

  • Personal profile / background (like your own CV)

  • Product offering

  • Ideal customer profile (ICP)

  • Brand voice guide

  • Content strategy document

These get reused across multiple skills. Your brand-voice.md works in your copywriting skill, your content skill, your outreach skill, your proposal skill.

Step 5: Step-by-Step

The step-by-step process inside the SKILL.md body # Instructions. This is where you define exactly what Claude does, in what order, referencing which dependencies at each step.

For each step, answer:

  1. What does Claude do?

  2. Does the user need to provide input or make a decision here? (Human-In-The-Loop)

  3. Which reference file should Claude read before this step?

  4. What’s the expected output before moving to the next step?

If you find yourself answering this question in A LOT of detail (e.g. adding examples), it’s likely you’re embedding knowledge into the SKILL.md file - of which can be offloaded to a separate reference file.

Writing tips from the official docs:

  • Use the imperative form (“Ask the user”, “Read the reference file”, “Score the copy”)

  • Be specific and actionable, not vague (“Validate the data before proceeding” is bad)

  • Keep the SKILL.md under ~200 lines; offload detail to references

  • Reference files clearly with guidance on when to read them

Step 6: Edge Cases

What happens when things go wrong, get weird, or the input is unexpected. This is the difference between a fragile skill and a robust one.

Define:

  • Common failure modes and how to handle them

  • Edge cases (vague input, missing data, conflicting instructions)

  • Validation checks before critical steps

  • Graceful fallbacks instead of silent failures

  • Testing criteria (does it trigger correctly? does the output meet the bar?)

references/copywriting-frameworks.md

Copywriting Frameworks

AIDA — Attention, Interest, Desire, Action

Best for: Landing pages, sales pages, long-form copy

  • Attention: Hook with a bold claim, question, or stat

  • Interest: Expand on the problem or opportunity

  • Desire: Paint the outcome, use social proof, show benefits

  • Action: Clear, specific CTA with urgency

PAS — Problem, Agitate, Solution

Best for: Ads, short-form copy, email openers

  • Problem: Name the pain point directly

  • Agitate: Make it feel urgent — what happens if they don’t act?

  • Solution: Present your offer as the answer

BAB — Before, After, Bridge

Best for: Email sequences, case studies

  • Before: Describe the reader’s current situation

  • After: Describe what life looks like after the transformation

  • Bridge: Your product/service is the bridge between the two

4U Formula — Useful, Urgent, Unique, Ultra-specific

Best for: Headlines, subject lines, social media hooks

Score each headline 1-5 on all four dimensions.

FAB — Features, Advantages, Benefits

Best for: Product descriptions, comparison content

  • Feature: What it does

  • Advantage: Why that matters vs alternatives

  • Benefit: What the customer gets emotionally or practically

4Ps — Promise, Picture, Proof, Push

Best for: Email sequences, webinar invites

  • Promise: Lead with the outcome

  • Picture: Help them visualise achieving it

  • Proof: Testimonials, stats, case studies

  • Push: CTA with a reason to act now

How Progressive Disclosure Works in Practice

Here’s what actually happens at runtime:

User: “Can you review this landing page copy for me?”

Claude sees: YAML frontmatter (Level 1) → “review this copy” matches → loads SKILL.md

Claude reads: SKILL.md body (Level 2) → follows Step 1, collects the copy and asks user questions

Claude hits Step 2: → reads references/product-offering.md (Level 3)

→ reads references/ideal-customer-profile.md (Level 3)

→ reads references/brand-voice.md (Level 3)

Claude hits Step 3: → reads references/copywriting-frameworks.md (Level 3)

→ selects AIDA for landing page

→ scores the copy

Claude hits Step 4: → rewrites using framework + brand voice + ICP knowledge

Claude hits Step 5: → explains each change with framework reasoning

At no point was all of the reference material loaded at once - so only ~200-300 lines loaded in at any one time. Claude pulled in exactly what it needed, when it needed it. That’s progressive disclosure in action.

5. Testing your skill

Before shipping, run three types of tests:

  1. Trigger tests: Does it load on obvious requests? Does it load on paraphrased requests? Does it stay quiet on unrelated topics?

  2. Functional tests: Does each step produce the expected output? Do reference files load correctly? Does error handling work?

  3. Quality comparison: Is the output with the skill meaningfully better than without it? Does it save the user from having to correct or redo work?

What makes it interesting: it evolves. If you keep correcting the same behaviour during sessions, the wrap-up step notices the pattern and proposes a SOUL.md update. You approve or reject. Over time, the personality sharpens to match exactly how you want to work – without you having to manually edit config files.

USER.md - Your Profile

Your name, role, communication preferences, working style. The /start-here flow fills this in.

Skills read it to calibrate their approach. If you’re a senior marketer, you get different output than someone new to copywriting. If you prefer bullet points over long paragraphs, it adapts.

Daily Memory - Session Continuity

One file per day. Each session gets a structured block:

  • Goal – what you set out to do

  • Deliverables – what was produced, with file paths

  • Decisions – what was decided and why

  • Open threads – anything unfinished for next time

When you start a new session, it reads today’s file and yesterday’s. It specifically looks for Open threads from the last session – those become the starting context. So when you come back and say “let’s keep going,” it already knows what “going” means.

Learnings - The Long-Term Feedback Loop

Every skill has its own section in context/learnings.md. When you give feedback during wrap-up – “that copy was too salesy” or “the research missed Reddit entirely” – it gets logged under that skill’s section.

Next time the skill runs, it reads its learnings first. Session 1 copywriting and session 20 copywriting are measurably different because every correction compounds.

This is separate from memory. Memory is what happened. Learnings is what was learned.

How it all connects

At the start of every session, the agent reads SOUL.md, USER.md, and the last two days of memory. That’s how it knows who it is, who you are, and what you were working on.

At the end of every session (via wrap-up), it writes back: updates memory, logs learnings, and proposes personality tweaks if needed.

No database. No external service. Just files you can open, read, and edit yourself. The agent reads them at session start and writes to them at session end. You always know exactly what it knows.

Agentic OS - Self-Maintaining

One of the biggest differences between Agentic OS and a set of prompt templates: it maintains itself.

You don’t manually update config files. You don’t register new skills by hand. You don’t keep a mental list of what’s installed. The system tracks all of that and keeps itself in sync.

How self-maintenance works

Every session, the agent runs a sync check at startup. It compares what’s actually on disk against what’s documented:

  • New skill folder appeared? It reads the skill, registers it in the system files, links it to related skills, and tells you it’s ready.

  • Skill folder deleted? It asks if you want to clean up the references.

  • New MCP tool connected? It documents it automatically.

  • Brand context files getting stale? It flags anything older than 30 days and asks if you want to refresh.

At the end of every session, wrap-up runs the same check again to catch anything that changed during your work.

The result: the system always knows what it has available, and the documentation always matches reality.

How skills self-improve

Every skill has two self-improvement mechanisms built in:

Learnings - feedback you give during wrap-up (“that was too salesy”, “the format was wrong”, “it missed the CTA”) gets logged to the skill’s section in the learnings file. Next time the skill runs, it reads its learnings first and avoids repeating mistakes.

Direct fixes - if feedback points to a broken step or wrong approach, the wrap-up doesn’t just log a note. It edits the skill file directly. Missing step? Added. Wrong output format? Fixed. Bad default? Changed. The skill is immediately better for next time.

Rules - every skill has a Rules section that accumulates hard constraints. When something goes wrong and you flag it, a dated rule gets added: “2026-03-13: LinkedIn posts must NOT end with a question.” The skill reads these rules before every run.

Building your own skills

This is where it gets powerful for your specific business.

Say you write weekly investor updates, or client onboarding emails, or Skool lesson posts. If you’re doing the same type of task repeatedly, you can turn it into a skill.

Say “build me a skill for writing weekly Skool lessons.” The skill creator asks for 2-3 examples of good output, extracts the pattern, scaffolds the methodology, tests it, and registers it. From then on, that skill runs with your brand context, logs to its own learnings section, and improves with feedback - just like the built-in skills. It uses Anthropics skill-creator skill but I’ve enhanced it by ensuring that each Skill follows conventions of this project (e.g. referencing brand context in the correct place, putting outputs in the projects folder).

Your custom skills live alongside the ones that ship with the system. They get the same self-improvement mechanisms, the same brand context loading, the same wrap-up integration.

What to update vs what to leave alone

Update regularly: Pull bash scripts/update.sh whenever we announce new releases. Zero risk to your data UNLESS you’ve updated Skills files.

Don’t manually edit: CLAUDE.md, the README, skill registry tables. The system manages these itself. If you edit them by hand, the next sync check might overwrite your changes.

Do manually edit if you want: SOUL.md (personality tweaks), USER.md (your preferences), individual skill files (if you want to change a methodology). These are yours. The update script never touches them.

Let the system handle: Skill registration, context matrix updates, learnings sections, daily memory files.

You focus on the work you want to complete, let the system keeps itself organised with it’s self-maintenance features :)

Agentic OS - Multi-Client Setup

The Agentic OS has been designed to handle multiple brands inside one setup, with clean separation and appropriate context inheritance.

So one installation, as many clients as you want. Each one gets its own brand context, memory, learnings, and outputs. But they all share the same methodology and skills.

This builds on the way claude code treats agents / subagents -> “Subagents are specialized AI assistants that handle specific types of tasks. Each subagent runs in its own context window with a custom system prompt, specific tool access, and independent permissions”

In our case -> by separating clients into their own folder with context, persistent memory and Skill access we are treating every new client as an agent. Setting up a new client means spinning up their agent - giving it memory, context, skills, and specific scheduled tasks.

The Architecture

It’s split into two layers:

Shared layer (one copy, all clients use it):

Stored in the root folder agentic-os/

  • CLAUDE.md - all methodology, skill registry, routing rules

  • SOUL.md - the agent’s personality

  • USER.md - your operator profile

  • Skills - every skill you’ve installed

  • Scripts - install, update, add-client, etc.

Separate layer (one copy per client):

Stored in the client folder agentic-os/clients/client-name

  • Brand context - voice, positioning, ICP. Unique per brand.

  • Memory - session history. What you did, what’s still open.

  • Learnings - accumulated feedback, per-skill. Diverges over time.

  • Projects - all outputs and deliverables.

  • Cron jobs - scheduled tasks specific to that client.

The split is deliberate. When you improve a skill or update CLAUDE.md, every client benefits immediately. When you build a voice profile for Client A, Client B never sees it.

Context Inheritance

In the root folder agentic-os/ we have system-wide context that is inherited by all child folders agentic-os/clients/

CLAUDE.md is written at the system-wide level and even when working inside a specific client folder e.g. agentic-os/clients/client-one it reads from the root project CLAUDE.md.

Now, this does not mean you can’t add any client specific rules to your client folder agentic-os/clients/client-one inside it’s own CLAUDE.md. These merge with the root directory CLAUDE.md and if any clashes occur - the lowest level client-specific instructions take priority.

agentic-os/

├── CLAUDE.md <- parent (methodology, shared)

└── clients/

└── client-one/

├── CLAUDE.md <- child (client-specific instructions)

├── brand_context/ <- this client’s brand

├── context/

│ ├── learnings.md <- this client’s feedback history

│ └── memory/ <- this client’s session history

├── projects/ <- this client’s outputs

└── .claude/skills/ <- working copy, synced from root

Here’s how it works when you jump into a client folder using cd clients/client-one && claude

When you cd clients/client-one && claude:

  1. Root CLAUDE.md loads automatically - all methodology, every rule

  2. Client CLAUDE.md layers on top - client-specific notes

  3. Heartbeat reads SOUL.md and USER.md from the root

  4. Brand context, memory, and learnings come from the client folder

  5. Skills are loaded from the client’s local copy

Note: Skills are duplicated inside the individual client folders as Claude Code cannot inherit Skills from the parent folder (therefore we duplicate them & force update them whenever the update scripts are run)

The alternative would be to install the individual Skills globally (system-wide) and therefore all your Claude Code instances have access - however I personally prefer to see what Skills I have installed and modify them directly inside the root folder.

If you want to make edits to Skills - you can make them at the root levelagentic-os/clients/

Client-only skills are the exception. If you need a skill specific to one client, create it directly in that client’s .claude/skills/ folder. Skills that don’t exist at root are preserved during updates – the sync only overwrites skills that have a root counterpart.

clients/client-one/.claude/skills/

Solo operator, one business? Work from the root folder. No clients/ directory needed. Everything in this module is optional until you add a second client / brand.

Usually no. The same Firecrawl, OpenAI, or YouTube key works across all clients. The script copies your .env automatically on client creation.

Not automatically. Each client’s learnings accumulate tuned to that brand. New clients start with a copy of your root learnings, then diverge from there.

What if I edit CLAUDE.md at the root?
All clients see the change immediately on their next session. Parent inheritance is automatic.

What if I want a different personality for one client?
Put a context/SOUL.md in that client’s folder. The heartbeat uses the local copy instead of the root.

Can I go back to single-client mode?
Yes. Just work from the root folder. The clients/ directory doesn’t interfere with solo use.

How many clients can I run?
As many as you want.

Quick note before you start: If you installed the Agentic OS after 20th April 2026, skip this — you’re already on the latest version. Just follow the standard install steps in the modules before this one. The migration guide below is only for people who were set up before that date.

This is a significant update. We’ve reorganised how the system works under the hood to make it faster, more compatible with other AI tools, and better at handling multiple projects at once. Your data (brand context, memory, projects, learnings) is all safe and doesn’t need to change.

But the system files around it have moved. This guide walks you through updating manually, step by step.

Why we made these changes

Three things drove this update:

  1. Cross-tool compatibility - Until now, everything lived in CLAUDE.md — a single 500+ line file that only Claude Code could read. Other tools like Codex use a file called AGENTS.md instead. We’ve split the instructions so the shared operating rules live in AGENTS.md (which any AI tool can read) and Claude-specific behaviour stays in CLAUDE.md (which is now much smaller and just imports AGENTS.md). Same brain, more ways to access it.

  2. Multiple GSD projects at once - Previously, the GSD planning system (the .planning/ folder) sat at the root of your install, which meant you could only run one structured project at a time. Now each project owns its own .planning/ folder inside its project directory. You can have multiple GSD projects running in parallel.

  3. Silent, fast startup - The old system ran a long checklist every time you opened Claude — reading files, scanning skills, checking for stale context, running reconciliation. Now startup is silent. Claude reads what it needs and gets to work. Everything else is deferred to when it’s actually needed.

What’s new

  • Command Centre — a full task board UI with Kanban view, project dashboards, cron scheduling, and client management. Lives inside projects/briefs/command-centre/.

  • Scheduled tasks — scheduled jobs now run in-process (they stop when the server stops, no orphaned tasks). Full Windows support.

  • 4 new skills — ops-cron (scheduled jobs), tool-stitch (fetch UI designs), viz-interface-design (dashboard/admin UI), viz-stitch-design (design generation).

  • Version tracking — CHANGELOG.md and VERSION file so you can see what changed between updates.

Skill Systems

What are Skill Systems

Multi-skill packages that chain together to deliver a complete outcome - turn a YouTube video into edited short-form clips, a topic into a polished presentation, or build your own systems with the Meta-Skill-System-Creator Skill.

Under the hood, they’re an orchestrator skill that coordinates multiple child skills in sequence. You give it one input, and it runs through every phase automatically.

A Real Example: Longform-to-Shortform

You paste a YouTube URL and walk away. Behind the scenes, the system chains together 7 separate skills:

Eight phases, zero manual handoff between them.

Without Skill Systems

Without a system, you’d run each skill manually, one at a time.

You’d paste the YouTube URL into the transcript skill. Copy the transcript into the clip selection skill. Take those timecodes and pass them to the extractor. Take the extracted clips and feed them to the editor. Take the edited clips and manually set up posting for each platform.

Every handoff between skills is manual. You’re the glue - copying outputs, remembering what goes where, keeping track of file paths. It works, but it’s slow, error-prone, and you have to babysit every step.

A Skill System removes you as the bottleneck. The orchestrator handles every handoff, passes the right files between phases, logs timing, and recovers from errors - so you get the same result with a single command instead of seven separate conversations.

Why Not Just Build One Big Skill?

You could skip all of this and write a single mega-skill that does everything - downloads, transcribes, selects clips, reframes, edits, and posts. One massive instruction file, one monolithic block of logic.

It would work. The problem is what happens next. When you want to build YouTube-to-Ebook, you need transcription again — but it’s buried inside your mega-skill, tangled up with clip selection and video editing code. You can’t pull it out. So you rewrite it from scratch. Now you have two copies of the same transcription logic. You improve one, the other stays stale.

Mega-skills are faster to build the first time. Skill systems are faster to build the second, third, and tenth time — because you’re assembling proven parts instead of starting from zero.

Why This Architecture Matters

The real power isn’t just automation - it’s that each skill stays independent and reusable.

Take the Skill System diagram below, there are two Skill Systems:

  1. Longform-to-shortform - takes a YouTube URL and outputs shortform videos.

  2. YouTube-to-Ebook - takes a YouTube URL and outputs long form written content.

They both use tool-youtube to download and process the YouTube video, they both use tool-transcription to transcribe the downloaded video. All other skills are individual to that System, but they might be used elsewhere e.g. the humanizer skill is re-used across all written content.

This means:

  • Fix once, fix everywhere — improve the transcription skill and every system that uses it gets better

  • Build faster — new systems reuse existing skills instead of rebuilding from scratch. YouTube-to-Ebook chains 6 skills together, and 3 of them already existed from other work

  • Skills don’t break each other — each one has its own inputs and outputs, no tangled dependencies

This becomes even more evident when you start scaling it - imagine 4 Skill Systems where multiple Skills are re-utilised (see image).

Skills: Meta (System Skills)

meta-skill-creator

Build custom skills for your business. This is how Agentic OS grows beyond the skills that ship with it.

When to use it: You find yourself doing the same type of task repeatedly – writing investor updates, producing weekly newsletters, generating client reports – and you want a repeatable, improving process for it.

How it works:

  1. You describe what the skill should do

  2. It asks for reference examples – “show me a good investor update you’ve written” or “what does the ideal output look like?”

  3. It scaffolds the skill: SKILL.md (methodology), references/ (depth material), the right YAML frontmatter for triggering

  4. It tests the skill using a subagent – runs it in isolation to see if the output matches expectations

  5. You review, give feedback, it iterates

  6. Once you’re happy, it registers the skill in CLAUDE.md and README

What you get: A new folder in .claude/skills/ with a tested methodology that Claude follows every time you invoke it. The skill reads its own learnings section, so it improves with feedback just like the built-in skills.

The eval system: After building, you can benchmark the skill. It runs multiple iterations and measures consistency, quality, and adherence to the methodology. Useful for skills you’ll use heavily and want to be reliable.

Real example: “Build me a skill for writing weekly Skool lesson posts.” You’d show it 2-3 examples of lessons you’ve written, it would extract the pattern (structure, tone, length, CTA style), and produce a skill that generates lessons in that format every time.

Tip: Start with skills that have clear input -> output patterns. “Given a topic, produce a newsletter” is a good first skill. “Be creative” is not.

meta-wrap-up

End-of-session memory and learning capture. Covered in detail in the Core section, but here’s the quick version.

Say “wrap up” at the end of any session. It:

  1. Reviews everything you produced

  2. Asks for feedback (quick – one question for routine sessions)

  3. Logs feedback to the right skill’s learnings section

  4. Fixes broken skills directly (not just notes – actual edits)

  5. Updates your daily memory file with structured session notes

  6. Proposes SOUL.md personality updates if it noticed repeated corrections

  7. Syncs the skill registry (catches new/removed skills)

ops-cron

Schedule recurring tasks that run automatically in the background, even when Claude Code is closed.

The concept: You define a job in a markdown file. A background watchdog checks every hour, sees what’s due, and runs it. Each job gets its own spending cap. You get logs of every run.

  2. The skill asks three questions: what task, how often, any constraints

  4. Install the watchdog (bash scripts/install-watchdog.sh) and it runs in the background

enabled: true # flip to false to pause without deleting

Below the frontmatter is the prompt – exactly what Claude will execute each time. Jobs are self-contained. Each run has no memory of the last, so the prompt includes everything needed (file paths, output locations, what to do if something fails).
