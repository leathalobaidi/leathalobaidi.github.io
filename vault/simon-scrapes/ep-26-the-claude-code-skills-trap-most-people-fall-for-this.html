Concept · Skills

A Claude Code skill is a folder containing a SKILL.md file with YAML frontmatter (name and description) plus optional reference files, scripts, and assets. Claude Code uses progressive disclosure: only the name and description sit in context at all times (contributing to the system-prompt budget). The full body of the SKILL.md — and any reference files — are only loaded when Claude actually invokes that skill. This is the key scaling property that lets you have many skills without burning context on every interaction.

Trap 1 — The hoarding trap: installing everything

Let's start with one almost everyone is making right now, which is called the hoarding trap — installing absolutely everything. I've noticed a trend where people are sharing their "ultimate skill setups," and I often see 50, 100, sometimes 200 skills installed. It looks impressive, and everyone says, "Wow, I need to install all these skills." But here's what's actually happening under the hood when you do that.

Claude has a 15,000-character budget for the skills list in its system prompt. All the names and descriptions combined — what we call the YAML front matter — are fed into the system prompt. So if you've got a thousand skills installed, all of those names and descriptions are fed into it. Think about that: the more skills you have installed, the noisier the menu that Claude has to pick from. Claude has to pick the right skill based on a one- or two-line description, and if you have 200 generic skills, you're probably going to have some crossover. You might have three different SEO skills, four different copywriting skills — and this leads to ambiguity. What happens if you try to run your SEO audit two days in a row and it uses a separate set of skills each day? Installing these might be unintentional; you might think it's not harmful, but you're literally making your agent less smart by installing more. And if you've got skills installed globally, you might not even be able to see what your agent has access to.

With verbose descriptions or large numbers of skills, you can saturate this budget, forcing truncation or degrading how reliably Claude selects the right skill. If you're guilty of installing a library of, say, 20 marketing skills, it's probably time to remove them and build them from the ground up.

Concept · Reference files inside a skill

A skill folder can contain any number of additional files — brand voice documents, example outputs, banned-words lists, scoring rubrics, domain reference sheets — that the SKILL.md process document points to and loads only when needed. This is the official design: SKILL.md = the process; reference files = the knowledge. Keeping them separate preserves progressive disclosure (reference files don't sit in context permanently) and makes the skill debuggable (you know whether a bad output came from wrong process or wrong knowledge).

So when you install a skill, try installing it locally next time using the --local flag. Read through the files, understand the SKILL.md, and understand the reference files where the skill should actually be tailored to your business. Maybe in the references you can add a brand voice document, some example posts you've actually published before, some banned words you've never used — and that is how you get a genuinely customised experience rather than something out of the box. Remember: the value isn't necessarily in the SKILL.md, it's in the reference files you add around it. The SKILL.md is just our process document — the step-by-step instructions for Claude on how to take some action — but the reference files are what give it context for your business.

I'd highly recommend that with every skill you install, you spend 15 minutes customising it: add a reference file with your brand context, and drop in three to five examples of what you actually like the output to look like. Then update the SKILL.md to say, "Refer to my brand references or brand voice before you write anything." That's it. 15 minutes turns a generic skill into your skill. You can of course customise these for clients too.

Trap 3 — The monolith trap: cramming everything into one file

Okay, the last one — and this is the one that even experienced skill builders are missing, because it's about how the skill is built, not what's inside it. You've customised your skills, you've added your brand context, your examples, your rules — but you've put all of it inside the SKILL.md file. One massive document including process steps, brand guidelines, examples, scoring sheets, everything. It feels right: everything in one place, easy to find, easy to modify. But this is actually quietly killing your output quality, and you probably can't see why.

There's a reason people build like this: in the spec, a skill only needs the SKILL.md file. References, scripts, and everything else are optional. So technically a single-file skill is the correct minimum. But it just misses the entire point of why Anthropic designed the architecture the way they did: SKILL.md for the process, YAML front matter for what's loaded into context, and then additional files referenced in the SKILL.md that bring context and knowledge to that process — loaded only when necessary.

Concept · Progressive disclosure and context efficiency

In Claude Code's skill system, progressive disclosure means only the skill's name and description (from YAML front matter) sit in the system prompt permanently. The full SKILL.md body is loaded only when Claude invokes that skill; reference files are loaded only when the SKILL.md process explicitly calls for them. If you pack thousands of words of brand guidelines and example outputs directly into the SKILL.md, every invocation of that skill loads all of it — even when the task only needs 10% of the content. This wastes token budget and degrades context quality.

A few reasons why people default to monolith skills: marketplaces like Skills MP display the SKILL.md content. When people are browsing, they only see that file — they don't open the other files — so everyone puts everything on the front page so people can see the capability. And people think longer context equals better context, but that's just not true. Then others browse those, copy them, and learn from them, so they never see the folder structure behind well-built skills because the marketplace tends to flatten everything to one screen. Most tutorials when teaching how to build skills also stop at the SKILL.md file; even the getting-started guides from Anthropic show you how to write a SKILL.md and call it done. They don't necessarily include the context to build all the reference files. Those references, scripts, and assets are treated like an advanced layer when really they should be the default for anything serious, anything where you want good-quality output.

And if you think about how people are generating skills, a huge chunk of those 280,000-plus skills on Skills MP are auto-generated — batch created by LLMs. If you're doing that, it's far easier to just generate one file than a proper folder structure with curated reference docs.

Glossary — terms from this episode

Skill
A reusable Claude Code workflow defined by a SKILL.md file with YAML frontmatter plus optional reference files. Invoked by name; Claude selects which skill to use based on name and description alone.
YAML front matter
The ---name: … description: …--- block at the top of a SKILL.md. This is all that sits in Claude's system prompt permanently. Keep it tight and unique or Claude's skill-selection becomes unreliable.
Progressive disclosure
The skill system's scaling mechanism: only names and descriptions are in context at session start; the SKILL.md body loads on invocation; reference files load only when the process explicitly needs them. Conserves token budget.
Reference files
Non-process files bundled inside a skill folder: brand voice docs, example outputs, scoring rubrics, domain reference sheets. Loaded on demand by the SKILL.md process. The key to contextualising a generic skill to a specific use case.
System-prompt budget (15,000 chars)
The approximate character limit Anthropic allocates for all skill names and descriptions in the system prompt. Exceeding it forces truncation and degrades skill selection. The main reason to keep installed skill counts low and descriptions concise.
Hoarding trap
Installing large numbers of skills without curation, causing ambiguity in Claude's skill-selection and inconsistent outputs. Fix: curate down to 20–30 purpose-built, non-overlapping skills.
Generic-context trap
Installing marketplace skills without adding domain-specific reference files, producing competent-but-generic outputs that ignore your actual context. Fix: spend 15 minutes adding reference files and updating the SKILL.md to load them.
Monolith trap
Packing all process steps, brand guidelines, examples, and rules into a single SKILL.md, making the skill expensive to run (all tokens loaded every time) and impossible to debug (can't tell if the error is process or knowledge). Fix: separate process into SKILL.md, everything else into reference files.
Skills MP
A third-party skill marketplace for Claude Code where users share and install community-built skills. Simon notes the majority of its 280,000+ listings are auto-generated monoliths, making curation essential. Community tooling, not an Anthropic product.
--local flag
When installing a skill, --local scopes it to the current project's .claude/skills/ directory rather than installing globally. Recommended: install locally, read the files, customise before using.
