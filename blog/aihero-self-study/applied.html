AI Hero, Applied

Matt Pocock's self-study syllabus is a learning document. This is the doing document — the seven phases turned into numbered steps and worked against one project: a canonical database for the consultancy series, with people and firms as the two primary tables.

By Leath Al Obaidi · May 2026

tl;dr

The consultancy series at blog/consultancy-series/ is a 9-part narrative built on Companies House filings, press cuttings and GCR archives. A reviewer (Mike) flagged that several facts are wrong — including the founding date and founders of London Economics — and that the underlying sourcing isn't strong enough. The fix is structural, not editorial: the prose should be a view over a database, not the source of truth itself. This page applies Matt Pocock's 7-phase AI Hero playbook to that build, step by step. By the end of it you have a data/consultancy/ folder with five CSVs, a script that diffs the HTML against them, and an eval harness that catches the next mistake before a reviewer does.

The original AI Hero post lives at index.html in this folder. It is 13,500 lines of study notes, organised around Matt Pocock's seven phases of AI development. The phases work, but the document is shaped to teach them, not to execute them. This page reshapes them into a checklist and uses one ongoing project — the consultancy database — as the worked example throughout. Anywhere you see "Applied:" below, that's the consultancy project. Anywhere you don't, the step generalises to any other AI project you want to run the same way.

Phase 1

The Idea

Goal: name the thing you are building, the user who needs it, and the one outcome that defines success. One paragraph. No more.

Do

  1. Write a one-sentence description of the project.
  2. Name the single user (you, a colleague, a customer cohort).
  3. Name the one outcome that defines success — a measurable change in behaviour, accuracy, or cost.
  4. Write the smallest version of the idea that still delivers that outcome. Cut everything else.
Applied. Idea: a canonical database for every firm and person in the UK economics consultancy market, against which the existing blog series can be fact-checked and from which future parts can be generated. User: me, plus any reviewer (Mike, future readers) who wants to verify a claim. Success: every dated factual claim in blog/consultancy-series/*.html traces to a row in data/consultancy/events.csv, and a script that diffs the two flags every disagreement. Smallest version: five CSVs (firms, people, events, affiliations, sources), seeded with only the facts already verified in this branch, plus a corrections file.

Phase 2

Research

Goal: know what already exists, what the canonical sources are, and which of those sources are reliable. Skip this only if you already know the answers.

Do

  1. List the canonical data sources for your domain. For each, note: cost, reliability, refresh rate, machine-readability.
  2. List the secondary sources you would use to corroborate. Mark which are journalism, which are commentary, which are aggregators.
  3. For every source you plan to lean on heavily, name one specific past failure (where it was wrong) so you know its failure mode.
Applied. Canonical sources for the consultancy series: Companies House (free, authoritative for UK entities, machine-readable PDFs, but says nothing about trading names or founders before incorporation); firms' own About pages (authoritative on founding dates, but inconsistent and often missing); founders' personal sites and Wikipedia (good for personal history, weak on legal entity dates). Secondary: GCR (lots of coverage, but Mike's central complaint is "GCR is not a reliable source" — he is largely right: it reports rankings, not entity history); FT, consultancy.uk, PRNewswire. Known failure mode: every party tells the founding story differently and conflates "trading start" with "Companies House incorporation". The London Economics row in the existing blog is exactly that conflation.

Phase 3

Prototyping

Goal: build the crappiest end-to-end version that proves the loop closes. Avoid premature schema, premature evals, premature UI.

Do

  1. Pick the smallest input set that still exercises the full system.
  2. Build the end-to-end loop on it: input → transform → output. Hard-code anything you can.
  3. Show the output to one real user immediately. Do not pretty it up first.
Applied. Smallest input set: the five firms Mike's review actually touched — London Economics, CRA, Frontier, NERA, RBB — with the people directly named in their founding events. End-to-end loop: hand-write the CSV rows, hand-write the diff against the existing HTML, output a corrections file (data/consultancy/corrections.md). The first reader is Mike: the corrections file is the first thing he sees, because it's the artefact he asked for.

Phase 4

PRD

Goal: a single document a stranger could pick up tomorrow and execute. Schema, behaviour, non-goals, edge cases. Live document, not a deliverable.

Do

  1. Write the schema explicitly: tables, primary keys, foreign keys, allowed values.
  2. Write the non-goals. What this system does not do is half the document.
  3. Write the edge cases you have already identified. List them by name, not "we'll handle errors gracefully".
  4. Commit it next to the code (data/consultancy/README.md in this case). Update it inline as you change the schema.
Applied. The PRD is data/consultancy/README.md. Five tables: firms.csv, people.csv, events.csv, affiliations.csv, sources.csv. Composite key on affiliations is (person_id, firm_id, start_date). Each firm with multiple Companies House entities under one trading name gets a parent row and child __ch_NNNNNNNN rows — that's the structural fix for the "London Economics was founded in 1986 but the current CH entity dates to October 2000" problem. Non-goals: the database does not store narrative voice, opinion, or the prose itself. Edge cases listed by name: trading name vs CH entity (London Economics), spin-out lineage (Frontier from LE), partial-precision dates (year-only vs month-only), unconfirmed claims (Nick Morris co-founder), and partner walkouts where the new firm pre-dates its own legal incorporation (RBB).

Phase 5

Implementation plan (kanban)

Goal: chop the PRD into pieces small enough to do in one sitting. Each piece is a ticket, each ticket is a commit.

Do

  1. For each row in the PRD, write one ticket. Title = imperative verb. Body = acceptance criteria.
  2. Mark tickets blocked vs unblocked. Do unblocked ones first.
  3. Limit work-in-progress to two tickets at a time.
Applied — current backlog.
  1. Done. Seed firms.csv, people.csv, events.csv, affiliations.csv, sources.csv with the facts Mike's review exposed (London Economics, CRA, Frontier, NERA, RBB, Capital Economics, Oxera, FTI/Lexecon/COMPASS).
  2. Done. Write data/consultancy/corrections.md — the immediate artefact for the reviewer.
  3. Blocked on user. Drop ~/Downloads/ch Econ consultancy into data/consultancy/raw/ and push. This is the desktop file that contains the broader population the blog cites.
  4. Next. Write scripts/check_blog.py — loads the CSVs, walks blog/consultancy-series/*.html, regenerates corrections.md on each run.
  5. Next. Apply the five edits in corrections.md (C-001 through C-005) to the HTML in one commit. Run the script. Verify the corrections file goes to zero open items.
  6. Later. Replace at least the London Economics row with an interview-sourced claim (talk to Kay, Morris, or someone who was at LE in 1986–1988), and mark the source row type=interview. This is the answer to Mike's "you need to actually talk to the people involved".
Phase 6

Execution

Goal: do the tickets. Cleanly, in order, in committed commits.

Do

  1. Take the top ticket. Write code, write tests, commit.
  2. Push frequently. Use a feature branch.
  3. When you discover a missing requirement, edit the PRD first, then write the code. Not the other way round.

The check-blog script (sketch)

What the next ticket looks like in code. Real implementation lands once ~/Downloads/ch Econ consultancy is in the repo, because the script's evals need rows to test against.

# scripts/check_blog.py — sketch
import csv, re, pathlib, json

DATA = pathlib.Path("data/consultancy")
BLOG = pathlib.Path("blog/consultancy-series")

firms  = list(csv.DictReader((DATA / "firms.csv").open()))
events = list(csv.DictReader((DATA / "events.csv").open()))

# index founded-year by firm
founded_year = {
    e["firm_id"]: e["date"][:4]
    for e in events
    if e["type"] == "founded"
}

YEAR_RE = re.compile(r"\b(19|20)\d{2}\b")

diffs = []
for html in BLOG.glob("*.html"):
    text = html.read_text()
    for firm in firms:
        name = firm["name"]
        truth = founded_year.get(firm["firm_id"])
        if not truth:
            continue
        # find every paragraph that mentions the firm and a year
        for m in re.finditer(re.escape(name), text):
            window = text[max(0, m.start()-200):m.end()+200]
            years = YEAR_RE.findall(window)
            if years and truth not in years:
                line_no = text.count("\n", 0, m.start()) + 1
                diffs.append({
                    "file":   str(html.relative_to(".")),
                    "line":   line_no,
                    "firm":   firm["firm_id"],
                    "truth":  truth,
                    "found":  list(set(years)),
                    "window": window.strip()[:240],
                })

pathlib.Path("data/consultancy/blog_diffs.json").write_text(
    json.dumps(diffs, indent=2)
)
print(f"{len(diffs)} candidate diffs written")
Applied. The script will be intentionally noisy in v1 — "Frontier mentioned near 1999, 2000, 2003" will look like a near-miss because year-windowing is crude. That's fine: v1 finds candidates, you triage by hand. v2 (later) tightens to founding-context patterns ("founded in", "launched in", "established in"). Don't build v2 until v1 has caught a real bug that v2 would have caught earlier — that's the eval criterion for justifying the work.

Phase 7

Quality assurance — the evals

Goal: have a fixed set of cases that prove the system is doing what you said it would. Run them on every change. Don't ship without them.

Do

  1. Pick three to five concrete cases. Real ones, not synthetic.
  2. For each, write the expected output. Commit it as part of the eval.
  3. Run the evals automatically — pre-commit hook, CI, or `npm test`. Failing evals block merging.
  4. When you find a bug in production, add it to the eval set before you fix it.
Applied — the eval set for the database.
  1. London Economics founding. Input: blog mentions LE near a year. Expected: only 1986 passes; 2000, 1980s, "late 1980s", "2000" all flagged. This is the eval Mike just ran by skim, automated.
  2. CRA London opening. Input: blog mentions CRA + London + year. Expected: 14 August 2000 passes; "Oct 2000" flagged. This is the eval that catches Mike's own mistake about CRA — useful for the next reviewer.
  3. Frontier launch month. Input: blog mentions Frontier + 1999. Expected: "June 1999" passes; "May 1999" flagged (unless the source row distinguishes incorporated-May / launched-June, in which case both pass).
  4. Founder attribution. Input: blog text mentions a firm in a "founded by X and Y" pattern. Expected: affiliations.csv for that firm at start_date = founded date must include all named people.
  5. Source coverage. Input: every event in events.csv. Expected: every row has a non-empty source_id that resolves in sources.csv. No anonymous facts.

The evals run as a single Python script. Output: pass/fail count, and on fail, the same lines the diff script produces. The pre-commit hook is one line: python scripts/check_blog.py && python scripts/eval.py.

The doing list, all in one place

Strip out the phase headings and what's left is a checklist. Print this; cross items off; that's the project.

  1. Write the one-paragraph idea + success criterion. Commit as data/consultancy/README.md §1.
  2. List canonical and secondary sources. Commit as sources.csv.
  3. Hand-seed the five CSVs for the smallest input set. (done in this branch)
  4. Hand-write the corrections file against the existing prose. (done in this branch)
  5. Drop the desktop file ~/Downloads/ch Econ consultancy into data/consultancy/raw/ and push.
  6. Write the import script that maps that file's columns to the CSV schema.
  7. Write scripts/check_blog.py v1. Triage its output by hand.
  8. Apply C-001 through C-005 to blog/consultancy-series/*.html in one commit.
  9. Write scripts/eval.py with the five eval cases above. Wire it to a pre-commit hook.
  10. Interview at least one person on the London Economics or NERA-era story. Add the source_id with type=interview.
  11. Backfill the rest of the population: every firm currently in the blog gets a row, every named person gets a row, every dated event gets a row.
  12. Add a small static site generator: scripts/render_blog.py reads the CSVs and re-emits the consultancy-series HTML with the canonical numbers. The prose stays in templates; the data lives in the CSVs.

Footnotes on Matt Pocock's syllabus that I dropped here

The 13,500-line AI Hero document covers more than the seven phases. It also covers tokens, context windows, the Vercel AI SDK, MCP servers, evals, security, and a long tail of Claude Code skills. Most of those don't apply to a CSV-and-Python project; they apply once the database starts powering a model. The ones that will apply, in order of when you hit them:

  • Structured outputs (Vercel AI SDK / generateObject). When you eventually use an LLM to extract events from press cuttings into events.csv, this is the right primitive: define the schema in code, get back a typed object, fail loudly if it doesn't validate.
  • Tool calling. Once events.csv is non-trivial, a "research agent" with one tool — add_event(firm_id, date, type, source_url) — is more useful than free-text generation. The tool's validation logic is the schema.
  • Evals. Already addressed in Phase 7 above. The crucial AI Hero point is that evals come before shipping, not after, and that fixed eval cases beat generic "looks good" reviews.
  • MCP. Eventually, expose data/consultancy/ as an MCP server. Then any AI you work with (Claude, Cursor, ChatGPT) can answer "when was London Economics founded?" from the canonical source. That removes the entire class of hallucinated-fact errors that produced the original London Economics row.
  • Claude Code skills. Two specific ones are useful here. A /check-blog skill that runs the diff script and shows the report inline. A /grill-me skill (Matt's name, not mine) that role-plays an industry old hand and stress-tests every fact in a draft before publication — this is the cheap synthetic version of the Mike call, and it should run on every draft before a human ever sees it.

Everything else in the AI Hero post — tokens, context windows, sampling, the seventeen techniques — is useful background. Read it when it's relevant to the next ticket, not before.
