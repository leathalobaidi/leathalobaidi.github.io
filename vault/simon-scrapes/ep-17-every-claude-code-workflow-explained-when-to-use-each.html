Every Claude Code Workflow Explained (& When to Use Each)

17:49 runtime 101K views Published 7 Apr 2026 youtu.be/38t5UBCa4OI

Claude Code isn't a chatbot you ping one task at a time — it's an orchestration platform with five distinct agentic patterns, and knowing which to reach for (and when) is the difference between saving minutes and saving days.

What this episode covers

Simon walks through five agentic workflow patterns built into Claude Code: sequential flow (single-thread conversation), the operator pattern (you managing multiple parallel terminals with git worktrees), split and merge (Claude autonomously fanning out to up to ten subagents within one session), agent teams (agents coordinating through a shared task list — an experimental Opus 4.6 feature), and headless mode (fully autonomous, scheduled execution via the -p flag). Before the five patterns, he reveals three hidden built-in subagents — explore, plan, and general-purpose — that Claude already uses automatically to keep your main context window clean.

Why it matters for an applied economist

Every pattern maps to real research bottlenecks. Sequential flow is the Companies House deep-dive you run today; the operator pattern is running three independent firm analyses in parallel worktrees; split and merge is extracting financials from ten filings simultaneously; headless plus cron is your 7 a.m. "pull yesterday's SPE submissions and flag outliers" automation. Understanding the ceiling of each pattern prevents you from hitting context rot mid-project or burning tokens on agent teams when a single session would do.

So, it actually has already built-in subagents that it uses automatically without you having to tell it to. So, there are actually three built-in subagents baked into Claude Code.

The first one is called explore, and explore is basically a fast cheap scout. So, it runs on Haiku, which is Anthropic's fastest and cheapest model, and all it can do is read your files. The second one is similar, it's called plan, but it specifically activates when you're in plan mode. Again, separate context window, read-only.

And the third one that they've implemented behind the scenes is general purpose. And this, as it sounds, is quite general purpose, so it's the one that does the heavy lifting. It runs on Sonnet, and it has full tool access, so it can read and write, and Claude uses it for complex multi-step tasks. So, if you're asking Claude to do something that requires both exploring the codebase and making changes across multiple files, that's when it will often delegate that to a general-purpose subagent.

And the key thing here is you've not told it to do anything. Claude is deciding when to use these automatically. It's looking at the complexity of your task and routing it to the right subagent. And all of those subagents run with their own context window, so that your main conversation never gets bloated with all the reading and searching of those files. So, even before we get into the five patterns, just know that Claude Code is already doing some of this work for you. Concept · Built-in subagents (explore, plan, general-purpose)

Claude Code ships with three automatic subagents it can invoke without any instruction from you. Explore (runs on Claude Haiku, read-only) scouts files and folder structures. Plan (read-only, triggered by plan mode) researches a codebase before proposing a strategy. General-purpose (runs on Claude Sonnet, full read/write tool access) handles complex multi-step edits. Each gets its own isolated context window, so work it does never bloats your main session. You can see them appear in the terminal output.

Concept · Plan mode

Plan mode is a read-only investigation phase you trigger with /plan or Shift+Tab twice. In this mode Claude cannot edit files — it can only read and reason. It proposes a strategy and waits for your approval before making any changes. This is an official Claude Code feature, not community tooling. Use it before any destructive or irreversible operation to catch misunderstandings cheaply.

Applied economist lens

Even in a single Companies House deep-dive session — "extract turnover, operating profit, and headcount from these 10 filed PDF accounts" — the explore subagent is already firing off in the background to read your folder structure without polluting the main context. You don't have to do anything to get this benefit. What you should do is keep your project's CLAUDE.md concise and accurate so the explore agent surfaces the right files quickly: list your SQLite schema, your target fields, and where your output CSVs live.

Pattern 1 — Sequential flow: building on shared context

Okay, so with that context, let's start at the beginning with pattern one. And sequential flow is exactly as it sounds. You open a terminal, you start a Claude Code session, and every task you give it builds on the last one. So, you've got you here, we assign a task, we then move to the next task once task one is finished, move to the next, move to the next, and our shared context is growing over time. So, it's building on the context that we've been giving it throughout the different tasks.

So, it might be something like build me a landing page, Claude then builds it as task one, or a first version of it. You say add a hero image to page one, add a contact form. All of these are different tasks that build on the first task. And the output of task one can feed into task two, and the outputs of task one and two can feed into task three because it's all stored in the context window. And this might seem like you're not using any subagents, but remember that Claude has those built-in subagents. So, even in this single conversation, Claude might spin up its explore, plan, or general-purpose agents that help by actually offloading some context at the right time to get stuff done faster or more effectively on your behalf. And you'll see these pop up in the terminal, but from your perspective, it's still just one conversation flow between you and Claude.

Now, the really important thing to understand about sequential flow is that it has a ceiling, and that ceiling is dictated by the context window, and that's shown by the little green bar on the bottom of the terminal. So, the longer you work in that one session, the more the context is going to accumulate, and at that point, Claude starts forgetting things or not being able to find things, which is what we call context rot.

So, this is where skills and commands become super valuable because if you've got a well-structured CLAUDE.md and some well-structured skills that are described well, then Claude is able to load in that skill and any reference files at the right time and then offload it at the right time also to not bloat the conversation window. And then using commands like /clear and /compact will help you actually keep the summary of the conversation in the context. But, eventually with this, you're going to hit a wall, and that's when we need to move to pattern two.

Concept · Context rot

As a Claude Code session grows, earlier exchanges, file reads, and tool outputs accumulate in the context window. Eventually the model can no longer reliably access information from the early part of the session — it "forgets" or conflates things. Simon calls this context rot. The green bar at the bottom of the terminal is the context-fill indicator. The built-in /compact slash command summarises the conversation and replaces the full history with that summary, buying more room. /clear wipes context entirely and starts fresh.

Concept · Skills and CLAUDE.md

A skill is a SKILL.md file with YAML frontmatter (name, description, optional tools) that packages a reusable workflow. Claude reads skill names and descriptions and decides when to load one. A well-described skill lets Claude pull in relevant context on demand — and unload it when done — rather than holding everything in the main conversation. CLAUDE.md is the persistent project-instructions file auto-loaded into context; keeping it tight and accurate is the primary lever for fighting context rot.

Applied economist lens

Sequential flow is the right choice for a single, self-contained research task: "take firm X, download their last three years of CH accounts, extract the key financials, and add a row to the database." Each step feeds the next naturally. The danger is that a 96-firm sweep in one session hits context rot around firm 20. The fix: keep your CLAUDE.md to a tight schema definition and file-path map, and run /compact keep the schema and target fields every 10 firms or so. For anything broader than a handful of firms, graduate to pattern two or three.

In practice

/compact keep the 96-firm schema and AQuA QA rules — run this mid-session to summarise history while preserving your key instructions in context.

Pattern 2 — The operator: you as orchestrator across parallel terminals

And pattern two is called the operator. And this time, you are acting as an orchestrator. So, instead of running one Claude session, you're going to open up multiple terminal windows. You might have found yourself doing this already to get things done quickly. Each has its own Claude instance. So, you can treat these like separate agents. Terminal 1, 2, 3, and 4, completely separate agents. They have their own context window, and therefore, you can give them specific context for specific tasks.

So, this time, let's say you're building out a SaaS app. You need a new onboarding flow built. You also need your checkout bug fixed, and you want to experiment with a new design for, let's say, your user settings page. All of these tasks that we've mentioned don't depend on each other, and that's one of the reasons why we can actually orchestrate these in completely separate terminals that don't blow each other's context. Because there's no connection between those terminals, we are purely coordinating.

So, you open three terminals like this inside VS Code, and Claude actually has now a built-in flag to do this much easier. So, in the first terminal, you type in claude -w, which is important, new onboarding flow. The second one, claude -w, fix checkout bug. And the third one, claude -w, redesign user settings. And you'll see that in the project folder here, we've now opened up what's called a worktree, where you've got new onboarding flow, fixed checkout bug, and redesign user settings. And now if we go into our dot Claude folder, we've effectively got these different worktrees, which is a separate copy of your project with its own branch, and it drops it straight into the Claude Code session. Each one in its own isolated workspace, and they can't interfere with each other. So they all have a clean context window, so there's no context rot from other tasks, and us as the operator are the ones coordinating between these different terminals here.

So you're probably at this point checking in on the different terminals, the different tasks going on. You're copy-pasting findings from one to another if you needed, and then you're going to have to decide when each one is done and when you want to merge that back into the main project. But this is a massive upgrade from this sequential flow, because actually we can get multiple things done in parallel as long as they're not dependent on each other.

But if they do depend on each other, then this flow isn't going to work for us. And here's the nice touch about closing a worktree session. As soon as we close this window, Claude is going to handle the cleanup for us. So if nothing's changed, it removes that workspace automatically. But if there is work to keep, then it's going to ask you what to do. And you can see as I close these sessions, they disappear from the worktrees up here.

So the operator pattern is you running multiple Claude sessions in parallel, each with its own isolated workspace, but you're still the one coordinating. So it's perfect for independent tasks where you want maximum control and clean context windows where they will not interfere with each other. Now the operator pattern is great, but as you'll have probably noticed, you can only manage a certain number of terminals at any one point. Four to five terminals and you're flicking between those terminals, it becomes really difficult checking in on all of the sessions. So what if Claude could actually handle the parallel tasks itself?

Concept · Git worktrees (claude -w)

A git worktree is a separate checked-out copy of your repository, each on its own branch, living in its own folder. Claude Code's -w flag creates a worktree automatically when you start a session — no manual git worktree add needed. Each worktree has its own Claude instance and its own clean context window. Changes in one worktree cannot bleed into another. When you close the session, Claude cleans up unchanged worktrees automatically; if there's work to keep, it prompts you. This is an official Claude Code feature as of early 2026.

Applied economist lens

The operator pattern is ideal for a batch of independent firm analyses. Open three terminals: claude -w "extract PA Consulting accounts 2019-2024", claude -w "extract Oxera accounts 2019-2024", claude -w "extract FTI Economics accounts 2019-2024". Each runs in isolation — no context contamination, no risk of one firm's numbers bleeding into another's. You are the coordinator: check back periodically, copy findings from each terminal into your master SQLite database when done, then close and let Claude clean up the worktree. For your 96-firm database this pattern alone could cut your data-collection time by 3–5×.

Pattern 3 — Split and merge: Claude orchestrates its own subagents

So pattern three then is the split and merge. And this is where it gets really interesting, because the core idea is that within a single Claude Code session, one terminal, Claude itself can split work across multiple subagents that then run in parallel. So, if you're thinking about all four of these terminals, we could effectively spin off multiple subagents inside each terminal to get more done quicker. And then effectively at the end, it's going to merge all the results back into the main agent so that you can reap the rewards.

And if you remember those built-in subagents I mentioned earlier, explore, plan, and general purpose, those are good examples of that, but you're not limited to just those. You can create your own custom subagents inside your .claude folder. And Claude can spin up multiple of them at the same time.

And here is how that works in practice. You give Claude a task, Claude is going to analyze that task and realize that it can be broken down into independent pieces. It will then fan out into multiple subagents, each one running in its own context with no, and each one focused on the specific piece of work. And then when it's all done, it will merge all those results automatically and give you the final output. So, a lot is happening in the background here that we don't have to manually tell it to do.

So, let's say you ask Claude to research five different competitors for a client proposal. Instead of researching them one by one, which could take ages if we ran it in that sequential flow pattern one, Claude can spin up five subagents, one per competitor, and they all research simultaneously. So, each one is going to come back with their findings, and the main agent is going to synthesize the five findings into one single report. And Boris Cherney even, one of the creators of Claude Code, has talked about sometimes spinning up 15 subagents at a time to get things done. But actually, the limit on this is 10 at once. So, 10 subagents at once. Claude will actually queue any additional tasks as additional subagents.

But here's the critical limitation here. Subagents can only report back to the main agent, so they can't actually talk to each other. So, subagent one and subagent three have no idea what each other are doing. It's this hub-and-spoke methodology where the main agent, or our main instance where we're interacting with Claude, is acting as the hub, and it's receiving information back and forth from that subagent.

And if you've ever used the framework GSD or Get Done by Tash, this is like a framework I'd highly recommend for comprehensive projects. It's basically a planning framework that will split your initial project brief into subtasks and execute on those subtasks. So it's designed for big projects, but not in an enterprise fashion. We can look at the agents inside the agents folder. They've got quite a few agents in here. And actually what you can do is go into the agents listed and see exactly, like we do with skills, the name, the description. So research is a single gray-area decision and returns a structured comparison table with rationale. So we're effectively giving this agent a role to do and a specific set of context that it goes out and performs a certain process, as well as giving that agent a certain set of tools that it actually has access to. Now Claude will read the name of all agents in our .claude folder and decide when it's suitable to offload a task to those subagents. It will automatically use it when it thinks the task matches the description. Or you can specifically say, "I want to use the advisor researcher subagent in this task."

And one of the most powerful applications of this pattern is the builder-validator chain. So you have one subagent build something, and then you have another subagent actually review it. But what that requires is the subagent one to build it first, pass it back to the main agent, pass that to the second subagent through here, and then that passes the results back to the main agent. So you can see because they can't communicate with each other, they're actually orchestrating it through this main agent. But you can effectively get a built-in quality check without doing any of the reviewing yourself by using that builder-validator chain. So split and merge is Claude doing the parallelization for you within a single session. So it can fan out work to those subagents and merge the results for you. And again, they keep your context window clean. So this is really powerful.

So split and merge is awesome, but it has one limitation. Subagents can't talk to each other. So everything has to funnel through the main agent, and for some tasks, that is a genuine bottleneck. So, what if your researcher agent needs to check in with your reviewer agent? Or what if you have a front-end developer that needs to coordinate with a back-end developer?

Concept · Subagents and custom agents (.claude/agents/)

Beyond the three built-in subagents, you can create your own by adding agent definition files to your .claude/agents/ folder. Each file specifies the agent's name, description, and the tools it can access — similar in structure to skills. Claude reads all agent names and descriptions and automatically routes tasks to the right agent when the task matches the description. You can also invoke a specific agent explicitly in your prompt. The hard cap is 10 parallel subagents; further tasks are queued. Crucially, subagents cannot communicate directly with each other — all coordination flows through the main orchestrating agent (hub-and-spoke).

Worth flagging

Simon mentions "GSD" (Get Done / Get Shit Done) by Tash as a framework that uses this pattern well. GSD is community tooling — a popular third-party skill/agent framework — not an official Anthropic feature. Its agents folder structure and phase names (discuss → plan → execute → verify) are Tash's design conventions. The underlying mechanism (Claude reading .claude/agents/ definitions and routing automatically) is official. Treat GSD as useful reference architecture, not a stable API.

Applied economist lens

The builder-validator chain is the closest Claude Code equivalent of the AQuA Book QA loop. For your SPE salary survey, one subagent could extract year-on-year figures from submitted Excel files; a second subagent, given only the extracted table, could flag sign errors, implausible growth rates, and missing fields — all without any manual review. Wire this into a single prompt: "use the extractor agent to pull the 2026 submissions, then run the validator agent over the output and write any anomalies to qa-flags-2026.md." You walk away; you come back to a flagged report. That is a direct AQuA pass-3 equivalent for zero incremental cost.

In practice

To invoke a custom agent explicitly: "Use the companies-house-extractor subagent to pull accounts for firms 41–50, then pass the results to the margin-validator subagent to check for unit-scaling errors." No flag needed — natural language in the prompt is enough once the agent definitions exist in .claude/agents/.

Pattern 4 — Agent teams: agents that talk to each other

And so, this is where the fourth pattern comes in, which is agent teams. An agent team is effectively a team of agents, as it sounds, that can share findings, challenge each other, and adapt together. So, the way that they share information is through a shared task list. So, they're no longer interacting with the team lead, which is our main orchestration window. They're interacting with that shared task list. Agent one understands what tasks agent two is doing, and vice versa. They can send messages between the agents.

And it's the most advanced coordination pattern. It's the newest addition to Claude Code, and it is genuinely a game-changer for complex projects, but it should only be used in complex projects because the token usage is extremely high when you've got cross-collaboration between the agents. Now, this is completely still experimental. It shipped with Opus 4.6 as a research preview. So, you need to still enable it by adding a flag to your settings.json, which is "Claude Code experimental agent teams one." And you need to add that as an environment variable in your settings.json. But once it's enabled, you just tell Claude in your prompt that you want to use an agent team. So, you must still specify that you want to use an agent team, not like subagents, where it will choose those automatically.

You go in, you describe the task you want, you describe the team structure, or Claude will actually determine the team structure itself, and Claude will then create that team, spawn the teammates, and coordinate the work. And in your terminal, you can actually navigate between the teammates using Shift+Up and Shift+Down. And you can message directly any teammate and bypass this team lead entirely, or you can talk directly to the team lead.

Now, as I mentioned, they use significantly more tokens because we've got this back and forth between the shared task list, the team lead, between the agents. And each teammate is its full Claude Code instance with its own context window again. So, it's got a portion of the main context passed in it from that shared task list and the team lead. And they roughly estimate it's going to be four to seven times more tokens than a single session when you're actually using agent teams.

And the thing that actually matters here is you don't need agent teams for most of your work. You should actually only reach for them when subagents or even a single session couldn't do the job. So say you're trying to build out a complex SaaS application where you have a front-end developer, a back-end developer, and then a testing developer that spins up tests and needs to communicate to agent one and two as they build. That is an example where you would actually want an agent team to save from back and forth between the orchestration layer.

But a lot of people in the community are just saying that actually this is just a way to produce large quantities of work very quickly, which isn't necessarily a good thing. You still need the work to be the right work. So it's powerful but expensive, so only reach for this when the task genuinely needs that cross-collaboration.

Concept · Agent teams (experimental)

Agent teams is an experimental Claude Code feature, shipped as a research preview with Opus 4.6. Unlike the split-and-merge hub-and-spoke model, agent teams use a shared task list — agents can read each other's progress and send messages peer-to-peer, not just report to the orchestrator. You must opt in via settings.json ("claudeCodeExperimentalAgentTeams": 1 as an environment variable) and must explicitly request an agent team in your prompt. Token cost is 4–7× a single session; each teammate is a full Claude Code instance with its own context window. Use only when true cross-agent coordination is genuinely needed.

Worth flagging

As of April 2026 this feature is a research preview — experimental, behind a flag, and liable to change. Simon's note that it "shipped with Opus 4.6" is correct at the time of recording, but the API and settings key may evolve. Don't build production workflows around it yet; use split-and-merge (pattern 3) for anything that needs to ship reliably.

Applied economist lens

For virtually all of your economics research work, agent teams are overkill. The split-and-merge hub-and-spoke model handles 95% of parallelisable tasks (multi-firm extraction, parallel PE mapping, concurrent outlier checks) at a fraction of the token cost. The genuine use case for agent teams in your world would be something like: a live Wenta RCT monitoring pipeline where a data-ingestion agent and an analysis agent need to negotiate on data completeness before the analysis agent proceeds — a genuinely iterative dialogue between two specialised agents. Outside that kind of real-time inter-dependency, stick to pattern 3.

Pattern 5 — Headless mode: Claude working without you

So patterns one through four all have one thing in common. You're there in the terminal, you're either typing, coordinating, or at least watching the task. But what if you didn't need to be there at all? So this brings us on to the last pattern, which is Claude working without you. So this is the dream of autonomous workflows where you don't need to be in the loop. You just set a task, you walk away, and you come back to the results. And this is called headless, and it's where Claude Code goes from being a tool you sit with to a team member that's going to go and work independently.

This requires you having no conversation back and forth, no terminal window open, and no human in the loop at all. So when we're running this, we're using the -p flag. So we're saying claude -p, and we're entering a prompt after the P flag. So we're saying Claude, process this prompt. We don't want any interaction, no approvals. You've got full permissions, just go get it done and return to me the result when you're finished. And that on its own is kind of iterating with that task, claude -p prompt, and will give us that report.json.

But when it gets really groundbreaking, when it gets really powerful, is like when you plug this into other systems. So, if you plug this in to your Windows or your Mac scheduling function, your cron functionality that can say at 7:00 a.m. every day, fire in this command to my terminal, and then Claude goes and iterates and gets the report and sends it back to you, that is when it genuinely becomes a game changer. Because then what we're doing is actually creating workflows that can run on a schedule without your input.

So, it could be review yesterday's work and write a summary to a morning report.md. So, when Claude wakes up, it's going to read all the work you did yesterday, analyze them, write the report, and then you just have to go and look at the output before you've even done anything. You didn't open a terminal, you didn't even type a prompt in, it basically just happened. Or say you're actually running content for your business, you could have a script that pulls your latest video transcript, runs it through Claude with specific prompt, gets you back a set of social media posts saved to a file that you can then just schedule automatically. And you can chain these together, you can add skills into the prompt so that invoke specific skills. And this is all completely automated.

Now, the big limitation in this is trust. You're not asking for an iterative conversation, you're giving Claude autonomy to do things on your behalf. So, you're not checking each step. So, this works best for tasks where the output is extremely easy to verify. At the moment, you probably don't want to go headless on anything that's hard to undo. But you can actually put guardrails on this as well. If you want it to only read and not write, for example, we can do --allowedTools, and then make sure that we add specific things that it can do.

And some people in the community have taken this even further with things like the Ralph loop, which keeps feeding the same prompt back in so that Claude iterates on its own work until it gets it right. And each time it iterates, it understands what has been done in the last cycle. And people have actually been using this to ship entire projects overnight. So, this is brilliant. This is like Claude working without you. You set a task, you walk away, and you come back for the results. But like we said, it's best for batch processing things and anything where the output is really easy to verify.

Concept · Headless mode (claude -p)

Running claude -p "your prompt" fires Claude Code in headless mode — no interactive terminal, no approval prompts, no human in the loop. Claude processes the prompt, does its work, and exits, writing any output to files you specify. This is an official Claude Code flag. Combined with OS-level scheduling (macOS launchd / Linux cron), it becomes a fully automated agent that runs on a timer. Use --allowedTools read to restrict it to read-only when you don't want it writing files.

Worth flagging

The Ralph loop — a community pattern where a wrapper script repeatedly feeds the same prompt.md back into Claude until a termination condition is met — is community tooling, not an official Claude Code feature. It's a clever use of headless mode, but its behaviour depends on wrapper scripts that vary by implementation. Don't confuse it with built-in iteration. The -p flag itself is official; the looping wrapper around it is not.

Applied economist lens

Headless mode is where your SPE salary-survey pipeline becomes genuinely automated. Write a tight prompt: "Read all Excel files in ~/Downloads/SPE/2026-submissions/, extract the salary rows, apply the outlier rules in qa-rules.md, and write any flagged rows to qa-flags-2026.md. Do not modify source files." Schedule this via launchd to run at 6 a.m. each morning during the submission window. You arrive at your desk to a ready-made flags file. For your DCC regulatory reporting, a nightly claude -p "read the PI Tracker CSV, compute variance against the £719.9m baseline, and append a one-paragraph summary to morning-report.md" takes the daily sense-check off your plate entirely. Key guardrail: add --allowedTools read,write explicitly so it cannot call external APIs or run shell commands beyond file I/O.

In practice

claude -p "$(cat prompt.md)" --allowedTools read,write > run.log 2>&1 — run headless with a file-based prompt, restrict tools to read/write only, capture all output to a log for post-hoc verification.

Glossary — terms from this episode

Sequential flow
Pattern 1: a single Claude Code session where each task builds on the last, sharing one growing context window. The simplest workflow; hits a ceiling when context rot sets in.
Context rot
The degradation in model reliability as a session's context window fills up — Claude starts losing or conflating information from earlier in the conversation. The green bar in the terminal shows fill level. Mitigated by /compact and well-structured skills.
Operator pattern
Pattern 2: the human manually manages multiple parallel Claude sessions, each in its own terminal and git worktree, coordinating between them. Maximum control; limited by how many terminals a human can track.
Git worktree (claude -w)
An official Claude Code flag that creates a separate branch and working directory for each session, keeping parallel tasks fully isolated. Claude cleans up unchanged worktrees on session close.
Split and merge
Pattern 3: within a single session, Claude fans out work to up to 10 parallel subagents, then merges their results. The human writes one prompt; Claude handles all parallelisation. Subagents cannot communicate directly — everything routes through the main (hub) agent.
Subagent
A child agent spawned by the main orchestrator with its own isolated context window and (optionally) restricted tool set. Built-in types: explore, plan, general-purpose. Custom agents can be defined in .claude/agents/.
Builder-validator chain
A split-and-merge pattern where one subagent builds output and a second subagent reviews it — a built-in QA loop that requires no human review step.
Agent teams
Pattern 4: an experimental feature (research preview, Opus 4.6) where multiple full Claude Code instances share a task list and can message each other directly, bypassing the orchestrator. 4–7× token cost of a single session; must be opted into via settings.json.
Headless mode (-p)
Pattern 5: running claude -p "prompt" with no interactive terminal — Claude executes the task autonomously and exits. Composable with OS scheduling (cron, launchd) for fully automated recurring workflows.
Ralph loop
A community wrapper that repeatedly feeds the same prompt file back into headless Claude until a termination condition is met. Not an official feature; useful for iterative refinement tasks left to run overnight.
Hub-and-spoke
The coordination topology of the split-and-merge pattern: a central main agent routes tasks to subagents and receives their results; subagents have no direct channel to each other.
Plan mode
An official Claude Code mode (/plan or Shift+Tab twice) that locks Claude to read-only investigation. It proposes a plan and waits for approval before making any edits.
