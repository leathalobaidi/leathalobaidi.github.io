Tip 1 — Agent teams: when they compound, when they just cost more

If you're using Claude Code the way that most people do, you're leaving a huge amount on the table. It's not because the tool itself is limited, but because of how you're approaching it. Most tutorials show isolated tricks, and what they don't show you is how those techniques can compound when used together. In this video, I'm walking you through ten Claude Code techniques that build on each other, starting with changes that immediately improve your results and ending with workflows even experienced users have never used.

Let's start with the newly released agent teams. Most people are either running one Claude Code session or starting to use subagents — that's fine for simple tasks, but if you're working on more complex projects it's not good enough anymore. Agent teams let multiple Claude instances work in parallel and talk to each other. Before agent teams were released, people tended towards using subagents with a delegation architecture: a main parent agent delegates to individual subagents. Say we have a content writer as our main agent and we want help from a researcher and a content reviewer — those would be specialist subagents with delegated tasks that pass their outputs back to the main agent. The problem as you expand the scope of this project is that more subagents feeding into this main agent creates a communication bottleneck. Those subagents aren't able to communicate to each other; everything must be processed by the main agent.

Concept · Subagents / Agent teams

Subagents are Claude Code processes spawned via the Task (Agent) tool, each with their own isolated context window and tool set; only their final summary returns to the main agent. Agent teams extend this with a shared task list and direct peer-to-peer communication between agents — so agents can coordinate without routing every message through the orchestrator. This reduces orchestrator bottleneck for deeply collaborative tasks. The feature is configured by adding a line to settings.json and running claude update to get the latest version.

In contrast to the new architecture, with Claude Code agent teams we have a main agent that can still talk to those teammates or subagents, but the key differences are: they have a shared task list so each teammate can contribute and pull from it, and they can talk to each other directly — knowing exactly what's going on through collaboration with other teammates, the shared task list, and the main orchestrator. This comes at higher cost because more agent-to-agent conversations means higher token usage. So there are only certain use cases where I'd recommend it.

The tip is actually all about where not to use agent teams. From non-development use cases, there aren't many places I'd start using this. The first area where I would not use it is creating an agent team that researches and writes social media posts for LinkedIn and X, with three agents: a researcher, a reviewer, and a writer. The reason is the scope is really limited — you're just writing say five to ten social media posts, something that can probably be done with a single task. At best you have three agents that don't need to cross-collaborate: the researcher doesn't need to talk to the reviewer; both just need to pass information back to the main agent (the writer). So it's totally overkill for that use case.

Where agent teams really start to shine is when you have more complex projects where cross-collaboration is critical. In the same content-creation flow, you could drop a YouTube transcript into a project folder and have Teammate 1 write a long-form SEO-optimised blog article, Teammate 2 write a LinkedIn carousel script focusing on hooks, and Teammate 3 write an email newsletter version. What you want is for there to be no contradicting hooks or information across the three outputs — so you actually need collaboration between teammates: "what angle are you working on so I don't duplicate it?" Where it really shines is when you're building complex apps — say a Next.js app with Stripe — where you have three distinct agent layers: an API layer, a front-end layer, and a testing agent. The API and front-end agents constantly need to talk to the testing agent so it can spin up their own tests. In the subagent format that becomes a bottleneck; with a shared task list and peer collaboration it becomes much more effective, though still more expensive. By the way, with agent teams you can bypass the main agent entirely: navigate to a specific teammate using Shift+Up/Down on your keyboard and give tasks directly to that agent.

My summary: don't start with agent teams. Start simple, graduate to subagents, and only reach for agent teams when the complexity and collaboration requirements genuinely demand it.

Applied economist lens

The "bottleneck test" Simon describes maps directly to Leath's Companies House extraction pipeline. Pulling accounts for 96 consultancy firms one at a time is a classic single-agent bottleneck. A two-agent team — one agent fetching and parsing PDF accounts, a second simultaneously cross-checking the extracted turnover figures against prior-year entries in the SQLite database for consistency — would flag discrepancies in parallel without routing every comparison through a master agent. But for a simpler task like drafting a single Substack post per firm, a lone subagent is cheaper and sufficient. Apply the "do agents actually need to talk to each other?" test before spinning up a team.

Tip 2 — Hooks, skills, and slash commands: the simplest way to keep them straight

Tips two, three, and four all get confused constantly: hooks, skills, and slash commands. Here's a simple way to remember them.

Slash commands live in the .claude/commands/ folder — one markdown file per command. Think of it as "you press a button and an action is taken because you told it to take that action." So you might have a /linkedin-post command where the markdown file says "write three LinkedIn post variations" and you pass your arguments in. You run it by typing /linkedin-post and then adding your topic — say "MCPs" — and it writes three LinkedIn post variations about MCPs. When it starts running you'll see it begins to use certain skills automatically.

Concept · Slash commands

Slash commands are /name shortcuts stored as markdown files in .claude/commands/. Custom ones let you package a repeatable prompt into a single keystroke. Unlike skills, you explicitly invoke them — you press the button. They can accept arguments appended after the command name. They are simpler than skills: one file, user-triggered, no automatic inference about when to activate.

This brings us to skills. Unlike slash commands — where you're telling Claude to act at a specific time by pressing a button — Claude understands from the description of the skill when to use it. Skills can be more comprehensive, and Claude decides when to invoke them rather than you explicitly telling it. For example, inside your Claude skills you might have a brand voice skill with examples for different platforms (LinkedIn, X, Instagram), a banned-words list, and a SKILL.md that includes a description telling Claude to always use the brand voice guidelines every time it writes social media content. That's how it knows to use that skill without being told explicitly. You back that up by putting a line in your CLAUDE.md: "every time you write content, make sure to use these skills." But actually, Claude would know from the skill description alone — without the CLAUDE.md line — to use that skill. Skills are more comprehensive: you can give more context than a command's single file allows, and they're invoked automatically.

Concept · Skills

A skill is a SKILL.md file with a name, description, and optional supporting files (examples, reference data, scripts). Claude reads the description at the start of a session and infers when the skill should be activated — it doesn't need to be explicitly triggered. The key scaling property is progressive disclosure: only the name and description are loaded into context upfront; the full body is fetched only when the skill is actually invoked, keeping the context window lean.

Hooks are completely different. Hooks don't require any LLM tokens at all — they're for things that can be done programmatically without AI. For example, in a content pipeline you might have a banned-word checker added to settings.json: any time output drafts are added to a specific file, check them for banned words and report whether any were found. In the full content pipeline: you run /linkedin-post to write about MCPs, Claude automatically uses the brand voice skill, and then the hooks check for banned words in the background. They work together as a three-part system.

Concept · Hooks

Hooks are shell commands the Claude Code harness executes automatically on lifecycle events — PreToolUse, PostToolUse, SessionStart, Stop, etc. — configured in settings.json. They run deterministically, outside the LLM loop, so they consume zero tokens and cannot be bypassed by the model. Typical uses: running a linter after every file edit, blocking dangerous commands, auto-formatting output, writing a changelog entry. They complement skills and slash commands but operate at a different layer.

Applied economist lens

Map the three primitives to Leath's SPE salary-survey workflow. A slash command /run-aqua-checks triggers the full AQuA-style QA pass on a year's submission on demand. A skill salary-survey-outlier-detector is described as "use this whenever reviewing salary data rows" — so it activates automatically when Claude encounters the annual Excel export. A hook on PostToolUse runs a Python script that checks every file written to the Publish_2026/ folder for PII fields (names, emails) and logs any found — zero tokens, deterministic, always fires. Three tools, three jobs; none steps on the others.

Tip 3 — CLAUDE.md: point, don't dump

The one tip I'll give you for CLAUDE.md files is: point, don't dump. Most people dump their entire brand guide, content pillars, brand voice, audience research, and example posts into their CLAUDE.md file. Do not do that. Keep CLAUDE.md lean and point to where the detail lives; don't dump the detail itself. The recommendation is 20 to 30 lines in your CLAUDE.md — this is loaded into context every time you have a conversation with Claude. So you want to include only the relevant framing context and then point to the other bits it will need to know at certain times. If it's writing content, it knows to look in skills/humanizer to understand how it should write, and the same for brand voice. Claude can read those files only when it needs them, so you get a clean, scannable CLAUDE.md that doesn't eat up your context window by passing in too much information every time.

Concept · CLAUDE.md

CLAUDE.md is the primary persistent-instructions file auto-loaded into every Claude Code session's context. It is hierarchical: a user-global one at ~/.claude/CLAUDE.md applies everywhere; a per-project one at ./CLAUDE.md applies to that project. Because it loads every time, its size directly affects how much context window is consumed before any work starts. Simon's "20–30 lines" guideline is a practical way to keep this overhead low while relying on skills and referenced files for the detail.

Applied economist lens

For the 96-firm consultancy research project, a bloated CLAUDE.md that contains the full data-collection protocol, all 96 firm names, and the SQLite schema will silently eat context on every session — including sessions where Leath is just writing a Substack draft. Instead: keep CLAUDE.md to ~25 lines that say "see skills/companies-house-extractor for account-pull protocol" and "see docs/db-schema.md for table definitions." Claude loads those only when the relevant work starts. The same logic applies to the SPE survey master workbook: don't embed the data dictionary in CLAUDE.md; point to it.

Tip 4 — Plugins: one install, the whole workflow

So we have skills, hooks, and slash commands. But what if you want to share a collection of those with someone else, or install someone else's setup? This is where plugins come in. They're not additional to those three — they're how you distribute them or install them on your own system. A plugin is simply a package of skills, commands, hooks, and potentially agents that you can install in one go. So you go to the Claude plugin marketplace, add a certain plugin with two commands, and you've literally got all of their commands, skills, and hooks in one step.

An example: someone has a Reddit research plugin. It includes a skill that teaches Claude how to analyse Reddit threads, a Reddit research command that triggers deep research, and a hook that automatically summarises the findings into a markdown file. Instead of installing three things individually, you run two commands on the plugin and get the whole workflow straight away. Think of it as a packaged collection of skills, hooks, commands, and agents — install once, get the whole workflow. One caveat: the ecosystem for plugins is still very developer-focused, but the concept is powerful and hopefully there will be less dev-focused plugins coming to Claude Code over time.

Concept · Plugins

Plugins are distributable bundles — a zip or repo — that package skills, slash commands, hooks, and optionally agents into a single installable unit. They use Claude Code's plugin marketplace (or a direct URL install) and are the distribution mechanism that lets creators share complete workflows rather than individual files. The underlying components (skills, commands, hooks) are exactly what Simon described in Tip 2; plugins just give them a packaging and installation layer.

Applied economist lens

If Leath were to package his Companies House research workflow — the accounts-downloader skill, the /ch-deep-dive slash command that extracts revenue and headcount into a CSV row, and the hook that runs a duplicate-company-number check — as a plugin, he could re-install the entire setup in a new project directory with two commands. More practically, he can look for community plugins that already do parts of this (Companies House API wrappers, PDF-extraction skills) rather than building from scratch. The plugin ecosystem is thin right now for economics tooling, but monitoring it is worth the ten minutes a month.

Tip eight is about built-in commands, again highlighted by YK. These are super useful and you'll use them fairly frequently. Concept · Built-in slash commands

/clear, /usage, and /stats are native Claude Code commands (not custom markdown files). /clear compresses/discards conversation history to free context — useful when a session has drifted far from its original task. /usage shows current session and weekly token consumption against the plan limit. /stats shows cumulative session history. These are distinct from custom slash commands you write yourself in .claude/commands/.

Tip 10 — Fact verification: a table of what Claude can and can't confirm

Tip ten addresses something we've all come across: Claude confidently writing a statistic or claim without actually verifying it. The quick fix is to tell it: "Double-check every claim and statistic, and make a table of what you could and couldn't verify, including the source."

In the demo: a bunch of LinkedIn post variations about MCPs with facts in them — each post using different facts so they don't feel repetitive. Saying "double check every claim and statistic, make a table of what you could verify" produced a markdown fact-check table listing every claim across all three versions, with source, verification status, and confidence. The results were instructive: "MCP launched November 2024" — verified. "Downloads 100K to 8 million in five months" — unverified (no primary source, no methodology cited, single blog post; not proof enough). Several claims came back "cannot verify." Even asking it to do that once, it went out, found a single point of validation, and called it a fact — and then the table immediately exposed that. The output: remove claims 5, 6, 7; soften claims 2 and 3; or swap for official numbers. So even one pass of verification produces a structured, actionable audit of the facts in a draft.

Concept · Verification workflow

This is a usage pattern, not a named Claude Code feature. The mechanic is: after generating a draft with factual claims, pass the draft back with an instruction to verify each claim independently and produce a structured table. Claude will web-search each claim and report its confidence level. The table format (claim | source | verdict) makes the output immediately actionable. It doesn't eliminate hallucination, but it surfaces the weakest claims explicitly rather than leaving them buried in confident-sounding prose.

Glossary — terms from this episode

Agent teams
Multiple Claude Code instances that share a task list and can communicate directly with each other, not just through a central orchestrator. Useful when agents genuinely need to coordinate outputs; overkill when tasks are sequential or non-overlapping.
Subagent
A Claude Code process spawned via the Task (Agent) tool with its own isolated context window; only the final summary returns to the main agent. The building block before agent teams.
Slash command
A user-triggered shortcut (/name) stored as a markdown file in .claude/commands/. You explicitly invoke it; Claude doesn't decide on its own when to use it.
Skill
A SKILL.md file with a description that tells Claude when to activate it automatically. Progressive disclosure: only name and description load into context by default; the full body loads on invocation. More powerful and context-efficient than slash commands for recurring workflows.
Hook
A shell command configured in settings.json that runs automatically on Claude Code lifecycle events (e.g., after a file edit, at session start). Uses zero LLM tokens; deterministic; cannot be skipped by the model. Ideal for linting, PII checks, banned-word scans.
Plugin
A distributable bundle of skills, commands, hooks, and optionally agents. The packaging and installation layer that lets creators share complete workflows in one install step.
CLAUDE.md
The primary persistent-instructions file auto-loaded into every Claude Code session. Hierarchical (global and per-project). Keep it lean (20–30 lines); point to skills and docs rather than embedding detail directly.
Gemini CLI workaround
A community technique (not an Anthropic feature) that installs the Gemini command-line interface and wraps it in a Claude Code skill, enabling access to Reddit and other sites Claude Code itself cannot fetch from directly.
Last 30 days skill
A community-built slash command + skill that uses OpenAI and xAI API keys to scan Reddit and X for discussions on a given topic over the past 30 days and synthesise them into structured findings with source counts.
/clear
Native Claude Code built-in that compresses/discards conversation history to free context window space. Useful when switching tasks mid-session.
Fact-verification table
A workflow pattern — not a named feature — where you ask Claude to verify each claim in a draft and produce a structured table (claim | source | verdict). Surfaces hallucinated or weakly-sourced statistics before they appear in published output.
